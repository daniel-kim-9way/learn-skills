## EXPLAIN

### C2 기획서 → 동작하는 코드

C2(PLAN) 과정에서 여러분은 서비스 기획서를 만들었습니다. "어떤 기능이 필요한지", "누가 쓰는지", "어떻게 동작해야 하는지"를 글로 정리했죠.

이제 그 기획서를 Claude에게 주면 됩니다.

비유: 건축 설계도를 건축가에게 주면 건물을 지어주는 것처럼, 기획서를 Claude에게 주면 코드를 만들어줍니다. 여러분은 의뢰인이고, Claude는 시공사입니다.

### "Claude에게 무엇을 만들지 말하는" 패턴

이 패턴이 C3 전체에서 가장 중요한 기술입니다:

1. **무엇을**: 만들어야 할 기능을 명확히 설명
2. **왜**: 이 기능이 사용자에게 어떤 도움이 되는지
3. **어떻게 보여야 하는지**: 완성됐을 때 어떤 모습인지

예시:
```
"사용자가 독서 목록에 책을 추가할 수 있는 기능을 만들어줘.
책 제목, 저자, 읽기 시작한 날짜를 입력할 수 있어야 해.
저장 버튼을 누르면 목록에 바로 나타나야 해."
```

### 구현 → 테스트 → 피드백 사이클

한 번에 완벽한 기능이 나오지 않아도 됩니다. 아래 사이클을 반복합니다:

1. Claude가 기능 구현
2. 브라우저에서 테스트
3. "이 부분이 달랐으면 좋겠어"라고 피드백
4. Claude가 수정
5. 다시 테스트

이 과정이 실제 서비스 개발의 기본 리듬입니다.

## EXECUTE

### 단계 1: C2 기획서를 Claude에게 공유

C2에서 만든 기획 문서를 Claude에게 붙여넣거나 파일로 전달합니다:

```
이게 내 서비스 기획서야:
[C2 기획 내용 붙여넣기]

여기서 핵심 기능 1번인 [기능명]을 먼저 구현해줘.
Rails로 만들어야 해.
```

기획서가 없다면 Claude에게 정리를 도와달라고 합니다:
```
내 서비스는 [서비스 설명]이야.
핵심 기능을 정리해줘. 가장 중요한 기능 1개부터 시작하자.
```

### 단계 2: Claude가 핵심 기능 구현

Claude가 필요한 파일들을 생성합니다:
- Model (데이터 구조)
- Controller (요청 처리)
- View (화면)
- Routes (URL 경로)

진행하는 동안 무엇을 만들고 있는지 Claude에게 물어봐도 좋습니다:
```
지금 뭘 만들고 있는 거야? 비개발자도 이해하게 설명해줘.
```

### 단계 3: 브라우저에서 테스트

서버가 실행 중이면 `http://localhost:3000`에서 확인합니다.

테스트 체크리스트:
- [ ] 기능이 실제로 동작하는가?
- [ ] 데이터가 저장/조회되는가?
- [ ] 에러 없이 작동하는가?
- [ ] 화면이 이상하지 않은가?

문제가 있으면:
```
[설명한 문제 상황]이 발생했어. 해결해줘.
```
