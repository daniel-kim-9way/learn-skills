## EXPLAIN

### 결제·API·webhook은 사실 모두 같은 패턴이에요 💳

이번 레슨은 비개발자가 가장 큰 오해를 푸는 lesson이에요.

가장 흔한 오해: **"결제하려면 우리도 카드 번호를 받아서 저장해야 하나요?"**

답: **아니요. 절대 안 됩니다.** 한국 법(전자금융거래법)상 그래도 안 되고, 보안 사고 시 책임이 너무 커요.

그러면 어떻게 결제가 일어날까요? **PG(Payment Gateway)**라는 결제 전문 회사에 위탁해요. 한국에서는 **토스페이먼츠 / 포트원(아임포트) / KG이니시스**가 대표적.

그리고 더 중요한 깨달음 — **이 결제 흐름은 다른 외부 서비스 호출과 완전히 같은 패턴**이에요.
- 결제 = 우리가 토스페이먼츠에 "이 사용자 1만원 결제해줘" 요청 → 결과 받음
- AI 요약 = 우리가 Claude API에 "이 텍스트 3줄 요약해줘" 요청 → 결과 받음
- 카톡 알림 = 우리가 카카오 API에 "이 사람한테 메시지 보내줘" 요청 → 결과 받음

**전부 같은 패턴**: ① 우리 서버 → 외부 서비스에 요청 ② 결과 받음 ③ DB에 결과만 저장.

이 패턴 한 번 익히면 새로운 외부 서비스가 나타나도 "아 그것도 똑같지" 됩니다.

이번 레슨에서 다룰 5가지:

1. **결제 흐름 — "카드 입력 → PG → 우리 DB는 결과만"**
2. **한국 PG 3가지** (토스 / 포트원 / KG이니시스)
3. **API 호출도 같은 패턴** (Claude API / 카톡 / 디스코드)
4. **API key 안전 보관** (.env 파일, 절대 GitHub에 X)
5. **webhook — 외부가 우리에게 거는 전화**

### 진행 방식

총 7개 Turn으로 진행해요. "충격 → 풀이 → 같은 패턴 확장 → 본인 SaaS 적용" 순서로.

### Turn 3: 결제 흐름 — 7단계

> "결제가 어떻게 일어나는지 7단계로 따라가 봐요."

```
[결제 7단계 — 토스페이먼츠 기준]

1. 사용자가 본인 SaaS에서 "결제" 버튼 클릭

2. 본인 서버가 토스 결제창 URL 만들어서 사용자에게 전달
   (토스 API 호출 + 주문번호·금액 등 정보 함께)

3. 사용자가 토스 결제창(별도 페이지)에서 카드 정보 입력
   ⚠️ 이때 카드 정보는 본인 서버에 안 옴!
   사용자 폰 → 토스 서버로 직접 갔다 옴

4. 토스가 카드사와 통신 → 승인 결과 받음
   (이 부분 5~30초 걸림)

5. 토스가 본인 서버에 결과 통보 (webhook 또는 redirect)
   "결제 ID xyz123, 1만원, 성공"

6. 본인 서버가 DB에 저장
   payments 표:
   id | user_id | payment_key      | amount | status   | created_at
   1  | 42      | toss_xyz123abc   | 10000  | success  | 2026-05-05

7. 사용자에게 "결제 완료" 화면 보여줌
```

본인 DB의 payments 표에 들어가는 정보:

```
[payments 표 — 우리가 보관하는 것]

id              자동 1, 2, 3...
user_id         어느 사용자가 결제했는지 (users.id 참조)
payment_key     토스가 부여한 결제 식별자 (예: "toss_xyz123abc")
amount          금액 (10000)
status          상태 (success / fail / canceled)
created_at      결제 시각

⚠️ 이 표에 카드 번호·CVC·유효기간 어디에도 없어요.
   payment_key는 토스가 발급한 식별자. 환불·조회 시
   이 키로 토스에 다시 물어보면 됨.
```

> "이 흐름의 핵심: **카드 정보는 사용자 ↔ 토스 사이만 흐름. 본인 서버는 결과만 받음.**"
>
> "그래서 본인 책임이 가벼움. PCI-DSS 같은 카드 보안 인증도 본인이 받을 필요 없음 (토스가 다 해놓음)."

### Turn 4: API 호출도 완전히 같은 패턴

> "여기서 핵심 깨달음. **이 결제 흐름은 결제만의 특별한 방식이 아니에요.** 사실 모든 외부 API 호출이 같은 패턴이에요."

같은 패턴 표 출력:

```
[전부 같은 패턴 — 외부 서비스 호출]

시나리오          외부 서비스         우리가 보내는 것      받는 결과
────────────────────────────────────────────────────────────
결제              토스페이먼츠         금액·주문번호         결제 ID·성공/실패
AI 자동 요약      Claude API          긴 텍스트            3줄 요약
번역              DeepL API           한국어 텍스트         영어 텍스트
카톡 알림         카카오 비즈니스 API  받는 사람·메시지      전송 성공 여부
디스코드 봇 응답  Discord API         채널·메시지          메시지 ID
이미지 생성       OpenAI DALL-E API   "고양이 그림" 텍스트  이미지 URL
이메일 발송       SendGrid API        받는 사람·내용        전송 ID
SMS 발송         네이버 클라우드     받는 사람·메시지      전송 ID

→ 전부 패턴이 같음:
   ① 우리 서버 → 외부 서비스에 요청
   ② 결과 받음
   ③ DB에 결과만 저장
```

> "예시 — 본인 SaaS가 '독서 노트 자동 요약' 기능을 넣고 싶다면?"

```
1. 사용자가 긴 노트 작성 후 "AI 요약" 버튼 클릭
2. 본인 서버가 Claude API에 호출:
   "이 텍스트를 3줄로 요약해줘: [노트 본문]"
3. Claude가 응답: "요약: ① ... ② ... ③ ..."
4. 본인 서버가 DB의 ai_generations 표에 저장:
   note_id | summary           | created_at
5. 사용자에게 요약 결과 보여줌
```

> "결제와 완전히 같죠? **요청 → 결과 받음 → DB에 결과만**. 패턴이 똑같으니까, 한 번 익히면 다른 모든 외부 API도 같은 방식으로 다룰 수 있어요."

질문:

> "외부 서비스에 요청을 보내려면 한 가지가 필요해요. **'이거 누가 보낸 거야?'** 라는 질문에 답할 수 있어야 함. 답이 **API key**예요."

API key 예시:

```
[API key 예시 — 비밀번호급 비밀]

ANTHROPIC_API_KEY=sk-ant-api03-Kix3jL.PqrM...8bN3vXz

KAKAO_API_KEY=a1b2c3d4e5f6789...
SENDGRID_API_KEY=SG.AbCdEfGh.XyZ...
```

> "이 키는 **비밀번호급 비밀**이에요. 누구든 이 키만 손에 넣으면:"
> - 본인 이름으로 외부 서비스를 호출할 수 있음
> - 비용도 본인에게 청구됨 (Claude API key 유출 → 누가 1억 토큰 쓰면 본인 청구서 폭탄)

>
> "그래서 절대 두 가지를 하면 안 돼요:"
> 1. **코드에 직접 적기 X** — `api_key = "sk-ant-api03-..."` 같은 식
> 2. **GitHub에 올리기 X** — public repo면 1초 만에 발견됨

올바른 보관 방식:

```
[.env 파일 + .gitignore]

ANTHROPIC_API_KEY=sk-ant-api03-실제키
TOSS_SECRET_KEY=test_sk_실제키
KAKAO_API_KEY=실제키

[.gitignore — git이 .env를 무시하게]

.env.local
.env.production
```

> "비유: **.env = 잠긴 서랍**. 본인 PC에만 있고, 코드에는 '서랍에서 키 꺼내서 써' 라고만 적혀 있음. 그 서랍은 GitHub 같은 공개 장소에 절대 안 올림."
>
> "Rails의 경우: `ENV['ANTHROPIC_API_KEY']` 한 줄로 .env에서 키를 읽어요. AI에게 '결제 붙여줘' 한마디면 알아서 .env 패턴으로 짜요."

질문:

> - "처음 들어요"
> - "어렴풋이 들어봤어요"

### Turn 6: webhook — 외부 서비스가 우리에게 거는 전화

> "마지막 개념. **webhook**이에요."
>
> "지금까지는 **우리가 외부에 거는 전화** (= API 호출)을 봤죠. 반대로 **외부 서비스가 우리에게 거는 전화**를 webhook이라고 해요."

webhook 비유:

```
[API vs webhook — 전화 거는 방향]

[API 호출 — 우리가 거는 전화]
우리 서버 ──"결제 처리 부탁해"──> 토스
                                      ↓
우리 서버 <──"결과: 성공/실패"────── 토스

[webhook — 외부가 거는 전화]
                              "결제 완료됐어!"
우리 서버 <──────────────── 토스
   ↓
우리 서버가 받아서 처리
   (DB 업데이트, 사용자 알림 등)
```

> "**가장 자주 쓰이는 곳이 결제예요.** 사용자가 토스 결제창에서 카드 입력하고 카드사 승인 받는 데 5~30초 걸려요. 그동안 우리 서버는 가만히 있다가 — 토스가 '결제 완료됐어!' 하고 전화 걸어줘요. 그게 webhook."
>
> "webhook을 받으려면 우리 서버에 **'토스가 부를 수 있는 URL'**이 있어야 해요."

```
[webhook URL 예시]

본인 서비스 URL:    https://본인서비스.com
토스 webhook URL:   https://본인서비스.com/webhooks/toss

토스 대시보드에 위 URL을 등록해두면, 결제 완료 시
토스가 자동으로 호출해서 "결제 완료" 알려줌.
```

> "다른 webhook 예시:"
> - 깃허브 → 코드 푸시되면 우리 서버에 알림 (자동 배포 트리거)
> - Stripe (해외 PG) → 환불·취소 시 우리 서버에 알림
> - 슬랙 → 봇 메시지 받았을 때 우리 서버 호출

핵심:
> "**API = 우리가 거는 전화 (요청 → 결과)**"
> "**webhook = 외부가 거는 전화 (외부가 알려줌)**"
>
> "결제는 보통 둘 다 써요. 우리가 결제창 URL 달라고 API 호출 → 사용자 결제 진행 → 토스가 webhook으로 결과 알려줌."

질문:
> "본인 SaaS에 결제가 들어가면 webhook URL 1개가 필요해요. 그 외 다른 webhook이 필요할 것 같은 시나리오가 있어요?"
>
> "예시: '깃허브 자동 배포', '카카오 챗봇 메시지 수신', '없음'"

학생 답변 받기.

### Turn 7: 본인 SaaS 외부 서비스 지도 + 파일 저장

> "오늘 본 거 다 모아서 본인 SaaS의 외부 서비스 지도를 만들어 봐요."

학생이 Turn 4에서 답한 외부 서비스 후보를 다시 정리. 각 서비스에 대해:
- .env에 어떤 키가 필요한지
- DB에 어떤 결과 표가 필요한지
- webhook이 필요한지

본인 SaaS 외부 서비스 지도를 텍스트로 출력 (예시 — 독서 노트 SaaS):

```
[본인 SaaS 외부 서비스 의존성 지도 — 예시]

[본인 SaaS]
  ↔ 토스페이먼츠 (PG)
      .env 키:    TOSS_CLIENT_KEY, TOSS_SECRET_KEY
      webhook:    /webhooks/toss
      DB 표:      payments

  ↔ 구글 OAuth (로그인)
      .env 키:    GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET
```
