## EXPLAIN

### 데이터베이스 = 엑셀

"데이터베이스"라는 말이 어렵게 들릴 수 있습니다. 개발자들만 쓰는 전문 용어 같죠.

하지만 사실 여러분은 이미 데이터베이스를 써본 적이 있습니다. **엑셀(또는 구글 시트)**을 써보셨다면요.

데이터베이스는 엑셀과 거의 같습니다. 이름만 다를 뿐이에요:

| 엑셀에서 부르는 이름 | 데이터베이스에서 부르는 이름 | 설명 |
|---------------------|---------------------------|------|
| 통합 문서 (워크북) | 데이터베이스 (DB) | 시트들의 모음 — 전체 파일 |
| 시트 (Sheet) | 테이블 (Table) | 데이터를 담는 하나의 큰 표 |
| 열 제목 (Column) | 칼럼 (Column) | 데이터의 종류 — 이름, 나이, 이메일 등 |
| 행 (Row) | 레코드 (Record/Row) | 실제 데이터 한 줄 — 한 사람의 정보 |

그림으로 보면 더 쉽습니다. 독서 모임 서비스를 예로 들어볼게요.

**"책" 시트 (= 테이블):**

| 제목 (string) | 저자 (string) | 출판년도 (integer) | 장르 (string) | 추천 (boolean) |
|---------------|--------------|-------------------|--------------|---------------|
| 해리포터 | J.K. 롤링 | 1997 | 판타지 | true |
| 어린왕자 | 생텍쥐페리 | 1943 | 소설 | true |
| 코스모스 | 칼 세이건 | 1980 | 과학 | false |

이게 데이터베이스의 전부입니다. **엑셀 시트에 데이터를 정리하는 것과 같습니다.**

엑셀에서 시트를 여러 개 만들 수 있듯이, 데이터베이스에도 테이블을 여러 개 만들 수 있습니다. "책" 시트, "감상" 시트, "모임" 시트 — 이런 식으로요.

### 칼럼 타입: 데이터의 종류를 정하기

엑셀에서 "숫자" 칸에 글자를 쓰면 오류가 나듯, 데이터베이스도 각 칼럼에 "이 칸에는 어떤 종류의 데이터가 들어가나" 를 정합니다.

5가지만 알면 됩니다:

| 타입 | 한국어로 | 예시 | 언제 쓰나 |
|------|---------|------|----------|
| `string` | 짧은 글 | 이름, 제목, 이메일, 카테고리 | 한 줄짜리 텍스트 |
| `text` | 긴 글 | 설명, 후기, 본문, 메모 | 여러 줄이 필요한 텍스트 |
| `integer` | 숫자 | 나이, 수량, 가격, 평점 | 계산이 필요한 숫자 |
| `boolean` | 예/아니오 | 공개 여부, 완료 여부, 추천 여부 | 둘 중 하나만 고르는 것 |
| `datetime` | 날짜+시간 | 생성일, 마감일, 예약일 | 날짜/시간이 필요한 것 |

고르는 법도 간단합니다:
- "이 데이터가 한 줄이면 충분해?" → **string**
- "여러 줄이 필요해?" → **text**
- "숫자야?" → **integer**
- "예/아니오 둘 중 하나야?" → **boolean**
- "날짜나 시간이야?" → **datetime**

### 모델(Model) = 엑셀 시트의 설계도

Rails에서 **모델(Model)**은 "이 시트에 어떤 열이 있는지" 정의하는 것입니다.

엑셀에서 새 시트를 만들고 첫 행에 "제목, 저자, 출판년도, 장르"라고 열 제목을 적는 것 — 이것이 모델 정의입니다.

### 마이그레이션(Migration) = 설계도를 실행하기

마이그레이션은 "이런 시트를 만들어주세요"라는 설계도입니다.

비유:
1. 건축 설계도를 그린다 → **마이그레이션 파일 생성** (아직 건물은 없음)
2. 시공사에게 설계도를 넘긴다 → **마이그레이션 실행** (실제 건물이 세워짐)

설계도(마이그레이션)를 만드는 것과 실행하는 것은 별개입니다. Claude가 설계도를 만들면, 여러분이 "실행해줘"라고 한 번 더 요청해야 실제 테이블이 만들어집니다.

### 이번 레슨에서 만드는 것

두 가지를 만듭니다:

**1. User 모델** (회원 테이블) — OAuth 로그인을 위한 최소 구조
- 칼럼: `email` (string), `name` (string), `avatar_url` (string, nullable)
- OAuth(구글/카카오)가 주는 정보만 저장. 비밀번호 칼럼은 없음.
- 다음 레슨(`/v3-auth`)에서 OAuth를 연결할 때 이 테이블을 사용.

**2. 핵심 콘텐츠 모델** — 서비스의 주인공 데이터
- 독서 모임이면: "책"
- 레시피 앱이면: "레시피"
- 할 일 관리면: "할 일"
- 학습자 서비스 기준으로 결정.

두 모델을 순서대로 만든 후, 콘솔에서 CRUD를 확인합니다.

## EXECUTE

### 단계 1: 내 서비스의 데이터 구조 설계 — "엑셀 시트 설계"

C2 기획서를 참고하여, 내 서비스에 어떤 데이터가 필요한지 Claude와 함께 정리합니다.

Claude에게 이렇게 요청하세요:
```
CLAUDE.md를 읽고, 내 서비스의 핵심 기능을 확인해줘.

이 서비스에 필요한 데이터 구조를 엑셀 시트처럼 정리해줘.
각 시트(테이블)마다:
- 시트 이름
- 열 제목들 (칼럼)
- 각 열의 타입 (짧은글/긴글/숫자/예아니오/날짜)
- 샘플 데이터 2~3줄

엑셀 표 형태로 보여줘. 비개발자도 이해할 수 있게.

단, User(회원) 테이블은 제외해줘. 그건 다음 레슨에서 할 거야.
```

Claude가 엑셀 형태로 데이터 구조를 보여줍니다.

예를 들어 레시피 서비스라면:

**"레시피" 시트:**
| 제목 (string) | 소개 (text) | 조리시간 (integer) | 난이도 (string) | 공개 (boolean) |
|--------------|------------|-------------------|---------------|---------------|
| 김치찌개 | 집밥의 정석... | 30 | 쉬움 | true |
| 파스타 | 간단한 크림... | 20 | 보통 | true |

이해가 안 되는 부분이 있으면 바로 물어보세요:
```
"조리시간"이 왜 integer야? string이면 안 돼?
```
```
테이블을 하나 더 추가하면 어떻게 돼?
```

> 꿀팁: 이 단계에서 완벽할 필요 없습니다. 나중에 칼럼을 추가하거나 변경할 수 있습니다. "일단 핵심적인 것만 넣자"가 좋은 전략입니다.

### 단계 2: 데이터 구조 확정

Claude와 대화하면서 데이터 구조를 확정합니다.

확인할 것:
- 이 서비스의 핵심 데이터가 맞나요?
- 빠진 칼럼은 없나요? (제목, 설명 등)
- 불필요한 칼럼은 없나요? (너무 많으면 복잡해짐)
- 칼럼 타입이 맞나요? (이름은 string, 설명은 text 등)

확정되면:
```
좋아, 이 구조로 가자. 먼저 [핵심 테이블명] 모델부터 만들어줘.
```

### 단계 3: 첫 번째 모델 생성 — "엑셀 시트 만들기"

가장 핵심적인 데이터 테이블 1개를 먼저 만듭니다.

Claude에게 이렇게 요청하세요:
```
[핵심 테이블명] 모델을 Rails로 만들어줘.
칼럼은 방금 정한 것으로 해줘.
```

Claude가 모델과 마이그레이션 파일을 생성합니다.

화면에 이런 것들이 나타납니다:
- `app/models/[모델명].rb` — 모델 파일 (시트의 설계도)
- `db/migrate/xxxxx_create_[모델명s].rb` — 마이그레이션 파일 (실행 대기 중인 설계도)

> 아직 실제 테이블은 만들어지지 않았습니다! 설계도만 만든 상태입니다.

### 단계 4: 마이그레이션 실행 — "설계도 → 실제 건물"

Claude에게 이렇게 요청하세요:
```
마이그레이션을 실행해줘. 실제 데이터베이스에 테이블을 만들어줘.
```

Claude가 `rails db:migrate` 명령을 실행합니다.

"== CreateXxx: migrating" → "== CreateXxx: migrated" 메시지가 나오면 성공입니다!

**축하합니다!** 엑셀 시트가 실제로 만들어졌습니다. 이제 여기에 데이터를 넣을 수 있습니다.

> 에러가 나면: 에러 메시지를 Claude에게 그대로 보여주세요. 마이그레이션 에러는 대부분 칼럼 이름이나 타입의 오타입니다. Claude가 고쳐줍니다.

### 단계 5: Rails 콘솔에서 데이터 다뤄보기 — "엑셀 시트에 첫 행 추가!"

이제 실제 데이터를 넣어보고 확인합니다. 엑셀에서 행(row)을 추가하는 것과 같습니다.

Claude에게 이렇게 요청하세요:
```
Rails 콘솔에서 [모델명] 데이터를 다뤄보여줘.
비개발자도 이해할 수 있게 각 단계를 설명해줘.

1. 데이터 하나 만들기 (엑셀에 새 행 추가)
2. 전체 데이터 보기 (엑셀에서 시트 전체 보기)
3. 데이터 하나 찾기 (엑셀에서 특정 행 찾기)
4. 데이터 수정하기 (엑셀에서 셀 값 변경)
5. 데이터 삭제하기 (엑셀에서 행 삭제)
```

Claude가 Rails 콘솔에서 하나씩 보여줍니다:

**1. 데이터 만들기** — 엑셀의 새 행 추가
```ruby
Recipe.create(title: "김치찌개", description: "집밥의 정석", cooking_time: 30)
```
→ "김치찌개" 데이터가 시트에 한 줄 추가!

**2. 전체 보기** — 엑셀의 시트 전체 보기
```ruby
Recipe.all
```
→ 시트에 있는 모든 데이터가 보임!

**3. 하나 찾기** — 엑셀에서 특정 행 찾기
```ruby
Recipe.find(1)
```
→ 1번 데이터의 상세 내용이 보임!

**4. 수정하기** — 엑셀에서 셀 값 변경
```ruby
recipe = Recipe.find(1)
recipe.update(title: "특제 김치찌개")
```
→ 제목이 "김치찌개"에서 "특제 김치찌개"로 바뀜!

**5. 삭제하기** — 엑셀에서 행 삭제
```ruby
Recipe.find(1).destroy
```
→ 1번 데이터가 시트에서 사라짐!

이 다섯 가지 동작(만들기, 보기, 찾기, 수정, 삭제)을 **CRUD**라고 부릅니다:
- **C**reate (만들기)
- **R**ead (읽기/보기)
- **U**pdate (수정하기)
- **D**elete (삭제하기)

모든 서비스의 핵심은 이 CRUD입니다. 배달의민족도, 인스타그램도, 넷플릭스도 결국 데이터를 만들고, 보고, 수정하고, 삭제하는 것입니다.

### 단계 6: (선택) 두 번째 모델 + 관계 설정

서비스에 테이블이 2개 이상 필요하다면, 두 번째 모델도 만듭니다.

Claude에게 이렇게 요청하세요:
```
[두번째 테이블명] 모델도 만들어줘.
마이그레이션도 실행해줘.

[첫번째 모델]과 [두번째 모델]의 관계도 설정해줘.
엑셀로 비유해서 관계를 설명해줘.
```

**관계(Association)란?** 엑셀에서 "시트 A의 데이터가 시트 B와 연결되는 것"입니다.

예:
- "레시피" 시트의 "김치찌개"에 → "재료" 시트의 "김치, 두부, 돼지고기"가 연결됨
- "책" 시트의 "해리포터"에 → "감상" 시트의 "재밌었다, 감동적이었다"가 연결됨

엑셀에서 VLOOKUP이나 참조로 시트를 연결하는 것과 비슷한 개념입니다.

### 단계 7: 데이터 구조 확인

Claude에게 이렇게 요청하세요:
```
지금 만든 데이터베이스 구조를 정리해서 보여줘.
엑셀 시트 형태로, 각 시트의 칼럼과 타입을 표로 보여줘.
```

이 표를 보고 "내 서비스의 데이터가 이렇게 정리되는구나" 확인합니다.

### 단계 8: Git 세이브 + CLAUDE.md 업데이트

Claude에게 이렇게 요청하세요:
```
변경사항을 Git에 저장해줘. 메시지는 "데이터베이스 모델 생성 완료"로.
CLAUDE.md의 현재 상태도 "v3-database 완료. [모델명] 모델 생성."으로 업데이트해줘.
CLAUDE.md에 데이터 구조 섹션도 추가해줘 — 어떤 모델에 어떤 칼럼이 있는지.
```

> CLAUDE.md에 데이터 구조를 기록해두면, 나중에 Claude가 "이 서비스의 데이터 구조는 이렇습니다"라고 바로 파악할 수 있습니다.

- 3번째 질문 → "설계도를 실행해서 실제 테이블을 만드는 것"

해설:
- 데이터베이스의 테이블은 엑셀의 시트와 같습니다. 각 시트에 열(칼럼)과 행(데이터)이 있듯이, 테이블에도 칼럼과 레코드가 있습니다. 통합 문서(워크북)가 데이터베이스 전체에 해당합니다.
- 사용자 이름은 "홍길동", "김철수"처럼 짧은 텍스트이므로 string(짧은 글)이 적절합니다. text(긴 글)는 후기나 설명처럼 여러 줄이 필요한 경우, integer(숫자)는 나이나 가격처럼 숫자인 경우, boolean(예/아니오)은 공개 여부처럼 둘 중 하나인 경우에 사용합니다.
- 마이그레이션은 "이런 테이블을 만들어줘"라는 설계도를 실행하는 것입니다. 건축 설계도를 시공사에게 넘기면 건물이 세워지듯, 마이그레이션을 실행하면(rails db:migrate) 실제 데이터베이스에 테이블이 만들어집니다.
