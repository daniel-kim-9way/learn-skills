## EXPLAIN

### 가입 버튼 한 번 누르면 뭐가 일어날까? 🔐

1-1에서 큰 그림(브라우저 → 서버 → DB)을 봤어요. 이번엔 그 흐름의 **가장 중요한 한 조각** — "가입 버튼을 누른 순간 데이터가 어떻게 저장되는가" — 를 자세히 따라가요.

이걸 한 번 손에 잡으면 Ch.4 BUILD에서 회원가입·로그인·OAuth 만들 때 거의 막히지 않아요. 머릿속에 그림이 있으니까요.

이번 레슨에서 다룰 3가지 핵심:

1. **DB 표는 사실 엑셀 시트와 같아요.** 새 사용자 = 새 행 추가. 그게 다.
2. **비밀번호는 절대 평문(원문)으로 저장하면 안 돼요.** 해시(bcrypt) 변환이 필수.
3. **"구글로 로그인" (OAuth)은 비번을 우리가 직접 안 받아요.** 더 안전.

### 진행 방식

총 6개 Turn으로 진행해요. 한 Turn당 한 개념씩, 비유 + 시뮬레이션 + 본인 답으로.

## EXECUTE

### Turn 1: 가입 버튼 클릭 → 7단계 시뮬레이션

> "1-2 시작합니다! 1-1에서 큰 그림(브라우저 → 도메인 → 서버 → DB)을 봤죠?"
>
> "이번에는 그 흐름의 **가장 중요한 한 조각** — '가입 버튼을 누른 순간'을 7단계로 천천히 따라가요."
>
> "본인이 본인 SaaS에 새로 들어온 사용자라고 상상해 봐요. 이메일 'test@a.com'과 비밀번호 'hello1234'를 입력하고 가입 버튼을 클릭해요."

이어서 7단계를 한 단계씩 천천히 펼침:

```
[가입 버튼 클릭 → 7단계]

1. 사용자가 폼에 입력 + "가입" 클릭
   화면 보이는 정보:
     이메일:    test@a.com
     비밀번호:  hello1234

2. 브라우저가 그 데이터를 본인 서버로 전송 (HTTP POST 요청)
   인터넷 구간은 HTTPS로 암호화돼서 중간에 누가 봐도 못 알아봄

3. 본인 서버가 데이터 검증
   - 이메일 형식 맞아? @ 들어갔어?
   - 비밀번호 길이 8자 이상?
   - 같은 이메일로 이미 가입된 사람 있어?

4. 비밀번호를 해시(bcrypt)로 변환
   "hello1234" → "$2a$12$Kix3jL.PqrM...8bN3vXz"  (긴 암호문)
   → 이게 핵심. 원문은 버려요.

5. DB의 users 표에 새 행 추가
   id | email      | password_digest        | created_at
   1  | test@a.com | $2a$12$Kix3jL.PqrM...  | 2026-05-05 10:30

6. 서버가 "가입 성공" 응답을 브라우저로
   + 로그인 세션도 같이 발급 (다음 페이지부터 "이 사람 로그인됨" 인식)

7. 브라우저가 환영 페이지로 이동
   사용자는 "축하합니다! 가입 완료" 메시지 확인
```

> "여기서 한 가지 깨달음 — **비밀번호 'hello1234'는 어디에도 평문으로 저장 안 됨**. 4번에서 해시로 변환되고 원문은 버려져요."
>
> "왜 그렇게 할까요? 한번 추측해 보세요."

- **"몰라요"** → "정상이에요. 다음 Turn에서 풀어드릴게요."

### Turn 2: DB 표 = 엑셀 시트

> "그 5번 단계 — '**DB의 users 표에 새 행 추가**' — 가 비개발자가 가장 두려워하는 부분이에요. 'DB?' 'SQL?' '스키마?' 어렵게 들리죠."
>
> "그런데 사실 **DB는 엑셀 시트와 거의 똑같아요**. 진짜로요."

DB와 엑셀 비교 표 출력:

```
[엑셀 시트 - 회원 명단]                  [DB 표 - users]

   A          B               C            id | email      | name
1  이메일     이름            가입일       1  | a@a.com    | 한규민
2  a@a.com    한규민          2026-05-05    2  | b@b.com    | 김영희
3  b@b.com    김영희          2026-05-05    3  | c@c.com    | 이철수
4  c@c.com    이철수          2026-05-05
                                            ↑      ↑           ↑
   ↑          ↑               ↑           열(column)
   행(row)이 사용자 한 명                  행(row) = 사용자 한 명
```

핵심:
> "다음만 기억하면 됩니다:"
> - **행(row) = 사용자 한 명**. 새 가입 = 새 행 추가.
> - **열(column) = 그 사용자에 대한 항목**. email, name, created_at 등.
> - **id** = 자동으로 1, 2, 3 부여되는 고유 번호. "이 사용자를 가리킬 때" 사용.
> - **created_at** = 가입한 시각. DB가 자동으로 채움.
> - **password_digest** = 비밀번호 원문이 아닌 **해시(암호화 결과)**. 절대 원문 저장 X.
>
> "엑셀과 차이점은 단 3가지: ① 영구 저장 (전원 꺼져도 안 사라짐) ② 빠른 검색 (사용자 100만명도 0.1초) ③ 동시 접속 안전 (여러 명이 동시에 가입해도 OK)."

이어서 질문:
> "본인 SaaS에는 어떤 표들이 필요할 것 같아요? users 표는 거의 모든 SaaS에 있어요. 그 외에 어떤 표가 더 있을까요?"
>
> "예시:"
> - 독서 노트 SaaS → `users` + `books` + `notes`
> - 스터디 매칭 → `users` + `studies` + `applications`
> - 일정 관리 → `users` + `events` + `categories`

학생 답변을 받아 본인 SaaS의 표 후보를 1-3개 정리. (Turn 6에서 파일에 저장)

### Turn 3: 비밀번호는 왜 평문 저장 X? — 해시

> "Turn 1에서 4번 단계 기억나요? **비밀번호를 해시로 변환**하는 단계."
>
> "왜 원문 그대로 저장하면 안 될까요?"

학생 답변 후:
> "정답: **DB가 만약 누군가에게 도난당하면(해킹) 큰일나요.**"
>
> "이런 상황을 상상해 봐요. 본인 SaaS에 1만 명이 가입했어요. 그런데 어느 날 본인 DB가 통째로 유출됐어요. 그 안에 사용자들의 평문 비밀번호가 다 있다면?"
>
> "그 사용자들은 다른 사이트(네이버·카카오·은행)에서도 같은 비번을 쓰고 있을 가능성이 높아요. 즉 본인 서비스 한 번 털리면 → 그 사람들의 모든 사이트가 털림. 본인 책임 어마어마해요."
>
> "그래서 모든 SaaS는 비밀번호를 **해시(hash)**로 변환해서 저장해요."

해시 시뮬레이션 출력:

```
[해시 = 도장 비유]

사용자 입력:        "hello1234"   (평문 = plain text)
                          ↓
                  bcrypt 해시 함수
                          ↓
DB에 저장되는 것:   "$2a$12$Kix3jL.PqrM...8bN3vXz"
                   (긴 암호문 — 60자 정도)

도장의 특징:
  ✓ 종이(평문) → 도장(해시) 가능
  ✗ 도장(해시) → 종이(평문) 불가능 (= 일방향)
  ✓ 같은 종이 + 같은 도장 → 같은 도장 자국 (검증 가능)
```

해시의 3가지 핵심 특성:
> 1. **일방향**: 평문 → 해시는 가능. 하지만 해시 → 평문은 불가능. **본인도 사용자 비번을 알 수 없어요.**
>    → 그래서 사용자가 비번 잊으면 "재설정 메일"이 필요한 거예요. 보내드릴 수가 없으니까.
>
> 2. **매번 약간 다른 결과 (salt)**: 같은 'hello1234'라도 매번 약간 다른 해시가 나와요.
>    → DB만 봐도 "어 이 두 사람 같은 비번 쓰네?" 알 수 없음.
>
> 3. **검증은 가능**: 로그인 시 사용자가 입력한 평문을 다시 해시해서, DB의 해시와 비교. 일치하면 로그인 성공.

> "가장 많이 쓰이는 해시 알고리즘은 **bcrypt**예요. Rails에서는 `has_secure_password` 한 줄로 자동 적용. AI에게 '회원가입 만들어줘' 한마디면 알아서 bcrypt 써요. 본인이 직접 알고리즘 짤 일 없음."

질문:
> "여기까지 따라오셨어요. 한 가지 확인할게요. **'본인이 직접 사용자 비번을 평문으로 저장하지 않는 이유'**를 본인 입으로 한 줄 설명해 보세요."

학생 답변 듣고 피드백.

### Turn 4: OAuth — "구글로 로그인"이 왜 더 안전?

> "비번을 본인이 직접 받는 방식은 안전하게 만들 수 있어요. 하지만 책임이 무거워요."
>
> "그래서 요즘은 **OAuth**를 많이 써요. 익숙한 표현으로는 **'구글로 로그인' / '카카오로 로그인' / '애플로 로그인'**이에요."

OAuth 핵심 비유:
> "비유: 본인이 클럽 문 앞 보안요원이라고 해봐요."
>
> "**자체 가입 방식**: 본인이 손님의 신분증을 직접 받아서, 진위 확인하고, 보관까지 책임짐. 분실하면 본인 책임.
>
> **OAuth 방식**: '저기 옆 정부 발급기관(=구글)에서 신분증 검사받고 도장 받아오세요' 한 다음, 도장만 확인. 신분증 자체는 본인이 안 받음."

OAuth 흐름 출력:

```
[OAuth 가입 흐름 — 구글 예시]

1. 사용자가 "구글로 로그인" 버튼 클릭
2. 본인 서버 → 구글 로그인 페이지로 이동시킴
3. 사용자가 구글 페이지에서 "본인 SaaS에 내 정보 줘도 됨" 승인
4. 구글이 본인 서버에 토큰(도장) + 사용자 이메일 전달
5. 본인 서버가 DB에 저장:
     users 표:
     id | email      | google_uid    | created_at
     1  | test@a.com | google_xyz123 | 2026-05-05

   ⚠️ password_digest 열이 없어요!
   비번을 받지 않으니까 저장할 게 없음.
```

> "정리:"
> - **자체 회원가입 (이메일+비번)**: 비번 안전 저장, 분실 처리, 보안 사고 모두 본인 책임.
> - **OAuth (구글/카카오)**: 비번은 구글이 검증, 본인은 "이 사람 진짜다" 도장만 확인. 책임이 훨씬 가벼움.
>
> "그래서 비개발자가 처음 SaaS를 만들 때는 **OAuth부터 붙이는 걸 강추**해요. 코드 더 짧고, 안전하고, 사용자 가입도 빠름. Ch.4-8에서 진짜 붙여요."

질문:
> "본인 SaaS는 자체 비번? OAuth? 어떻게 하실 건가요?"
> - "(추천) OAuth — 구글 또는 카카오"
> - "둘 다 — OAuth + 이메일/비번 둘 다 제공"
> - "자체 비번만 — 특별한 이유가 있어서"
> - "아직 모름 — 나중에 결정"

학생 답변 받기.

### Turn 5: 로그인 후 어떻게 '나'를 알아봐? — Session / JWT

> "한 가지만 더 풀고 마무리해요."
>
> "사용자가 가입 후 로그인했어요. 그 다음 페이지 — 예를 들어 본인 프로필 페이지 — 에 들어갔을 때, **서버는 이 사람이 'test@a.com'이라는 걸 어떻게 알까요?**"
>
> "매번 비번을 다시 입력시킬 순 없잖아요?"

학생 답변 후:
> "정답: **서버가 '이 사람은 test@a.com 맞다'는 도장을 발급해서 브라우저에 저장**시켜요. 그 도장을 들고 다니면 다음부터 본인이 누군지 인식돼요."
>
> "그 도장을 두 가지 방식 중 하나로 구현해요:"

```
[방식 1: Session (서버 저장)]
- 서버가 "user_id=1" 같은 짧은 표를 본인 메모리/DB에 만듦
- 브라우저에는 "session_id=abc123" 쿠키 하나만 줌
- 다음 요청 때 세션 ID로 서버에서 user_id를 찾아냄
- 비유: 클럽 입구에서 손목 스탬프 받음. 다음에 다시 들어올 때 스탬프 보여주면 됨.

[방식 2: JWT (브라우저에만 저장)]
- 서버가 "user_id=1" 정보를 암호화한 긴 문자열(JWT) 만들어 줌
- 그걸 브라우저가 LocalStorage 또는 쿠키에 저장
- 다음 요청 때 JWT를 같이 보내면 서버가 검증
- 비유: 위조 불가능한 입장권. 본인이 들고 다니면 됨.
```

> "어느 쪽이 더 좋다고 하기 어려워요. 보통 Rails는 Session이 기본, React/모바일 앱은 JWT가 흔함. AI에게 맡기면 알아서 적절히 선택."
>
> "지금은 '아 로그인 후에는 서버가 도장 발급해서 본인을 인식한다' 정도만 머리에 두면 충분해요."

> "한 줄씩 답해주세요:"

학생 답변을 받아 본인 SaaS 맞춤 DB 스키마 초안을 텍스트로 출력 (예시 — 독서 노트 SaaS):

```
[본인 SaaS DB 스키마 초안 — 예시]

users 표
  id              (자동 1, 2, 3...)
  email           (test@a.com)
  name            (한규민)
  google_uid      (OAuth 식별자, OAuth 쓰면)

  book_id         → books.id 참조

  created_at

[관계도]
  users (1) ─── (N) books (1) ─── (N) notes
   "한 사람이 여러 책 / 한 책에 여러 노트"
```

2. 브라우저가 데이터를 서버로 전송 (HTTPS 암호화)
3. 서버가 검증 (이메일 형식, 비번 길이, 중복 체크)
4. 비밀번호 → 해시(bcrypt) 변환 (원문 버림)
5. DB의 users 표에 새 행 추가
6. 서버가 "가입 성공" 응답 + 세션/JWT 발급
7. 브라우저가 환영 페이지로 이동
