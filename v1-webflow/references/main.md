## EXPLAIN

### 웹 서비스의 5부품을 한 장으로 그려봐요 🗺️

이번 레슨은 Ch.1의 첫 번째 — 가장 중요한 큰 그림을 잡는 레슨이에요.

Ch.0에서 만든 자기소개 페이지(`index.html`) — 이건 사실 **"웹 서비스"가 아니에요**. 그냥 한 장짜리 정적 파일이에요. 누가 카톡으로 받아서 열거나, 본인이 PC에서 더블클릭하면 보이는 단순한 HTML 한 장.

그런데 진짜 SaaS는 다음 3가지가 가능해야 해요:
1. 사람들이 **가입**해요
2. 본인 **데이터를 저장**해요
3. 나중에 다시 들어와서 **본인 데이터를 보거나 수정**해요

이게 가능하려면 HTML 한 장으로는 안 돼요. 여러 부품이 협력해야 하죠. 이 레슨에서는 그 부품들이 무엇인지, 어떻게 연결되는지 — **본인 머리에 한 장의 지도**로 그려놓을 거예요.

### 왜 이게 먼저 필요한가?

Ch.2~Ch.5에서 본인 SaaS를 진짜 만들 때, 부품 개념을 모르면 **"지금 이걸 왜 만드는 거지?"**가 계속 나와요.
미리 한 장의 지도가 머리에 있으면, Ch.4 BUILD에서 "아 이건 DB 부품이고, 이건 서버 부품이고…" 자연스럽게 잡혀요.

이번 레슨이 끝나면:
- 0-3 자기소개 페이지에 **"가입 버튼"을 붙이면 어디 어디가 추가로 필요**한지 안다
- 웹 서비스의 5가지 핵심 부품(브라우저·도메인·서버·DB·배포)을 본인 입으로 설명할 수 있다
- HTTP / API / DNS / OAuth 같은 영어 약어 30개를 "아 이거 그거구나" 정도로 안다

### 진행 방식

총 7개 Turn으로 진행해요. 각 Turn은 짧고, 본인 답을 받아서 다음 단계로 넘어가요. **"잘 모르겠어요"가 가장 정상적인 답**이에요. 그러면 더 쉬운 비유로 다시 풀어줄게요.

## EXECUTE

### Turn 1: 0-3 자기소개 페이지에 가입 기능을 붙이면?

> "안녕하세요! Ch.1 첫 번째 레슨입니다."
>
> "Ch.0에서 만든 자기소개 페이지(`index.html`) 기억나시죠? 그 한 장짜리 HTML 파일."
>
> "그 페이지에 이번에 **'회원가입' 기능을 붙인다**고 상상해 봐요. 사용자가 페이지에 와서 이메일·비밀번호를 입력하고 '가입' 버튼을 누른다고요."
>
> "첫 질문 — **그 입력값(이메일·비밀번호)은 어디에 저장될까요?** 본인 생각대로 답해주세요. '잘 모르겠어요'도 좋은 답이에요."

학생 답변 패턴별 분기:

- **"잘 모르겠어요" / "어디 저장?"** → "정상이에요! 사실 HTML 파일 안에는 저장할 곳이 없어요. 누가 새로고침하면 다 사라지죠. 그래서 **'다른 곳'에 저장해야 한다**는 게 핵심 깨달음이에요. 그 '다른 곳'이 바로 다음 부품이에요."
- **"데이터베이스에?"** → "정확해요! 이미 감이 있으시네요. 그런데 데이터베이스는 어디 있을까요? 본인 PC 안? 인터넷 어딘가?"
- **"브라우저에?" / "쿠키에?"** → "쿠키에도 저장할 수 있긴 한데, 그러면 다른 PC에서 로그인하면 사라져요. 사용자별로 영구 저장하려면 다른 곳이 필요해요. 어디일까요?"

핵심 메시지 요약:
> "정답: 입력값은 **'서버'라는 다른 컴퓨터로 보내고, 거기서 'DB'라는 영구 저장소에 들어가요.** 부품이 두 개 추가됐죠? 서버 + DB."

---

### Turn 2: 그 '서버'는 어떻게 찾아가요? — 도메인·DNS

> "좋아요. 입력값이 '서버'로 간다고 했어요."
>
> "그런데 사용자의 브라우저는 그 서버를 **어떻게 찾아갈까요?** 인터넷에는 컴퓨터가 수억 개 있는데."
>
> "예를 들어 본인이 친구한테 '내 자기소개 페이지 봐봐' 한다면, 친구한테 뭘 알려주죠?"

학생 답변:
- **"링크?" / "URL?"** → "맞아요! `https://본인이름.com` 같은 URL을 알려주죠. 그 URL의 핵심 부분을 **도메인(Domain)**이라고 해요. 예: `vuild.kr`, `naver.com`."
- **"IP 주소?"** → "기술적으로는 맞아요. 사실 컴퓨터끼리는 IP 주소(예: 1.2.3.4)로 통신해요. 하지만 사람이 외우기 어렵죠. 그래서 도메인 → IP 변환을 자동으로 해주는 게 **DNS**예요."

설명:
> "정리하면 두 부품이 더 추가됐어요:"
> - **도메인 (Domain)**: 본인 서비스의 주소 이름. `myservice.kr` 같은 것. 가비아 / Namecheap에서 연 1~2만 원에 구입.
> - **DNS (Domain Name System)**: 도메인을 실제 서버 IP 주소로 변환해주는 시스템. **"네이버 지도"** 같은 역할 — "vuild.kr라는 간판을 찾아가려면 IP 1.2.3.4로 가세요" 알려줌.
>
> "비유로 설명할게요. 본인이 친구한테 식당을 추천한다고 해봐요. '강남역 2번 출구 앞에 있는 BB식당' 이렇게 알려주죠? 그 '강남역 2번 출구'가 도메인, '서울시 강남구 역삼동 123-4번지'가 IP, 그리고 그 둘을 연결해주는 네이버 지도가 DNS예요."

---

### Turn 3: 서버는 받은 데이터를 어디에 저장? — DB

> "이제 사용자의 입력값이 도메인 → DNS → 서버까지 도착했어요."
>
> "서버는 그 데이터를 **어디에 저장**할까요?"

학생 답변:
- **"DB? 데이터베이스?"** → "맞아요! 서버는 받은 데이터를 곧바로 **DB(Database)**에 저장해요."
- **"서버 안에?"** → "기술적으로는 그래요. DB는 보통 서버 안에 또는 옆에 있어요. 그런데 DB는 단순한 폴더가 아니라 '구조화된 표' 형태예요. 엑셀 시트랑 거의 같아요."

설명:
> "DB는 **엑셀 시트와 거의 똑같아요**. 다음 lesson(1-2)에서 자세히 봐요. 지금은 이 정도만 기억:"
> - **users 표** = 행 한 개가 사용자 한 명. 가입할 때마다 새 행이 추가됨.
> - **books 표 / orders 표** = SaaS 종류에 따라 다른 표가 추가됨.
> - 영구 저장 + 빠른 검색 + 사용자별 분리 — 이게 DB의 핵심.

> "비유: 서버 = **주방**, DB = **냉장고·창고**. 주방(서버)이 받은 주문을 처리해서 결과를 냉장고(DB)에 보관."

---

### Turn 4: 외부 서비스(예: 구글 로그인)도 같은 방식 — API

> "여기서 잠깐. 만약 본인이 사용자의 이메일·비밀번호를 직접 받지 않고, **'구글로 로그인' 버튼**을 넣고 싶다면?"
>
> "이때는 본인 서버 + DB 외에 또 한 부품이 필요해요. 그게 뭘까요?"

학생 답변 후:
> "정답은 **외부 서비스 호출 = API**예요."
> - **API (Application Programming Interface)** = 외부 회사의 서비스에 "일 좀 시켜주세요" 요청 보내고 결과 받는 방식.
> - 구글 로그인을 붙이면 본인 서버가 → 구글에 "이 사용자가 진짜 a@a.com 맞아?" 물어봄 → 구글이 "맞아" 답하면 → 본인 DB에는 "이 사람 구글 인증 받음"만 저장.
>
> "비밀번호 검증은 구글이 다 해주니까, 본인은 비번 안 받아도 됨. 더 안전. 다음 lesson(1-2)에서 OAuth로 자세히 봐요."

핵심:
> "결제(토스페이먼츠), AI 호출(Claude API), 카톡 알림 — **전부 같은 패턴**이에요. '외부 서비스에 일 시키고 결과 받기.' 1-5에서 깊게 들어가요."

---

### Turn 5: 이 모든 게 동작하려면 — 배포·호스팅

> "지금까지 4개 부품이 모였어요: 브라우저 / 도메인·DNS / 서버 / DB."
>
> "그런데 한 가지 빠진 게 있어요. **본인이 작성한 코드가 어떻게 그 '서버'에 올라가서 24시간 돌아갈까요?**"
>
> "본인 PC가 24시간 켜져 있어야 하나요?"

학생 답변:
- **"PC를 켜놓으면?"** → "이론적으로 가능하지만, 인터넷도 끊기고 전기도 나가고… 실서비스는 못 돌려요. 그래서 **남의 컴퓨터를 빌려요**. 그게 호스팅 / 클라우드예요."
- **"클라우드 같은 거?"** → "정확해요!"

설명:
> "마지막 부품: **배포(Deploy) + 호스팅(Hosting)**"
> - **호스팅 회사**: Railway / Render / Vercel / AWS / Heroku — 24시간 돌아가는 컴퓨터를 빌려주는 회사. 월 $5~20 수준부터 시작.
> - **배포(Deploy)**: 본인이 작성한 코드를 그 호스팅 서버에 올려서 실제 서비스가 되게 하는 작업. `git push` + 배포 명령 한 번이면 됨.
>
> "비유: 호스팅 = **식당 자리 임대**, 배포 = **새 메뉴/인테리어를 매장에 적용**."

### Turn 6: 5부품 정리 + 영어 약어 30개

> "축하해요. 본인이 방금 웹 서비스의 5부품을 다 짚었어요. 정리해 볼게요."

5부품 표 출력:

```
① 브라우저 (Frontend)        — 사용자가 보는 화면. Chrome/Safari/카톡 인앱.
   비유: 식당의 테이블 + 메뉴판

② 도메인 (Domain + DNS)      — vuild.kr 같은 주소. DNS가 "주소 → IP" 변환.
   비유: 식당 간판 + 네이버 지도

③ 서버 (Backend)             — 24시간 돌아가는 컴퓨터. 가입 처리, 로그인 검증.
   비유: 식당의 주방 + 요리사

④ DB (Database)              — users·books·orders 같은 표를 영구 저장.
   비유: 냉장고 + 창고

⑤ 배포 (Deploy + Hosting)    — 본인 코드를 서버에 올림. Railway/Render/Vercel.
   비유: 새 메뉴/인테리어를 매장에 반영
```

가입 버튼 클릭 시 7단계 흐름:
```
1. 사용자가 브라우저에 vuild.kr 입력
   → DNS가 "그 주소는 IP 1.2.3.4야" 알려줌
2. 브라우저가 그 IP로 접속
   → 서버가 가입 폼 HTML 응답
3. 사용자가 이메일·비밀번호 입력 후 "가입" 클릭
   → 브라우저가 데이터를 서버로 전송 (HTTP POST)
4. 서버가 데이터 검증 (이메일 형식? 비번 길이?)
   → 비밀번호를 해시(암호화)로 변환
5. 서버가 DB에 저장 (users 표에 새 행 추가)
6. 서버가 "가입 성공!" 응답 → 브라우저가 환영 페이지로
7. 사용자는 즉시 로그인 가능
```

이어서 영어 약어 30개 표 출력:

```
[웹 통신]
HTTP        — HyperText Transfer Protocol. 웹 데이터 주고받는 약속(언어).
HTTPS       — HTTP + Secure. 자물쇠(암호화) 추가.
REST API    — REpresentational State Transfer API. 서버에 일 시키는 표준 방식.
JSON        — JavaScript Object Notation. 데이터 텍스트 형식 (key:value).
GET / POST  — HTTP 메서드. GET=가져와줘, POST=보내요.

[부품 이름]
Frontend    — 사용자가 보는 화면 영역. 식당의 객장.
Backend     — 서버 쪽 처리 영역. 식당의 주방.
DB          — Database. 데이터 영구 저장소.
SQL         — Structured Query Language. DB에 묻는 언어.
ORM         — Object-Relational Mapping. SQL을 한국어처럼 쓰게 하는 통역.
Server      — 서버. 24시간 돌아가는 다른 컴퓨터.

[주소·접근]
Domain      — 도메인 (vuild.kr). 식당 간판.
DNS         — Domain Name System. 간판 → IP 변환. 네이버 지도.
Hosting     — 호스팅. 서버 컴퓨터 임대.
Cloud       — 클라우드 (AWS, Vercel, Railway). 남의 서버를 인터넷으로 빌림.
CDN         — Content Delivery Network. 전 세계 캐시 서버.
SSL / TLS   — 암호화 인증서. 택배 상자에 자물쇠.

[인증·보안]
OAuth       — Open Authorization. "구글로 로그인" — 비번 안 받음.
API key     — 외부 서비스 인증 키. 고객 카드 번호.
.env        — Environment Variable 파일. 비밀 키 모아둔 잠긴 서랍.

[저장·상태]
Cache       — 캐시. 자주 쓰는 거 손에 두기.
Session     — 세션. "로그인 유지 중" 표시.
Cookie      — 쿠키. 브라우저가 가진 작은 메모지.
JWT         — JSON Web Token. 서버 도장 찍힌 입장권.

[고급]
CORS        — Cross-Origin Resource Sharing. "다른 도메인이 우리 서버 호출 OK?"
GraphQL     — REST의 다른 방식. 원하는 필드만 받기.
WebSocket   — 실시간 양방향 통신. 전화 통화.

[개발 도구]
IDE         — Integrated Development Environment. 코드 에디터 (VS Code/Cursor).
Git         — 버전 관리 도구. 코드 시간여행.
Repository  — 코드 저장소. 한 프로젝트 코드 + 역사 전체.
```

> "외울 필요 전혀 없어요. 본인이 Ch.2~Ch.5에서 진짜 만들 때 등장하면 다시 보면 돼요."
>
> "지금 본인에게 가장 막막해 보였던 약어 한두 개를 골라주세요. 그 한두 개를 더 자세히 풀어드릴게요."

학생이 약어를 고르면 한 개당 3-5문장으로 추가 설명. 본인 SaaS 맥락에 맞춰 예시 들기.

### Turn 7: 본인 SaaS 인프라 지도 + 파일 저장

> "마지막으로 본인 머릿속에 있는 SaaS 아이디어 — 아직 모호해도 좋아요 — 가지고 본인의 인프라 지도를 함께 그려볼게요."
>
> "다음 3가지를 한 줄씩 답해주세요:"
> 1. "본인 SaaS는 어떤 사람들이 쓸 거예요? (예: '20대 직장인', '대학생', '아직 모름')"
> 2. "기본 기능은 뭐예요? (예: '독서 노트 공유', '스터디 모임 매칭', '아직 모름')"
> 3. "외부 서비스는 뭐가 필요해 보여요? (예: '구글 로그인', '결제', '카톡 알림', '아직 모름')"

학생 답변을 받으면 본인 SaaS 맞춤 인프라 지도를 텍스트로 출력:

```
[본인 SaaS의 5-layer 인프라 지도 — 예시]

┌──────────────────────────────────────┐
│  ① [사용자 폰/PC 브라우저]            │
│     20대 직장인이 폰에서 접속          │
```
