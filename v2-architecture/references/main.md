### 레스토랑으로 이해하는 서비스 구조

여러분이 레스토랑을 운영한다고 상상해보세요. 아무 레스토랑이 아니라, 꽤 인기 있는 레스토랑이요.

```
레스토랑 = 웹 서비스

┌─────────────────────────────────────────────────┐
│  1층: 홀 (프론트엔드)                            │
│                                                  │
│  메뉴판     → 서비스 화면 (어떤 기능이 있는지)    │
│  인테리어   → 디자인 (색상, 글꼴, 레이아웃)       │
│  서빙       → 사용자 인터랙션 (클릭, 스크롤)      │
│  테이블 배치 → 화면 구성 (어디에 뭐가 있는지)     │
│  종업원     → 버튼, 입력창 (사용자를 안내)         │
│                                                  │

├─────────────────────────────────────────────────┤
│  2층: 주방 (백엔드)                              │
│                                                  │
│  요리사     → 서버 로직 (요청을 처리하는 두뇌)    │
│  레시피     → 비즈니스 규칙 (할인 계산, 권한 체크) │
│  조리 과정  → 데이터 처리 (정렬, 필터, 변환)      │
│  주문 전표  → API 요청 (프론트→백 통신)           │
│  위생 관리  → 보안 처리 (로그인, 인증)             │
│                                                  │
│  "손님은 못 들어가지만, 모든 일이 벌어지는 곳"    │
├─────────────────────────────────────────────────┤
│  지하: 창고 (데이터베이스)                        │
│                                                  │
│  재료       → 사용자 데이터 (이름, 이메일)         │
│  냉장고     → 데이터 저장소 (안전하게 보관)        │
│  재고 관리  → 데이터 관리 (추가, 수정, 삭제)       │
│  매출 장부  → 로그/기록 (누가 언제 뭘 했는지)     │
│  레시피북   → 설정 데이터 (카테고리, 옵션 값)      │
│                                                  │
│  "전원이 꺼져도 절대 사라지면 안 되는 것들"       │
└─────────────────────────────────────────────────┘
```

이 세 층이 모든 웹 서비스의 기본 구조입니다. 인스타그램도, 배달의민족도, 여러분이 만들 서비스도 다 이 구조입니다.

### 각 층을 더 자세히 볼까요?

**1층 - 프론트엔드 (홀)**

레스토랑에 들어서면 뭐가 보이나요? 메뉴판, 테이블, 조명, 음악. 이 모든 게 "손님 경험"을 만듭니다.

웹 서비스도 마찬가지예요. 사용자가 앱을 열면 보이는 모든 것 — 버튼, 글씨, 이미지, 색상 — 이게 프론트엔드입니다.

중요한 건, 홀이 예쁘다고 음식이 맛있는 건 아니라는 거예요. 하지만 홀이 지저분하면 손님이 나가버리죠. 프론트엔드도 마찬가지입니다. 기능이 아무리 좋아도, 화면이 불편하면 사용자는 떠납니다.

**2층 - 백엔드 (주방)**

손님이 "김치찌개 하나요!"라고 주문하면, 그 주문은 주방으로 갑니다.

주방에서 벌어지는 일:
- 재료가 있는지 확인 (데이터 조회)
- 레시피대로 조리 (비즈니스 로직 실행)
- 알레르기 체크 (권한/보안 확인)
- 완성된 요리를 홀로 보냄 (응답 전송)

손님은 주방에서 무슨 일이 벌어지는지 몰라요. 알 필요도 없고요. 그냥 맛있는 음식이 나오면 됩니다. 백엔드도 똑같습니다.

**지하 - 데이터베이스 (창고)**

냉장고에 뭐가 있는지, 오늘 뭐가 들어왔는지, 어제 뭐가 나갔는지. 이 기록이 없으면 레스토랑은 운영이 안 됩니다.

데이터베이스는 "기억"입니다:
- 이 사용자가 누구인지 기억
- 이 사용자가 뭘 했는지 기억
- 이 상품의 가격이 얼마인지 기억
- 어제 주문이 몇 건이었는지 기억

전원이 꺼져도, 서버가 재시작돼도, 데이터베이스의 내용은 사라지지 않습니다.

### 한국 서비스로 3층 이해하기

이론만으로는 와닿지 않죠? 매일 쓰는 서비스로 살펴봅시다.

**배달의민족**
```
┌─────────────────────────────────────────┐
│  프론트엔드 (홀)                         │
│  - 음식 카테고리 화면                    │
│  - 가게 목록 + 메뉴 사진                 │
│  - 장바구니, 결제 화면                   │
│  - 배달 추적 지도                        │
│  - 리뷰 쓰기 화면                        │
├─────────────────────────────────────────┤
│  백엔드 (주방)                           │
│  - 주문 접수 → 가게에 전달               │
│  - 결제 처리 (카드사 연동)               │
│  - 배달 기사 매칭 알고리즘               │
│  - 배달 예상 시간 계산                   │
│  - 쿠폰 적용 + 할인 계산                │
├─────────────────────────────────────────┤
│  데이터베이스 (창고)                     │
│  - 가게 정보 (이름, 위치, 영업시간)       │
│  - 메뉴 정보 (이름, 가격, 사진)          │
│  - 사용자 정보 (주소, 결제수단)           │
│  - 주문 내역 (언제, 뭐, 얼마)            │
│  - 리뷰 데이터 (별점, 내용, 사진)        │
└─────────────────────────────────────────┘
```

**당근마켓**
```
┌─────────────────────────────────────────┐
│  프론트엔드 (홀)                         │
│  - 동네 중고 물품 목록 화면               │
│  - 물품 상세 (사진, 가격, 설명)           │
│  - 1:1 채팅 화면                         │
│  - 내 동네 설정 화면                     │
│  - 프로필 + 매너 온도                    │
├─────────────────────────────────────────┤
│  백엔드 (주방)                           │
│  - 위치 기반 필터링 (6km 이내)           │
│  - 채팅 메시지 전송/수신 처리            │
│  - 거래 상태 관리 (판매중→예약→완료)      │
│  - 매너 온도 계산                        │
│  - 키워드 알림 매칭                      │
├─────────────────────────────────────────┤
│  데이터베이스 (창고)                     │
│  - 상품 정보 (제목, 가격, 카테고리, 사진) │
│  - 사용자 정보 (닉네임, 동네, 매너온도)   │
│  - 채팅 기록 (대화 내용, 시간)           │
│  - 거래 내역 (판매자, 구매자, 상태)       │
│  - 동네 인증 정보 (GPS, 인증 횟수)       │
└─────────────────────────────────────────┘
```

보이시나요? 서비스가 아무리 복잡해 보여도, 결국 이 3층입니다. 배달의민족은 3층이 크고 복잡하고, 작은 서비스는 3층이 작고 단순할 뿐입니다.

### 서비스 디렉터의 역할

여러분은 이 세 층을 직접 만들 필요가 없습니다. Claude가 만들어줍니다.

하지만 "내 서비스에서 각 층이 뭘 해야 하는지"는 여러분이 결정해야 합니다.

레스토랑 사장님을 생각해보세요:
- 사장님이 직접 요리하나요? 아니요, 요리사가 합니다.
- 사장님이 직접 서빙하나요? 아니요, 종업원이 합니다.
- 하지만 메뉴를 정하고, 인테리어를 결정하고, "어떤 식당을 만들 건지"는 사장님이 결정합니다.

그게 바로 **서비스 디렉터**의 역할입니다.

### Turn 1: 식당 이야기로 시작하기

학생에게 이렇게 말합니다:

"혹시 자주 가는 식당이 있으세요? 동네 분식집이든 카페든 프랜차이즈든 어떤 식당이든 괜찮아요. 하나 떠올려보세요!"

학생의 답변을 기다립니다.

**학생이 구체적인 식당을 말한 경우:**
"오, [식당 이름]! 거기에 들어가면 뭐가 보이세요? 메뉴판? 테이블? 인테리어? 그게 바로 **홀**이에요. 그런데 주방에서는 어떤 일이 벌어지고 있을까요? 재료는 어디에 보관할까요?"

**학생이 "잘 모르겠어요"라고 한 경우:**
"그러면 오늘 점심을 어디서 드셨어요? 또는 편의점이라도 괜찮아요! 편의점도 3층 구조예요 — 진열대(홀), 계산/재고관리(주방), 상품정보/매출기록(창고)."

**학생이 카페를 말한 경우:**
"카페 좋죠! 카페에 가면 메뉴판이 보이고(홀), 바리스타가 커피를 내리고(주방), 원두 재고와 매출이 기록되잖아요(창고). 사실 모든 가게는 이 3가지로 나뉩니다."

이어서:
"사실 모든 식당은 이 3가지로 나뉩니다: **홀, 주방, 창고**. 웹 서비스도 똑같아요."

### Turn 2: 레스토랑에서 서비스로 연결

위의 EXPLAIN 내용을 학생의 식당 이야기에 연결하며 설명합니다. 배달의민족이나 당근마켓 예시를 적절히 활용합니다.

핵심 메시지:
- "여러분이 매일 쓰는 앱도 다 이 구조예요."
- "복잡해 보이지만, 결국 3층일 뿐이에요."

그 다음 질문합니다:

"자, 이제 여러분의 **[서비스명]**을 이 3층으로 나눠볼까요? 너무 완벽하게 안 해도 돼요. 떠오르는 대로 말씀해주세요."

구체적인 질문 3개를 던집니다:
1. "사용자가 눈으로 보게 될 화면은 뭐가 있을까요? 어떤 버튼이 있고, 어떤 정보가 보일까요? (이게 **홀**이에요)"
2. "뒤에서 자동으로 처리돼야 하는 건 뭘까요? 사용자가 버튼을 누르면 뒤에서 뭐가 일어나야 하나요? (이게 **주방**이에요)"
3. "어떤 정보를 저장해야 할까요? 사용자가 다음에 와도 기억하고 있어야 하는 건 뭘까요? (이게 **창고**예요)"

학생의 답변을 기다립니다.

**학생이 잘 대답하면:**
"와, 벌써 감이 오시네요! [학생이 말한 것]이 딱 맞아요."

**학생이 3층을 혼동하면 (예: 화면에 "데이터 저장"을 넣음):**
"좋은 시도예요! '데이터 저장'은 사용자가 직접 보는 건 아니잖아요? 사용자는 '저장 버튼'을 보고(홀), 뒤에서 저장이 처리되고(주방), 실제 데이터가 보관되는 거예요(창고). 이렇게 하나의 행동이 3층을 관통합니다!"

"괜찮아요! 이렇게 말하면 돼요: '[서비스명]은 [홀 설명]을 보여주고, 뒤에서 [주방 설명]을 처리해서, [창고 설명]을 저장하는 서비스야.' 이 한 문장이면 충분합니다."

### 질문 1: 3층 구조 이해

"사용자가 버튼을 누르면 결과가 화면에 나타나는 과정을 순서대로 나열하면?"

- (A) 프론트엔드 → 백엔드 → 데이터베이스 → 백엔드 → 프론트엔드
- (B) 데이터베이스 → 백엔드 → 프론트엔드
- (C) 프론트엔드 → 데이터베이스 → 프론트엔드
- (D) 백엔드 → 프론트엔드 → 데이터베이스

정답: (A) — 홀에서 주문(프론트엔드) → 주방에서 접수(백엔드) → 창고에서 재료 꺼냄(데이터베이스) → 주방에서 요리 완성(백엔드) → 홀에 서빙(프론트엔드). 이 왕복 흐름이 모든 웹 서비스의 기본 동작입니다.

### 질문 2: 서비스 디렉터의 역할

"서비스 디렉터인 여러분이 결정해야 하는 것은?"

- (A) 각 층의 코드를 직접 작성하는 것
- (B) 각 층에서 무엇을 해야 하는지 정의하는 것
- (C) 서버를 직접 관리하는 것
- (D) 데이터베이스 쿼리를 작성하는 것

정답: (B) — 코드는 Claude가 작성합니다. 여러분은 "어떤 화면이 필요하고, 어떤 기능이 필요하고, 어떤 데이터를 저장해야 하는지"를 결정하는 사람입니다. 레스토랑 사장님이 요리를 직접 안 해도 메뉴를 결정하는 것과 같습니다.

### 질문 3: 실전 적용

"당근마켓에서 '매너 온도'가 올라가는 과정을 3층으로 나누면?"

- (A) 매너 온도 표시(프론트) → 온도 계산(백엔드) → 거래 기록 저장(DB)
- (B) 온도 계산(백엔드) → 매너 온도 표시(프론트) → 거래 기록 저장(DB)
- (C) 거래 기록 저장(DB) → 매너 온도 표시(프론트) → 온도 계산(백엔드)
- (D) 매너 온도 표시(프론트) → 거래 기록 저장(DB) → 온도 계산(백엔드)

정답: (A) — 사용자가 매너 평가 버튼을 누르면(프론트) → 서버에서 새 온도를 계산하고(백엔드) → 결과를 저장합니다(DB). 그리고 다른 사용자가 프로필을 볼 때, DB에서 온도를 가져와서(백엔드) 화면에 표시합니다(프론트).
