### 서비스 설계서란?

C2 과정에서 여러분은 6개의 레슨을 거치며 다양한 산출물을 만들었습니다.

이것들을 하나로 모으면 **서비스 설계서 v1.0**이 됩니다.

```
6개의 산출물이 설계서의 어느 섹션에 들어가나?

v2-architecture     ──→  [2. 서비스 구조]
  my-architecture.md       "3층 분해표 — 홀, 주방, 창고"

v2-service-type     ──→  [3. 서비스 유형]
  my-service-type.md       "웹앱/챗봇/자동화/랜딩페이지 결정"

v2-feature-spec     ──→  [4. 기능 명세]
  my-feature-spec.md       "SAFE 검토 완료된 기능 목록"

v2-priority         ──→  [5. MVP 우선순위]
  my-priority-list.md      "Must/Should/Could/Won't 분류"

v2-user-flow        ──→  [6. 사용자 흐름]
```

### 비유: 여행 계획서

여행을 떠나기 전에 계획서를 만드는 것과 같습니다:

| 여행 계획서 | 서비스 설계서 | 해당 파일 |
|:---:|:---:|:---:|
| 어디로 가는지 | 서비스 구조 | my-architecture.md |
| 어떤 교통수단인지 | 서비스 유형 | my-service-type.md |
| 뭘 할 건지 | 기능 명세 | my-feature-spec.md |
| 뭘 먼저 할 건지 | 우선순위 | my-priority-list.md |
| 동선이 어떻게 되는지 | 사용자 흐름 | my-user-flow.md |
| 숙소 예약은 했는지 | 블루프린트 | my-blueprint.md |

이 설계서가 있으면, C3(BUILD)에서 Claude에게 "이거 만들어줘"라고 할 때 **훨씬 정확하고 빠르게** 서비스를 만들 수 있습니다.

---

### 왜 설계서가 중요한가?

**설계서 없이 만들기 시작하면:**
```
"아, 이 기능도 넣어야 하는데..."          → 끝없는 추가
"잠깐, 이 화면에서 뭘 해야 하지?"        → 방향 상실
"이거 처음부터 다시 만들어야 하나?"        → 재작업
"뭘 먼저 해야 하지?"                      → 우선순위 혼란
"사용자가 여기서 뭘 하지?"                → 흐름 미설계
```

**설계서가 있으면:**
```
"이번 주에는 Must 기능 1번을 만들자"      → 명확한 목표
"흐름도 따라서 화면을 하나씩 만들자"      → 명확한 순서
"블루프린트대로 데이터를 설계하자"        → 일관된 구조
"SAFE 체크리스트로 빠진 거 확인하자"      → 품질 보증
"사용자가 3클릭 안에 핵심 기능 도달"      → 좋은 경험
```

설계서는 **지도**입니다. 지도 없이 여행할 수 있나요? 갈 수는 있지만, 돌아돌아 가게 됩니다.

### BUILD 준비도 점수

C3로 넘어가기 전에, 준비가 됐는지 점수로 확인합니다.

```
BUILD 준비도 점수 시스템

5개 항목 x 20점 = 100점 만점

1. 서비스 구조 이해 (20점)
   "내 서비스의 프론트엔드/백엔드/데이터베이스를
    친구에게 설명할 수 있나요?"
   → 20점: 자신있게 설명 가능
   → 10점: 대략은 알지만 헷갈리는 부분이 있음
   →  0점: 아직 잘 모르겠음

2. 기능 우선순위 결정 (20점)
   "Must 기능이 명확하고,
    전체의 30% 이하인가요?"
   → 20점: Must가 명확하고 30% 이하
   → 10점: Must가 있지만 좀 많음 (50%)
   →  0점: 우선순위가 정해지지 않음

3. 사용자 흐름 검증 (20점)
   "핵심 흐름에 막다른 길이 없고,
    3-클릭 룰을 만족하나요?"
   → 20점: 3가지 규칙 모두 통과
   → 10점: 일부 문제가 남아있음
   →  0점: 흐름을 그리지 않음

4. SAFE 검토 완료 (20점)
   "기능 명세서를 SAFE로 검토했나요?
    빠진 것 없이 보완했나요?"
   → 20점: 4가지 모두 검토 + 보완 완료
   → 10점: 일부 검토함
   →  0점: 검토하지 않음

5. 블루프린트 승인 (20점)
   "건물주 체크리스트 5가지를
    확인하고 승인했나요?"
   → 20점: 5가지 모두 확인 + 승인
   → 10점: 일부 확인함
   →  0점: 승인하지 않음

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
총점: ___/100점

80점 이상 → BUILD 준비 완료!
60~70점  → 거의 다 됐어요! 부족한 부분만 보완하면 OK
50점 이하 → 해당 레슨을 다시 방문해봅시다
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### C3 미리보기 — 이제 진짜 만들러 갑니다

C3에서는 여러분의 설계서가 **실제 서비스**로 바뀝니다.

```
C3 BUILD에서 벌어질 일들:

Block 0: Claude Code 설치
  → 개발 도구를 설치합니다

Block 1: 첫 체험
  → "rails new"로 프로젝트를 생성하고,
     브라우저에서 내 서비스를 처음 볼 수 있어요!

Block 6: Rails 프로젝트 시작
  → 설계서의 구조가 실제 코드로 변환됩니다

Block 7: 홈페이지 + 로그인
  → 사용자가 처음 보는 화면이 만들어집니다
     my-user-flow.md의 첫 화면이 실제로!

Block 8: 핵심 기능 구현
  → Must 기능이 하나씩 동작하기 시작합니다
     my-priority-list.md의 Must 1번부터!

Block 9: 데이터베이스
  → my-architecture.md의 "창고"가 실제로 만들어집니다
```

설계서가 있기 때문에, Claude에게 이렇게 말할 수 있습니다:

```
여러분: "이 설계서대로 만들어줘"
Claude: (설계서를 읽고) "네, 이해했습니다.
        Must 기능 1번부터 시작할게요.
        사용자 흐름은 이렇게 되는 거죠?"
```

설계서가 없었다면?

```
여러분: "뭔가... 만들어줘"
Claude: "뭘 만들까요? 어떤 기능이요?
        사용자는 누구예요? 데이터는?"
여러분: "음... 그건..."
```

**차이가 느껴지시나요?**

---

> **꿀팁: 설계서는 "완벽"할 필요 없습니다**
>
> 80% 맞는 설계서면 충분합니다. 나머지 20%는 만들면서 발견하고 수정합니다.
> "완벽한 설계서를 만들겠다"고 계속 수정하면, 영원히 만들기 시작하지 못합니다.
>
> 지금 가진 설계서가 "대략 맞다"면, 출발할 때입니다.

### 질문 1: 3층 구조 복습

"서비스의 3층 구조에서 '사용자가 보는 화면'은 어디인가요?"

- (A) 데이터베이스
- (B) 백엔드
- (C) 프론트엔드
- (D) 서버

정답: (C) — 레스토랑의 홀처럼, 사용자가 직접 보고 만지는 공간입니다.

### 질문 2: SAFE 프레임워크 복습

"AI가 만든 기능 명세서를 검토할 때 사용하는 프레임워크는?"

- (A) MoSCoW
- (B) SAFE (Steps, Actors, Failure, Edge cases)
- (C) 3-클릭 룰
- (D) MVP

정답: (B) — AI 산출물을 비판적으로 검토하는 도구입니다. Steps(단계), Actors(사용자), Failure(실패), Edge cases(특이 상황) 4가지 관점으로 검토합니다.

### 질문 3: MoSCoW 복습

"MoSCoW에서 'Must'의 기준은?"

- (A) 만들기 쉬운 기능
- (B) 경쟁 서비스에 있는 기능
- (C) 이 기능 없이는 서비스가 작동하지 않는 기능
- (D) 사용자가 가장 좋아할 기능

정답: (C) — "이게 없으면 서비스가 아니다"가 Must의 기준입니다. 가장 엄격한 기준으로 판단해야 합니다.

### 질문 4: 설계서의 가치

"설계서가 있으면 C3(BUILD)에서 가장 큰 차이는?"

- (A) 코드를 직접 쓸 수 있다
- (B) Claude에게 "이것대로 만들어줘"라고 명확하게 지시할 수 있다
- (C) 만드는 시간이 10배 빨라진다
- (D) 버그가 하나도 없는 서비스가 나온다

정답: (B) — 설계서는 Claude와의 소통 도구입니다. "뭔가 만들어줘" vs "이 설계서대로 만들어줘"의 차이는 엄청납니다. 명확한 지시 = 정확한 결과.
