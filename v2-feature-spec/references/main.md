## EXPLAIN

### AI의 결과물을 그대로 쓰면 안 되는 이유

AI는 똑똑하지만, 여러분의 서비스를 여러분만큼 잘 모릅니다.

비유: AI가 만든 명세서는 **"인턴이 작성한 보고서 초안"**입니다. 내용의 80%는 괜찮지만, 나머지 20%는 여러분의 맥락을 모르기 때문에 빠져있거나 틀렸습니다. 인턴의 보고서를 그대로 사장님께 올리진 않잖아요?

---

### "의도적 80% 명세서" — 게임의 시작

이 레슨에서는 특별한 일이 벌어집니다.

AI가 여러분 서비스의 기능 명세서를 만들어줄 건데요, **일부러 빈틈을 남겨둡니다.**

왜냐고요? 이게 바로 여러분이 "서비스 디렉터"로서 훈련하는 방법이거든요.

```
AI의 80% 명세서

┌─────────────────────────────────────┐
│                                      │
│    80% 완성  ████████████████░░░░    │
│                                      │
│    AI가 잘 하는 것:                  │
│    ✓ 일반적인 기능 나열              │
│    ✓ 기본적인 화면 구성              │
│    ✓ 표준적인 기술 구조              │
│                                      │
│    AI가 빠뜨리는 것:                 │
│    ✗ 여러분만 아는 맥락              │
│    ✗ 실패했을 때의 상황              │
│    ✗ 특수한 사용자 유형              │
│    ✗ 엣지 케이스 (특이 상황)         │
│                                      │
└─────────────────────────────────────┘
```

여러분의 역할은 이 80%를 100%로 만드는 것입니다:
- "이건 맞아" (승인)
- "이건 빠졌어" (보완)
- "이건 지나쳐" (축소)
- "이건 틀렸어" (수정)

마치 **보물찾기**처럼, AI가 숨겨놓은 빈틈을 찾아보세요. 찾을수록 여러분의 서비스가 더 탄탄해집니다.

---

### SAFE 프레임워크 — 빈틈을 찾는 4가지 렌즈

AI의 기능 명세서를 검토할 때, 4가지만 확인하면 됩니다.

```
SAFE 검토 프레임워크

S — Steps (단계)
  "사용자가 이 기능을 쓰려면 몇 단계를 거치지?"
  → 빠진 단계가 없는지 확인
  → 예: 로그인 → 글쓰기 → 저장...
    근데 "저장 후 확인 화면"이 빠져있네?

A — Actors (사용자 유형)
  "이 기능을 쓰는 사람이 한 종류만 있나?"
  → 관리자, 일반 사용자, 비회원 등
  → 예: 선생님과 학생이 쓰는 서비스인데 선생님 화면만 있네?

F — Failure (실패 상황)
  "만약 뭔가 잘못되면 어떻게 되지?"
  → 인터넷 끊김, 잘못된 입력, 서버 에러 등
  → 예: 결제 중에 인터넷이 끊기면?

E — Edge cases (특이 상황)
  "보통이 아닌 상황에서도 작동하나?"
  → 데이터가 0개일 때, 1000개일 때
  → 예: 첫 사용자라 글이 하나도 없을 때 화면은?
```

### 배달앱으로 SAFE 완전 이해하기

SAFE가 아직 추상적이라면, 배달의민족으로 하나씩 뜯어봅시다.

**S — Steps (단계): 주문 과정에 빠진 단계가 없나?**

```
AI가 만든 명세서:
  1. 음식 선택
  2. 장바구니 담기
  3. 결제
  4. 배달 완료

여러분이 찾아야 할 것:
  "잠깐, 2번과 3번 사이에 '배달 주소 확인'이 빠졌잖아?"
  "3번과 4번 사이에 '배달 추적'은요?"
  "4번 다음에 '리뷰 쓰기'는?"

수정된 흐름:
  1. 음식 선택
  2. 장바구니 담기
  2.5 배달 주소 확인/변경    ← 빠졌던 단계
  3. 결제
  3.5 배달 추적 화면          ← 빠졌던 단계
  4. 배달 완료
  4.5 리뷰 작성               ← 빠졌던 단계
```

**A — Actors (사용자 유형): 누가 이 서비스를 쓰나?**

```
AI가 만든 명세서:
  사용자: 주문하는 고객

여러분이 찾아야 할 것:
  "잠깐, 배달 기사는? 기사도 앱을 쓰잖아!"
  "가게 사장님은? 주문을 받고 조리해야 하잖아!"
  "비회원도 주문할 수 있어야 하나?"

빠진 사용자 유형:
  - 배달 기사 → 배달 수락, 경로 확인, 배달 완료 처리
  - 가게 사장님 → 주문 접수, 조리 시작/완료, 영업시간 설정
  - 비회원 → 메뉴는 볼 수 있지만 주문은 못 하게?
```

**F — Failure (실패 상황): 뭐가 잘못될 수 있나?**

```
AI가 만든 명세서:
  결제: 카드로 결제한다

여러분이 찾아야 할 것:
  "결제가 실패하면? 카드 한도 초과면?"
  "결제 중에 인터넷이 끊기면? 돈은 빠졌는데 주문은 안 된 거야?"
  "음식 준비 중에 가게가 갑자기 문을 닫으면?"

추가해야 할 실패 시나리오:
  - 결제 실패 → "결제에 실패했습니다. 다른 결제 수단을 선택해주세요."
  - 네트워크 오류 → "주문이 접수되었는지 확인 중입니다..."
  - 가게 영업 종료 → "죄송합니다. 가게 사정으로 주문이 취소되었습니다. 환불 처리됩니다."
```

**E — Edge cases (특이 상황): "보통"이 아닐 때도 작동하나?**

```
AI가 만든 명세서:
  배달 시간: 30~40분

여러분이 찾아야 할 것:
  "비 오는 날이면? 평소보다 배달이 2배 걸릴 텐데"
  "새벽 2시에 주문하면? 열려 있는 가게가 없으면?"
  "같은 가게에 주문이 100건 몰리면?"
  "배달 주소가 섬이면? 산꼭대기면?"

추가해야 할 특이 상황:
  - 악천후 시 → 배달 지연 안내 + 예상 시간 업데이트
  - 심야 시간 → 24시간 영업 가게만 표시
  - 주문 폭주 → "현재 주문이 많아 배달이 지연될 수 있습니다"
  - 배달 불가 지역 → "죄송합니다. 해당 지역은 배달이 어렵습니다"
```

### "명세서 검토관" 역할극

이 레슨의 하이라이트는 **역할극**입니다.

여러분은 **까다로운 PM(프로덕트 매니저)**이 됩니다. AI가 만든 명세서를 꼼꼼하게 검토하는 역할이에요.

```
역할 설정:

여러분: 까다로운 PM
  "이거 빠져있잖아요. 다시 해오세요."
  "이 경우에는 어떻게 되는데요?"
  "사용자가 이렇게 하면 어쩌죠?"

AI: 열심히 하는 인턴
  "아, 맞다! 그 부분을 놓쳤네요. 추가하겠습니다."
  "좋은 지적이에요. 이렇게 수정하면 될까요?"
```

좋은 PM은 "왜?"를 계속 물어봅니다:
- "왜 이 단계가 여기에 있지?"
- "왜 이 사용자 유형은 안 넣었지?"
- "왜 실패 상황을 고려 안 했지?"

---

> **꿀팁: SAFE 기억법**
>
> **S**teps가 다 있나? (단계)
> **A**ctors가 다 있나? (사용자)
> **F**ailure를 생각했나? (실패)
> **E**dge cases를 고려했나? (특이 상황)
>
> 외우기 쉽게: "**안전(SAFE)**하게 검토하자!"

## EXECUTE

### Turn 1: "AI가 완벽한 명세서를 만들어줄까요?"

학생에게 말합니다:

"AI가 완벽한 명세서를 만들어줄까요? 아닙니다! AI는 80%만 맞추고 20%는 빠뜨립니다. 그래서 '검토'가 중요합니다."

"지금부터 제가 [서비스명]의 기능 명세서를 만들어볼게요. 그런데 솔직히 말씀드리면, **제가 일부러 빈틈을 남겨둘 거예요.** 보물찾기라고 생각하세요. 뭔가 이상한 게 없는지 찾아보시는 거예요!"

AI가 학생의 서비스에 맞는 기능 명세서 초안을 생성합니다.

**중요: 의도적으로 아래 항목 중 2-3개를 빠뜨립니다:**
- 에러 처리 시나리오 생략 (F 빈틈)
- 관리자(또는 다른 사용자 유형) 시나리오 생략 (A 빈틈)
- 첫 사용(데이터 없음) 시나리오 생략 (E 빈틈)
- 중간 단계 1-2개 생략 (S 빈틈)

명세서를 보여준 후 질문합니다:

"자, 명세서가 완성됐어요. 쭉 읽어보세요. 느낌이 어떤가요? 뭔가 빠진 것 같은 게 있나요?"

학생의 답변을 기다립니다.

**학생이 빈틈을 찾은 경우:**
"오! 대단하세요! [찾은 것]을 발견하셨네요. 맞아요, 제가 일부러 빠뜨린 거예요. 혹시 더 찾을 수 있으세요?"

**학생이 "다 괜찮아 보여요"라고 한 경우:**
"정말요? 다시 한번 자세히 보세요. 힌트를 드릴게요 — 사용자가 [특정 행동]을 하면 다음에 뭐가 벌어지나요?"

**학생이 잘 모르겠다고 한 경우:**
"괜찮아요! 지금부터 빈틈을 체계적으로 찾는 방법을 알려드릴게요. SAFE라는 4글자 체크리스트입니다."

### Turn 2: SAFE 프레임워크로 검토하기

EXPLAIN의 SAFE 프레임워크와 배달앱 예시를 설명합니다.

그 다음 함께 적용합니다:

"자, [서비스명]의 명세서를 SAFE로 하나씩 봅시다. 제가 질문할 테니, 여러분이 대답해보세요."

**S — Steps 질문:**
"[핵심 기능]에서 사용자가 거치는 단계를 순서대로 말해보세요. 1번 → 2번 → 3번... 혹시 빠진 단계가 있나요?"

**A — Actors 질문:**
"이 서비스를 쓰는 사람이 [사용자A]만 있나요? 혹시 다른 유형의 사용자는 없나요? 관리자는요?"

**F — Failure 질문:**
"만약 [특정 상황]이 실패하면 어떻게 되나요? 사용자는 뭘 보게 되나요?"

**E — Edge cases 질문:**
"처음 가입한 사용자라 데이터가 하나도 없으면, 이 화면에 뭐가 보이나요?"

학생이 찾지 못한 문제가 있으면 힌트를 줍니다:
"좋은 검토입니다! 하나 더 생각해볼게요. 배달의민족에서도 비슷한 문제가 있었어요..."

### Turn 3: 역할극 — 까다로운 PM이 되어보기

"이제 역할극을 해봅시다! 여러분은 까다로운 PM이에요. 제가 인턴으로서 명세서를 가져왔으니, 마음껏 태클 걸어주세요."

학생에게 3가지 관점으로 질문하도록 유도합니다:
1. "이 부분은 왜 이렇게 했나요?" (의도 확인)
2. "이 경우에는 어떻게 되나요?" (빈틈 탐색)
3. "이건 꼭 필요한가요?" (불필요한 것 제거)

AI가 인턴처럼 대답하고, 학생이 지적한 부분을 반영합니다.

### Turn 4: AI가 의도적으로 숨긴 것 공개

"사실 제가 일부러 빠뜨린 것들이 있었어요. 공개합니다:"

의도적으로 빠뜨린 항목을 하나씩 공개하고, 학생이 찾은 것/못 찾은 것을 정리합니다.

```
빈틈 찾기 결과

여러분이 찾은 것:
  ✓ [찾은 항목 1] — SAFE의 [해당 항목]
  ✓ [찾은 항목 2] — SAFE의 [해당 항목]

숨겨져 있었던 것:
  ✗ [못 찾은 항목] — SAFE의 [해당 항목]
  → [왜 이게 중요한지 설명]

검토력 점수: [찾은 수]/[전체 빈틈 수]
```

**모두 찾은 경우:**
"와, 전부 찾으셨어요! PM으로서 자질이 훌륭합니다. 이 검토력이 있으면, AI와 협업할 때 절대 빈틈 있는 결과물을 그냥 넘기지 않을 거예요."

**일부만 찾은 경우:**
"[N]개 중 [M]개를 찾으셨어요! 처음 치곤 대단합니다. 못 찾은 부분은 [SAFE 항목]이었는데, 다음에는 이 부분을 특히 신경 써보세요."

### Turn 5: 수정된 명세서 완성

학생의 피드백 + SAFE 검토 결과를 반영하여 수정된 명세서를 생성합니다.

"처음 80%였던 명세서가 여러분의 검토로 거의 100%가 됐어요. 이게 바로 서비스 디렉터의 일입니다."

"두 명세서를 비교해보세요:"

```
처음 명세서:  ████████████████░░░░  80%
지금 명세서:  ████████████████████  98%

여러분이 추가한 것:
  + [SAFE 검토로 추가된 항목들]
  + [역할극에서 발견된 항목들]
```

### Turn 6: 파일 저장

완성된 명세서를 `my-feature-spec.md`로 저장합니다.

저장 내용에 포함:
- 기능 명세서 전문
- SAFE 검토 결과 요약
- 학생이 직접 추가/수정한 항목 표시

"다음 레슨: `/v2-priority` — 이 기능들 중에 뭘 먼저 만들지 우선순위를 정합니다!"

---

> **막혔을 때**
>
> "SAFE가 어려워요" → 하나씩만 하세요. 일단 S(단계)만 체크해보세요. "1번 다음에 2번, 2번 다음에 3번... 빠진 거 없지?" 이것만 해도 절반은 찾을 수 있어요.
>
> "빠진 걸 못 찾겠어요" → 이렇게 물어보세요: "만약 내가 이 서비스를 처음 쓴다면, 여기서 뭘 해야 할지 알 수 있을까?" 이 질문이 빈틈을 찾는 가장 좋은 방법입니다.
>
> "너무 많이 추가하고 싶어요" → 지금은 명세서 "검토"가 목적이에요. 기능을 추가하는 건 다음 레슨에서! 지금은 "빠진 것 찾기"에 집중하세요.

## QUIZ

### 질문 1: SAFE 프레임워크 적용

"AI가 만든 기능 명세서에서 '결제 기능'이 있는데, '결제 실패 시 처리'가 없습니다. SAFE 중 어떤 항목이 빠진 건가요?"

- (A) S (Steps) — 단계가 빠졌다
- (B) A (Actors) — 사용자 유형이 빠졌다
- (C) F (Failure) — 실패 상황이 빠졌다
- (D) E (Edge cases) — 특이 상황이 빠졌다

정답: (C) — 결제 실패는 전형적인 Failure 시나리오입니다. 인터넷 끊김, 카드 한도 초과, 서버 에러 등 "잘못될 수 있는 상황"은 F에 해당합니다.

### 질문 2: AI 산출물 활용법

"AI의 결과물을 다루는 올바른 태도는?"

- (A) AI가 전문가니까 결과물을 그대로 사용한다
- (B) AI의 결과물은 믿을 수 없으니 처음부터 직접 쓴다
- (C) AI의 초안을 기반으로, 내 맥락에 맞게 검토하고 수정한다
- (D) 여러 AI에게 같은 요청을 해서 다수결로 정한다

정답: (C) — AI는 80% 수준의 좋은 초안을 빠르게 만들어줍니다. 하지만 여러분의 서비스, 여러분의 사용자, 여러분의 맥락은 여러분이 가장 잘 압니다. AI의 초안 + 여러분의 판단 = 좋은 명세서입니다.

### 질문 3: 빈틈 찾기 실전

"온라인 독서 모임 서비스에서 '모임 참가' 기능의 명세서에 '모임 정원 초과 시 처리'가 빠져있습니다. SAFE 중 어떤 항목에 해당하나요?"

- (A) S (Steps) — 참가 단계가 빠졌다
- (B) A (Actors) — 사용자 유형이 빠졌다
- (C) F (Failure) — 실패 상황이 빠졌다
- (D) E (Edge cases) — 특이 상황이 빠졌다

정답: (D) — 정원 초과는 "보통은 아닌데, 일어날 수 있는 상황"이므로 Edge case입니다. "10명 정원에 11번째 사람이 참가하려고 하면?"이라는 특이 상황이죠. 대기 명단으로 안내하거나 "정원이 마감되었습니다" 메시지를 보여줘야 합니다.
