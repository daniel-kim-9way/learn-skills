## EXPLAIN

### 이사짐 게임

여러분이 원룸에서 새 집으로 이사한다고 합시다.

그런데 조건이 있어요:
- 트럭은 **1대**뿐입니다
- 상자는 **5개**만 실을 수 있습니다
- 짐은 **15개** 이상입니다
- 나머지는 다음 주에 가지러 와야 합니다

```
트럭 = 첫 버전 (MVP)

이사짐 목록:
  🛏 침대          → 이거 없으면 어디서 자요?
  🧊 냉장고       → 음식 보관을 못 하잖아요
  👕 옷            → 내일 출근해야 하는데...
  📚 책 300권      → 음... 급하진 않은데
  📺 TV            → 없어도 살 수 있긴 한데
  🛋 소파          → 불편하지만 바닥에 앉아도...
  🖼 액자 10개     → 이건 나중에
  🧹 청소도구      → 첫날부터 필요하겠지?
  ☕ 커피머신      → 편의점 커피 사 마시면 되잖아

  자, 5개만 고르세요. 어떤 걸 실으시겠어요?
```

자연스럽게 이렇게 되죠:
- 침대? → **당연히 실어야죠.** (Must)
- 냉장고? → **이것도 필수.** (Must)
- 옷? → **내일 출근해야 하니까 당연히.** (Must)
- 청소도구? → **첫날 청소는 해야 하니까.** (Must)
- 소파? → **없으면 불편하지만... 바닥에 앉아도 되긴 해.** (Should)
- TV? → **있으면 좋지만 없어도 살 수 있어.** (Could)
- 커피머신? → **편의점 가면 되니까.** (Could)
- 책 300권? → **다음 주에 가져오면 되지.** (Won't)
- 액자 10개? → **이건 정리 다 하고 나중에.** (Won't)

서비스 기능도 **똑같습니다.**

트럭 = 첫 버전(MVP), 짐 = 기능들. 전부 다 싣고 싶지만, 트럭은 하나뿐이에요. **뭘 먼저 싣느냐**가 서비스의 성패를 가릅니다.

### MoSCoW 방법론

이사짐 게임의 분류법을 정식으로 만든 것이 MoSCoW입니다.

```
┌──────────────────────────────────────────────────────┐
│  Must Have (없으면 서비스가 안 됨)                    │
│                                                       │
│  한국어: "이게 없으면 서비스라고 부를 수 없다"         │
│                                                       │
│  판단 기준:                                           │
│  "이 기능을 빼고 서비스를 열었을 때,                  │
│   사용자가 핵심 목적을 달성할 수 있나?"                │
│    → NO → Must                                       │
│                                                       │
│  예시:                                                │
│  - 배달의민족에서 "음식 주문" → Must (이거 없으면      │
│    배달앱이 아님)                                      │
│  - 당근마켓에서 "물건 올리기" → Must (거래 자체가      │
│    불가능)                                             │
│  - 카카오톡에서 "메시지 보내기" → Must (채팅앱의 존재   │
│    이유)                                               │
├──────────────────────────────────────────────────────┤
│  Should Have (없으면 불편하지만 돌아는 감)             │
│                                                       │
│  한국어: "있어야 사용자가 편하다"                      │
│                                                       │
│  판단 기준:                                           │
│  "첫 사용자가 이걸 기대하나? 없으면 실망할까?"         │
│    → 아마도 YES → Should                              │
│                                                       │
│  예시:                                                │
│  - 배달의민족에서 "주문 내역" → Should (없으면         │
│    불편하지만 주문은 가능)                              │
│  - 당근마켓에서 "찜하기" → Should (없어도 거래는 됨)   │
│  - 카카오톡에서 "읽음 표시" → Should (없어도 대화는    │
│    가능)                                               │
├──────────────────────────────────────────────────────┤
│  Could Have (있으면 좋지만, 없어도 핵심 경험은 OK)     │
│                                                       │
│  한국어: "있으면 멋지다, 하지만 지금은 아니어도"       │
│                                                       │
│  예시:                                                │
│  - 배달의민족에서 "다크 모드" → Could                  │
│  - 당근마켓에서 "AI 가격 추천" → Could                │
│  - 카카오톡에서 "프로필 음악" → Could                  │
├──────────────────────────────────────────────────────┤
│  Won't Have (좋은 아이디어지만 이번에는 안 함)         │
│                                                       │
│  한국어: "다음 버전에서 하자"                          │
│                                                       │
│  예시:                                                │
│  - 배달의민족에서 "요리 레시피 제공" → Won't           │
│  - 당근마켓에서 "해외 직구" → Won't                   │
│  - 카카오톡에서 "메타버스 채팅방" → Won't              │
└──────────────────────────────────────────────────────┘
```

### 판단 기준: 세 가지 핵심 질문

기능 하나하나를 분류할 때, 이 3가지 질문을 순서대로 던지세요:

**질문 1: "이 기능 없이 서비스를 써볼 수 있나?"**
- 못 쓴다 → **Must**
- 쓸 수는 있다 → 다음 질문으로

**질문 2: "첫 사용자가 이 기능을 바로 필요로 할까?"**
- 당장 필요하다 → **Should**
- 나중에 필요할 수 있다 → 다음 질문으로

**질문 3: "이 기능을 만드는 데 얼마나 걸릴까?"**
- 간단하다 → **Could** (여유 있으면 넣자)
- 복잡하다 → **Won't** (다음 버전으로)

### "MVP 함정" 경고

우선순위를 정할 때 빠지기 쉬운 함정이 있습니다.

**함정 1: "이것도 필요하고 저것도 필요한데..."**
```
학생: "로그인도 필요하고, 검색도 필요하고, 알림도 필요하고,
       공유도 필요하고, 통계도 필요하고, 다크 모드도..."

현실: 전부 Must로 넣으면
     → 개발 기간 3배
     → 중간에 방향 수정 불가
     → 지쳐서 포기

해결: "이 기능 없으면 서비스가 안 돌아가?"
     이 질문에 NO면, Must가 아닙니다.
```

**함정 2: "경쟁 서비스에 있으니까 우리도 있어야지"**
```
학생: "배달의민족에 리뷰 기능이 있으니까
       우리 서비스에도 리뷰 넣어야 해요"

현실: 배달의민족은 10년 된 서비스.
     지금의 배달의민족은 Version 100+.
     여러분은 Version 1을 만드는 중.

해결: "Version 1에서 이 기능이 있어야
     사용자가 핵심 가치를 느낄 수 있나?"
```

**함정 3: "다 쉬워 보이는데요?"**
```
학생: "이 기능 간단해 보여요.
       그냥 버튼 하나면 되잖아요?"

현실: 사용자에게 보이는 "버튼 하나" 뒤에는
     → 데이터 저장
     → 에러 처리
     → 보안 체크
     → 다양한 기기 대응
     간단해 보이는 기능도 만들면 복잡합니다.

해결: 간단해 보여도 일단 분류하고,
     진짜 간단한지는 Claude에게 물어보세요.
```

---

> **꿀팁: Must의 마법 숫자**
>
> Must 기능은 전체 기능의 **30% 이하**가 좋습니다.
> 기능이 10개라면 Must는 3개 이하.
> 기능이 15개라면 Must는 4-5개 이하.
>
> Must가 50%를 넘으면, 기준이 너무 느슨한 거예요.
> 다시 "이거 없으면 진짜 서비스가 안 되나?" 물어보세요.

## EXECUTE

### Turn 1: 이사짐 게임으로 시작

학생에게 말합니다:

"이사를 간다고 해보세요. 원룸에서 새 집으로! 그런데 트럭이 작아서 상자 5개만 실을 수 있어요."

이사짐 목록을 보여주며:
"침대, 냉장고, 소파, TV, 옷, 책 300권, 커피머신, 청소도구, 액자 10개... 뭘 먼저 싣겠어요? 5개만 골라보세요!"

학생의 답변을 듣고 연결합니다:

**학생이 잘 고른 경우 (필수품 위주):**
"역시! 침대, 냉장고, 옷 같은 건 당연히 먼저 싣죠. 소파나 TV는 나중에 가져와도 되고요. 서비스 기능도 마찬가지예요. 지금부터 [서비스명]의 기능들로 같은 게임을 해봅시다!"

**학생이 TV나 커피머신을 골랐다면:**
"오, 커피머신을 고르셨네요! 커피 없이는 못 사시는 거죠? 하지만 편의점에서 사 마실 수도 있잖아요? 이사 첫날 침대 없이 자는 건 진짜 힘들지만, 커피는 대안이 있어요. 서비스에서도 '대안이 있는 기능'은 나중으로 미룰 수 있습니다."

그 다음 기능 명세서(`my-feature-spec.md`)에서 전체 기능 목록을 꺼냅니다:

"[서비스명]의 기능 명세서에서 기능들을 쭉 뽑아볼게요."

전체 기능 목록을 나열하고, 학생에게 직감적으로 **"가장 중요한 기능 3개"**를 골라보라고 합니다.

"직감으로 골라보세요. 아직 정확하지 않아도 돼요. 느낌대로!"

### Turn 2: MoSCoW 분류

EXPLAIN의 MoSCoW 설명을 한국 서비스 예시와 함께 설명합니다.

학생이 고른 기능부터 시작합니다:

"[기능명] — 이거 없으면 서비스가 안 돌아가나요?"

**학생이 "네" → "좋아요, Must입니다. 트럭에 먼저 실어요!"**

**학생이 "글쎄요..." →**
"다르게 물어볼게요. [서비스명]을 이 기능 빼고 1주일 운영한다고 해보세요. 사용자가 '이게 뭐야, 안 쓸래' 하고 떠날까요?"
- "떠날 것 같아요" → "그럼 Must예요."
- "떠나진 않을 것 같아요" → "그럼 Should예요. 불편하지만 돌아는 가니까."

**학생이 "잘 모르겠어요" →**
"이렇게 생각해보세요. 배달의민족에서 '음식 주문' 기능을 빼면 어떨까요? 그건 배달앱이 아니죠? 그 정도로 핵심인가요?"

모든 기능을 하나씩 분류합니다.

### Turn 3: "전부 다 Must"에 대한 챌린지

학생이 모든 것을 Must로 넣으려 하면:

"잠깐! Must에 [N]개가 있네요. 트럭에 상자를 15개 실으려는 거예요!"

"**'전부 중요해요'는 '아무것도 중요하지 않다'와 같아요.**"

"가장 엄격한 기준으로 다시 볼게요. 각 Must에 대해 하나씩 물어볼게요:"

각 Must 기능에 대해:
1. "이거 없으면 서비스가 **정말** 안 돌아가나요?"
   - "사실... 없어도 돌아가긴 해요" → **Should로 이동**
2. "Version 1에서 꼭 있어야 하나요, 아니면 Version 2에서도 괜찮나요?"
   - "V2에서도..." → **Should 또는 Could로 이동**

최종적으로 Must가 전체의 30% 이하가 되도록 합니다.

### Turn 4: MVP 함정 체크

분류가 끝나면, MVP 함정을 하나씩 체크합니다:

"우선순위를 정했으니, 함정에 빠지지 않았는지 확인해봅시다."

**함정 1 체크:** "Must 기능이 전체의 30% 이하인가요?"
- 넘으면: "기준을 더 엄격하게 적용해봅시다."

**함정 2 체크:** "혹시 '다른 서비스에 있으니까' 넣은 기능은 없나요?"
- 있으면: "그 서비스는 Version 100이에요. 우리는 Version 1이에요."

**함정 3 체크:** "'쉬워 보여서' Must에 넣은 기능은 없나요?"
- 있으면: "쉬워 보여도, 진짜 필요한지가 기준이에요."

### Turn 5: 최종 MVP 기능 목록

```
[서비스명] MVP 기능 우선순위

Must Have (첫 버전에 반드시) — 이것 없이 서비스가 안 됨
  1. [기능] — 이유: [학생이 말한 근거]
  2. [기능] — 이유: [학생이 말한 근거]
  3. [기능] — 이유: [학생이 말한 근거]

Should Have (첫 버전에 가능하면) — 없으면 불편하지만 가능
  4. [기능] — 이유: [학생이 말한 근거]
  5. [기능] — 이유: [학생이 말한 근거]

Could Have (여유 있으면) — 있으면 좋지만
  6. [기능]
  7. [기능]

Won't Have (다음 버전으로) — 나중에 하자
  8. [기능]
  9. [기능]

━━━━━━━━━━━━━━━━━━━━━━━━━━
MVP 범위: Must [N]개 + Should [N]개 = 총 [N]개 기능
트럭에 실은 짐: [N]개 ✓
다음에 가져올 짐: [N]개
━━━━━━━━━━━━━━━━━━━━━━━━━━
```

학생에게 최종 확인:
"이 목록을 보세요. Must만으로 서비스가 돌아갈 것 같나요? Should까지 포함하면 사용자가 만족할 것 같나요?"

### Turn 6: 파일 저장

확정된 목록을 `my-priority-list.md`로 저장합니다.

"다음 레슨: `/v2-user-flow` — Must 기능을 사용자가 어떻게 쓸지 흐름을 그려봅니다!"

---

> **막혔을 때**
>
> "전부 다 중요해요!" → 이렇게 해보세요: 기능을 하나씩 빼면서 "이거 빼면 사용자가 떠날까?"를 물어보세요. 떠나지 않으면 Must가 아닙니다.
>
> "Must와 Should 구분이 안 돼요" → "없으면 서비스가 **작동 불가**" = Must. "없으면 **불편**" = Should. 작동 불가와 불편은 다릅니다.
>
> "Won't에 넣기 아까워요" → Won't는 "안 한다"가 아니라 "**이번에는** 안 한다"예요. 다음 버전에서 할 거예요. 버리는 게 아니라 미루는 겁니다.

## QUIZ

### 질문 1: MoSCoW 판단

"음식 배달 앱을 만든다고 합시다. '다크 모드'는 MoSCoW 중 어디에 해당할까요?"

- (A) Must — 필수 기능이다
- (B) Should — 있어야 좋은 기능이다
- (C) Could — 있으면 좋은 기능이다
- (D) Won't — 이번에는 안 한다

정답: (C) — 다크 모드는 사용자 경험을 개선하지만, 음식 배달의 핵심(메뉴 보기 → 주문 → 결제 → 배달 추적)과는 관련 없습니다. 있으면 멋지지만, 없어도 핵심 경험에는 문제없습니다.

### 질문 2: 우선순위 판단

"모든 기능을 Must로 넣고 싶을 때, 어떻게 판단해야 할까요?"

- (A) 전부 Must니까 다 넣으면 된다
- (B) "이 기능 없이 서비스가 작동하는가?"로 다시 판단한다
- (C) 가장 만들기 쉬운 것만 Must로 한다
- (D) 경쟁 서비스에 있는 기능만 Must로 한다

정답: (B) — "전부 중요해요"는 "아무것도 중요하지 않다"와 같습니다. 가장 엄격한 질문으로 다시 판단하면 진짜 Must가 걸러집니다. 보통 Must는 전체 기능의 30% 이하입니다.

### 질문 3: MVP 함정
