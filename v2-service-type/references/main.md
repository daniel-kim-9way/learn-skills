## EXPLAIN

### 4가지 서비스 유형

서비스를 만드는 방법은 한 가지가 아닙니다. 같은 아이디어라도 **어떤 형태로 만드느냐**에 따라 완전히 다른 서비스가 됩니다.

비유: 같은 요리사(Claude)에게 맡기더라도, "레스토랑을 열 건지", "푸드트럭을 할 건지", "밀키트를 팔 건지", "요리 블로그를 할 건지"는 여러분이 결정해야 합니다.

---

### 유형 1: 웹앱 (Web Application) — "레스토랑"

사용자가 직접 들어와서 다양한 것을 합니다. 회원가입하고, 데이터를 입력하고, 검색하고, 관리합니다.

```
특징:
- 사용자가 반복적으로 방문
- 데이터를 입력하고 관리
- 여러 화면과 기능이 있음
- 회원가입/로그인이 보통 필요

적합한 경우:
- 사용자가 직접 데이터를 쓰고/올리고/기록
- 반복적으로 사용하는 서비스
- 여러 사용자 유형이 있음 (일반/관리자)
```

**한국 서비스 예시:**
- **당근마켓**: 물건 올리기, 채팅, 거래 관리 — 사용자가 반복적으로 들어와서 다양한 행동을 합니다
- **클래스101**: 강의 등록, 수강, 과제 제출, 수강평 — 학생과 강사가 각자 다른 기능을 사용
- **노션**: 문서 작성, 공유, 팀 협업 — 매일 들어와서 데이터를 입력하고 관리
- **토스**: 송금, 투자, 가계부 — 다양한 금융 기능을 하나의 앱에서

**핵심 질문:** "사용자가 매주 여러 번 들어와서 뭔가를 하나요?" → 그렇다면 웹앱입니다.

---

### 유형 2: 챗봇 (Chatbot) — "컨시어지"

대화하면서 원하는 것을 해결합니다. 질문하면 답해주고, 추천해주고, 상담해줍니다.

```
특징:
- 대화 형태의 인터페이스
- 1:1 맞춤 응답
- 복잡한 화면 없이 대화창 하나
- AI가 핵심 엔진

적합한 경우:
- 사용자마다 질문이 다름
- 맞춤 추천이 핵심 가치
- 복잡한 화면보다 대화가 자연스러움
```

**한국 서비스 예시:**
- **카카오톡 채널 챗봇**: 음식점 예약, 제품 문의, 고객 상담 — "예약하고 싶어요"라고 하면 날짜/시간/인원을 물어봄
- **네이버 클로바**: "오늘 날씨 어때?", "근처 맛집 추천해줘" — 대화로 정보를 얻음
- **은행 챗봇**: "잔액 조회", "카드 분실 신고" — 메뉴를 찾는 대신 말로 해결

**핵심 질문:** "사용자가 '질문하고 답을 받는 것'이 핵심인가요?" → 그렇다면 챗봇입니다.

---

### 유형 3: 자동화 (Automation) — "밀키트"

한번 설정하면 알아서 돌아갑니다. 사용자가 매번 직접 할 필요 없이, 규칙을 정해두면 자동 실행됩니다.

```
특징:
- "한번 설정 → 자동 반복"
- 사용자가 직접 조작하지 않아도 됨
- 정해진 조건이 되면 자동 실행
- 주로 비용 절감/시간 절약 목적

적합한 경우:
- 반복적이고 규칙적인 작업
- 사람이 매번 하기에 비효율적
- "트리거 → 액션" 패턴
```

**한국 서비스 예시:**
- **업무 자동화**: 매일 아침 어제의 매출 데이터를 정리해서 슬랙으로 보내주는 봇
- **이메일 자동 분류**: 고객 문의 이메일이 오면 카테고리별로 자동 분류
- **SNS 자동 포스팅**: 블로그 글을 올리면 자동으로 인스타/트위터에도 공유
- **재고 알림**: 상품 재고가 10개 이하면 자동으로 알림

**핵심 질문:** "한번 설정해놓으면 사람 손 안 대고 돌아가야 하나요?" → 그렇다면 자동화입니다.

---

### 유형 4: 크롬 익스텐션 (Chrome Extension) — "도구 모음"

이미 쓰고 있는 사이트 위에 기능을 덧붙입니다. 새 사이트를 만드는 게 아니라, 기존 사이트(쿠팡·노션·유튜브 등)에 내가 원하는 기능을 추가합니다.

```
특징:
- 브라우저 오른쪽 위에 설치되는 아이콘
- 이미 쓰는 사이트 위에서 동작
- 별도 앱/사이트가 없어도 됨
- 본인만 써도 OK (배포 안 해도 됨)

적합한 경우:
- 특정 사이트를 쓸 때 반복적으로 하는 작업이 있을 때
- 쿠팡/유튜브/노션 등 기존 사이트에 기능을 더하고 싶을 때
- 브라우저에서 쓰는 업무 도구를 만들고 싶을 때
```

**예시:**
- **쿠팡 최저가 알림**: 쿠팡 상품 페이지에서 가격 추적 버튼 추가
- **유튜브 자막 요약**: 유튜브 영상 페이지에서 자막을 Claude API로 요약
- **노션 템플릿 삽입**: 노션 페이지에서 버튼 하나로 본인 템플릿 삽입
- **업무 자동완성**: 사내 시스템에서 반복 입력을 원클릭으로

**핵심 질문:** "이미 쓰고 있는 특정 사이트에서 반복하는 동작이 있나요?" → 그렇다면 크롬 익스텐션입니다.

---

### 유형 결정 트리

아직 감이 안 잡히시나요? 레슨이 안내하는 4가지 질문을 따라가보세요:

```
Q1: 누가 사용하나요?
  본인만 → 크롬 익스텐션 / 자동화 쪽 (가볍게 시작)
  다른 사람도 → 다음 질문으로

Q2: 어디서 사용하나요?
  이미 쓰는 특정 사이트 위에서 → 크롬 익스텐션
  카카오톡/디스코드 안에서 → 챗봇
  브라우저 밖에서 자동 실행 → 자동화
  새로운 나만의 사이트에서 → 다음 질문으로

Q3: 데이터는 어디에 있나요?
  다른 사람이 올린 데이터도 보여야 함 → 웹서비스 (DB 필요)
  내 PC 파일이나 API에서 가져옴 → 크롬 익스텐션 / 자동화
  대화로 실시간 생성 → 챗봇

Q4: 다른 사람과 데이터를 공유하나요?
  YES (여러 사람이 같은 데이터를 봄) → 웹서비스
  NO (본인 또는 1:1만) → 크롬 익스텐션 / 챗봇 / 자동화
```

---

> **꿀팁: 하나만 골라야 하나요?**
>
> 실전에서는 조합도 가능합니다. 당근마켓도 **웹앱**(거래) + **챗봇**(1:1 대화)이 합쳐진 거예요.
> 하지만 **첫 버전(MVP)에서는 하나에 집중**하는 게 좋습니다. 두 마리 토끼를 잡으려다 둘 다 놓칩니다.
>
> 먼저 핵심 유형 하나로 시작하고, 성공하면 다른 유형을 붙이세요.

## EXECUTE

### Turn 1: 4가지 유형 소개

위의 EXPLAIN 내용을 바탕으로 4가지 유형을 소개합니다.

"서비스를 만드는 형태가 4가지 있어요. 같은 아이디어라도 어떤 형태로 만드느냐에 따라 완전히 달라집니다."

4가지 유형을 한국 서비스 예시와 함께 설명한 후 질문합니다:

"4가지를 들으셨는데, 감이 오시나요? 아직 아니어도 괜찮아요. 지금부터 질문으로 딱 맞는 유형을 찾아볼게요."

### Turn 2: 4가지 핵심 질문으로 유형 결정

레슨에서 안내하는 4가지 질문을 순서대로 한 번에 하나씩 던집니다.

**Q1: "누가 사용하나요?"**
- "본인만 쓸 거예요?" / "지인 몇 명?" / "모르는 사람도 쓸 건가요?"
- 본인만 → 크롬 익스텐션·자동화 쪽 먼저 고려
- 다른 사람도 → Q2로

**Q2: "어디서 사용하나요?"**
- "이미 쓰는 특정 사이트 위에서?" / "카톡·디스코드 안에서?" / "새 사이트로?"
- 기존 사이트 위 → 크롬 익스텐션
- 메신저 안 → 챗봇
- 브라우저 밖 자동 실행 → 자동화
- 새 사이트 → Q3으로

**Q3: "데이터는 어디에 있나요?"**
- "여러 사람이 올린 걸 다같이 보는 구조?" / "내 파일·API에서 가져오는 것?" / "대화로 실시간 생성?"
- 공유 DB 필요 → 웹서비스
- 내 데이터만 → 크롬 익스텐션·자동화
- 대화 기반 → 챗봇

**Q4: "다른 사람과 데이터를 공유하나요?"**
- YES (여러 사람이 같은 데이터를 봄) → 웹서비스 확정
- NO (본인 or 1:1) → 크롬 익스텐션·챗봇·자동화 중 앞 답변 기준 선택

각 질문 후 학습자의 답변에 맞게 다음 질문으로 이동합니다.

### Turn 3: 유형 추천 + 이유 설명

답변을 종합하여 추천 유형을 제시합니다:

"[학생님]의 [서비스명]은 **[유형]**이 가장 적합해 보여요."

이유를 3가지로 설명합니다:
1. 자가 진단 결과 근거
2. 유사한 한국 서비스와의 비교
3. 학생의 타겟 사용자 패턴

그리고 물어봅니다: "이 유형이 맞는 것 같으세요? 혹시 다른 유형이 더 끌리시나요?"

**학생이 동의하면:**
"좋아요! 확신이 드셨다면 바로 확정합시다."

**학생이 고민하면:**
"어떤 부분이 고민되세요? 예를 들어 [서비스명]에 챗봇 기능도 넣고 싶으신 건가요? 그건 나중에 추가할 수 있어요. 첫 버전은 [유형]으로 핵심을 잡고, 안정되면 확장하는 게 좋습니다."

**학생이 다른 유형을 원하면:**
"좋아요, [다른 유형]도 가능해요! 다만 한 가지 체크해볼게요 — [다른 유형]으로 만들면 [주의할 점]이 있는데, 괜찮으시겠어요?"

### Turn 4: 확정 + 향후 확장 논의

학생이 유형을 확정하면, 향후 확장 가능성을 함께 논의합니다:

"[유형]으로 시작하면, 나중에 이렇게 확장할 수 있어요:"

확장 경로 예시:
- 웹앱 → +챗봇 (고객 상담 기능 추가)
- 웹앱 → +자동화 (알림, 보고서 자동 생성)
- 크롬 익스텐션 → 웹앱 (사전예약에서 실제 서비스로)
- 챗봇 → +웹앱 (관리 대시보드 추가)

"최종적으로 **[유형]**으로 결정하시겠어요?"

### Turn 5: 파일 저장

아래 형식으로 `my-service-type.md`에 저장합니다:

```
서비스 유형 결정서

서비스명: [서비스명]
선택한 유형: [웹앱/챗봇/자동화/크롬 익스텐션]

자가 진단 결과:
- 반복 방문 여부: [YES/NO]
- 데이터 입력 여부: [YES/NO]
- 대화 핵심 여부: [YES/NO]
- 자동 실행 여부: [YES/NO]
- 소개+신청 목적 여부: [YES/NO]

선택 이유:
1. [자가 진단 결과 기반 이유]
2. [유사 서비스 비교 근거]
3. [타겟 사용자 패턴 근거]

유사 서비스: [참고할 수 있는 한국 서비스]
향후 확장 가능성: [현재 유형]에서 시작하여 → [확장 가능한 유형/기능]으로 발전 가능
```

"다음 레슨: `/v2-feature-spec` — 이 유형에 맞는 기능 명세서를 AI에게 받아보겠습니다!"

---

> **막혔을 때**
>
> "웹앱이랑 크롬 익스텐션 차이를 모르겠어요" → 크롬 익스텐션는 '전단지', 웹앱은 '가게'예요. 전단지는 보여주고 연락받는 게 끝이고, 가게는 사람들이 매일 들어와서 다양한 걸 해요.
>
> "두 가지가 다 필요한 것 같아요" → MVP에서는 하나만! "이 서비스가 딱 하나의 일만 해야 한다면?"이라고 스스로에게 물어보세요.
>
> "자동화랑 웹앱 차이가 헷갈려요" → 사용자가 "매번 직접 조작"하면 웹앱, "한번 설정하고 놔두면" 자동화예요.

## QUIZ

### 질문 1: 서비스 유형 판단

"반려동물 건강 기록 서비스를 만든다면, 가장 적합한 유형은?"

- (A) 웹앱 — 매일 건강 데이터를 입력하고 이력을 관리해야 하니까
- (B) 챗봇 — 반려동물 증상을 물어보면 답해주니까
- (C) 크롬 익스텐션 — 정보를 보여주면 되니까
- (D) 자동화 — 알아서 건강을 체크하니까

정답: (A) — 반복적 데이터 입력/관리가 핵심이므로 웹앱이 가장 적합합니다. 다만 "증상 상담"을 추가하고 싶다면, 나중에 챗봇 기능을 결합할 수 있습니다.

### 질문 2: MVP 원칙

"MVP(첫 버전)에서 가장 중요한 원칙은?"

- (A) 최대한 많은 기능을 넣어서 완벽하게 만든다
- (B) 하나의 유형에 집중하고, 핵심 기능만 넣는다
- (C) 여러 유형을 섞어서 다양한 사용자를 만족시킨다
- (D) 경쟁 서비스의 기능을 모두 따라 만든다

정답: (B) — 첫 버전은 하나의 유형에 집중해야 합니다. 모든 것을 넣으면 아무것도 제대로 못 만듭니다. "이것 하나만큼은 확실하게"가 MVP의 핵심입니다.

### 질문 3: 유형 확장

"크롬 익스텐션로 시작한 원데이 클래스 서비스가 인기를 끌었어요. 다음 단계로 가장 자연스러운 확장은?"

- (A) 챗봇을 추가해서 대화로 클래스 추천
- (B) 자동화로 전환해서 클래스를 자동 생성
- (C) 웹앱으로 확장해서 예약/결제/수강평 관리
- (D) 더 예쁜 크롬 익스텐션를 만든다
