## EXPLAIN

### 사용자 흐름이란?

사용자 흐름은 **"사용자가 목적을 달성하기까지 밟는 발걸음"**입니다.

비유: 이케아 매장에 가면 화살표가 바닥에 그려져 있죠? 입구 → 거실 → 침실 → 주방 → 계산대. 손님이 길을 잃지 않도록 동선을 설계한 겁니다. 여러분의 서비스도 "사용자의 동선"을 설계해야 합니다.

```
좋은 흐름 vs 나쁜 흐름

좋은 흐름:
홈 → 회원가입 → 첫 기능 사용 → 결과 확인 → 저장
(모든 발걸음이 연결되어 있음)

나쁜 흐름:
홈 → 회원가입 → ???
(회원가입 후 어디로 가야 할지 모름 = 막다른 길)
```

---

### 흐름 그리기 3가지 규칙

**규칙 1: 3-클릭 룰**

사용자가 원하는 것을 **3번의 클릭 이내**에 도달해야 합니다.

한국 앱으로 체감해봅시다:

```
카카오톡에서 메시지 보내기:
  1. 앱 열기 → 2. 대화방 선택 → 3. 메시지 입력 + 전송
  = 3번! 딱 맞죠?

당근마켓에서 물건 올리기:
  1. + 버튼 → 2. 사진 선택 → 3. 제목/가격 입력 + 등록
  = 3번! 핵심 행동까지 빠르게.

나쁜 예:
  1. 홈 → 2. 메뉴 → 3. 카테고리 → 4. 서브카테고리
  → 5. 검색 → 6. 결과 → 7. 상세
  = 7번! 사용자는 3번쯤에서 이미 포기합니다.
```

> **꿀팁:** 여러분이 자주 쓰는 앱을 열어보세요. 핵심 기능까지 몇 번 누르나요? 대부분 3번 이내일 거예요.

**규칙 2: 막다른 길 없애기**

모든 화면에서 **"다음에 뭘 할 수 있는지"**가 명확해야 합니다.

```
막다른 길 (Dead End):
  ┌─────────────────────┐
  │                     │
  │   결제 완료!        │
  │                     │
  │   ... 그다음은?     │
  │   (버튼도 없고,     │
  │    어디로 가야      │
  │    할지 모름)       │
  │                     │
  └─────────────────────┘

좋은 설계:
  ┌─────────────────────┐
  │                     │
  │   결제 완료!        │
  │                     │
  │  [주문 내역 보기]   │
  │  [홈으로 돌아가기]  │
  │  [영수증 보기]      │
  │                     │
  └─────────────────────┘
```

**규칙 3: 에러 복구**

사용자가 실수했을 때 되돌아갈 수 있어야 합니다.

```
나쁜 설계:
  삭제 버튼 클릭 → 영구 삭제됨 (복구 불가)
  "아... 잘못 눌렀는데..."

좋은 설계:
  삭제 버튼 클릭
  → "정말 삭제할까요?" [취소] [삭제]
  → 삭제됨 (30일 내 복구 가능)
  "실수해도 괜찮네!"
```

---

### 흔한 실수: 사용자가 길을 잃는 2가지 패턴

**실수 1: 막다른 길 (Dead End)**

사용자가 어떤 행동을 했는데, 다음에 할 수 있는 것이 아무것도 없는 화면.

```
예시:
  회원가입 완료!
  ↓
  "가입 완료되었습니다" (끝. 버튼 없음.)
  ↓
  사용자: "...그래서 어쩌라고?"

수정:
  회원가입 완료!
  ↓
  "가입 완료! 첫 프로필을 설정해볼까요?"
  [프로필 설정하기] [나중에 하기 → 홈으로]
```

**실수 2: 미아 상태 (Lost State)**

사용자가 지금 어디에 있는지 모르는 상태.

```
예시:
  설정 → 알림 설정 → 이메일 설정 → ???
  사용자: "내가 지금 어디 있는 거지?"

수정:
  설정 > 알림 설정 > 이메일 설정   ← 현재 위치 표시
  [← 뒤로 가기] 항상 보이게
```

카카오톡에서는 대화방 이름이 항상 상단에 보이죠? "내가 지금 누구와 대화하고 있는지" 항상 알 수 있습니다. 이게 미아 상태를 방지하는 설계입니다.

---

### 텍스트로 흐름도 그리기

코드 에디터 없이도 흐름을 그릴 수 있어요. 텍스트로 충분합니다!

**기본 패턴:**
```
[시작]
  │
  ▼
[화면1: 설명]
  │
  ├──→ [행동A] ──→ [화면2]
  │                  │
  │                  ▼
  │               [화면3: 결과]
  │
  └──→ [행동B] ──→ [화면4: 대안]

에러 시: [에러 화면] ──→ [복구 방법]
```

**실전 예시 — 당근마켓 "물건 올리기" 흐름:**
```
[홈 화면]
  │
  ▼
[+ 버튼 클릭]
  │
  ▼
[사진 선택 화면]
  │
  ├──→ 사진 촬영 ──→ [카메라]
  │                     │
  │                     ▼
  │                  [사진 확인] ──→ [등록 폼]
  │
  └──→ 갤러리에서 선택 ──→ [등록 폼]
                              │
                              ▼
                           [제목, 가격, 설명 입력]
                              │
                              ▼
                           [카테고리 선택]
                              │
                              ▼
                           [등록 완료!]
                              │
                              ├──→ [내 물건 보기]
                              └──→ [홈으로 돌아가기]
```

이런 식으로 여러분의 서비스 흐름도를 그리면 됩니다. 예쁘게 안 그려도 돼요. **빠진 길이 없는지**가 핵심입니다.

### 까다로운 사용자 3가지 유형

서비스를 만들 때, "이상적인 사용자"만 생각하면 안 됩니다. 실제로는 다양한 사용자가 옵니다:

```
유형 1: 귀찮은 고객 (Lazy User)
  "회원가입? 안 할래요. 그냥 바로 쓸 수 있어요?"
  "비밀번호 입력하기 귀찮은데..."
  "3번이나 눌러야 해요? 너무 복잡한데요"

  → 이 사용자를 위해: 최소한의 단계, 소셜 로그인

유형 2: 급한 고객 (Hurry User)
  "빨리 주문하고 싶은데, 어디서 하는 거예요?"
  "로딩이 왜 이렇게 오래 걸려요?"
  "검색 결과가 너무 많아서 뭘 골라야 할지 모르겠어요"

  → 이 사용자를 위해: 핵심 행동을 첫 화면에, 즐겨찾기

유형 3: 처음 쓰는 고객 (New User)
  "이게 뭐 하는 앱이에요?"
  "여기서 뭘 해야 하는 거예요?"
  "실수로 뭔가 삭제했어요. 되돌릴 수 있어요?"

  → 이 사용자를 위해: 온보딩 안내, 실수 복구, 도움말
```

이 3가지 유형이 여러분의 흐름에서 길을 잃지 않는지 확인하는 것이 **"까다로운 사용자 테스트"**입니다.

## EXECUTE

### Turn 1: 핵심 흐름 선택

학생에게 질문합니다:

"사용자가 [서비스명]에 들어와서 제일 중요한 일을 하기까지, 몇 번 클릭해야 할까요? 한번 상상해보세요."

"눈을 감고, 사용자가 앱을 처음 여는 순간을 떠올려보세요:"
- "뭐가 보이나요?"
- "첫 번째로 뭘 하나요?"
- "그다음은요?"
- "목표를 달성하면 뭐가 보이나요?"

my-priority-list.md의 Must 기능을 참고하며:

"Must 기능 중에서, 사용자가 **가장 먼저** 하게 될 행동을 골라주세요."

학생이 골라주면:
"좋아요! 그 행동의 시작점(어디에서 시작?)과 끝점(뭘 달성하면 끝?)을 함께 정해봅시다."

### Turn 2: 흐름 다이어그램 그리기

학생이 선택한 핵심 흐름을 텍스트 다이어그램으로 그립니다:

```
[서비스명] 핵심 사용자 흐름

목표: [사용자가 달성하려는 것]
사용자: [어떤 사용자]

[시작: 홈 화면]
  │
  ▼
[1단계: 화면 설명]
  사용자 행동: [무엇을 하는지]
  │
  ▼
[2단계: 화면 설명]
  사용자 행동: [무엇을 하는지]
  │
  ▼
[완료: 결과 화면]
  다음 행동: [여기서 할 수 있는 것들]
```

"클릭 수를 세봅시다. 시작부터 목표까지 몇 번인가요?"
- 3번 이내 → "3-클릭 룰을 만족합니다!"
- 4번 이상 → "줄일 수 있는 단계가 있을까요? 합칠 수 있는 화면은?"

나머지 Must 기능 2-3개도 같은 방식으로 흐름을 그립니다.

### Turn 3: 까다로운 사용자 역할극

"이제 제가 까다로운 사용자가 되어볼게요. 3가지 유형의 사용자가 여러분의 서비스를 쓰는 거예요."

**역할 1: 귀찮은 고객**
AI가 귀찮은 고객처럼 말합니다:

"저 처음 왔는데, 회원가입 안 하면 아무것도 못 하나요? 회원가입 귀찮은데... 그냥 구경이라도 할 수 있어요?"

학생의 흐름에서 비회원 접근이 가능한지 확인합니다.

**역할 2: 급한 고객**
"저 지금 급한데요! [핵심 기능] 바로 쓸 수 있어요? 어디 있어요? 홈에서 한 번에 안 보이는데요?"

학생의 흐름에서 핵심 기능까지의 경로가 빠른지 확인합니다.

**역할 3: 처음 쓰는 고객**
"이 앱이 뭐 하는 건지 잘 모르겠어요. 여기서 뭘 해야 하는 거예요?"
"실수로 [중요한 것]을 삭제했어요. 되돌릴 수 있나요?"
"[기능]을 쓰다가 중간에 앱을 닫았어요. 다시 열면 어디부터예요?"

학생의 흐름에서 온보딩과 에러 복구가 있는지 확인합니다.

각 역할극 후, 학생의 흐름에서 발견된 문제점을 함께 정리합니다:

```
까다로운 사용자 테스트 결과

귀찮은 고객:
  [발견된 문제] → [해결 방안]

급한 고객:
  [발견된 문제] → [해결 방안]

처음 쓰는 고객:
  [발견된 문제] → [해결 방안]
```

### Turn 4: 문제점 수정

발견된 문제를 하나씩 수정합니다:

"[문제1]은 여기에 [해결 방법]을 추가하면 해결돼요."

학생과 함께 수정된 흐름을 만듭니다. 수정 전/후를 비교해서 보여줍니다:

```
수정 전:
  홈 → 회원가입 → ??? (막다른 길)

수정 후:
  홈 → 회원가입 → "가입 완료! 첫 [기능]을 해볼까요?"
                   [시작하기] [나중에 하기 → 홈]
```

### Turn 5: 3가지 규칙으로 최종 검증

완성된 흐름을 3가지 규칙으로 검증합니다:

"마지막으로 3가지만 체크합시다."

```
최종 검증 체크리스트

□ 막다른 길 없음?
  → 모든 화면에 "다음에 할 수 있는 행동"이 있나?
  → 완료 화면, 에러 화면도 포함해서 확인

□ 3-클릭 룰?
  → 시작부터 핵심 행동까지 3클릭 이내?
  → 자주 쓰는 기능일수록 클릭 수가 적어야 함

□ 에러 복구?
  → 실수했을 때 되돌아갈 수 있나?
  → "정말 삭제할까요?" 같은 확인 단계가 있나?
  → 뒤로 가기가 모든 화면에서 작동하나?
```

하나라도 문제가 있으면 함께 수정합니다.

### Turn 6: 파일 저장

검증 완료된 흐름을 `my-user-flow.md`로 저장합니다.

저장 내용:
- 핵심 흐름 다이어그램 (Must 기능별)
- 까다로운 사용자 테스트 결과
- 3가지 규칙 검증 결과
- 수정 사항 반영 여부

"다음 레슨: `/v2-blueprint` — 이 흐름을 기술 설계도로 만들어봅니다!"

> **막혔을 때**
>
> "흐름이 너무 복잡해졌어요" → 한 가지 목표의 흐름만 먼저 그리세요. "사용자가 [핵심 행동]을 하기까지"만. 나머지 흐름은 하나씩 추가하면 됩니다.
>
> "까다로운 사용자 대답을 못 하겠어요" → 여러분이 다른 앱을 처음 쓸 때를 생각해보세요. "어디를 눌러야 하지?" 했던 경험, "실수로 삭제해서 당황"했던 경험. 그때의 여러분이 바로 "까다로운 사용자"입니다.
>
> "3-클릭 안에 안 돼요" → 2개의 화면을 합칠 수 있는지 보세요. "설정 → 알림 설정"을 "설정(알림 포함)"으로 합칠 수 있나요? 또는 자주 쓰는 기능은 홈에서 바로 접근 가능하게 만드세요.

## QUIZ

### 질문 1: 사용자 흐름 문제

"사용자가 회원가입을 완료했는데, '가입 완료!' 화면만 보이고 아무 버튼도 없습니다. 이건 어떤 문제인가요?"

- (A) 3-클릭 룰 위반
- (B) 막다른 길 (Dead End)
- (C) 에러 복구 부재
- (D) 문제 없음

정답: (B) — "가입 완료!" 후에 버튼이 없으면 사용자는 어디로 가야 할지 모릅니다. "홈으로 가기", "프로필 설정하기" 같은 다음 행동을 반드시 제공해야 합니다.

### 질문 2: 사용자 경험 설계

"검색 결과가 0건일 때 가장 좋은 대응은?"

- (A) "결과 없음"만 표시
- (B) "결과 없음" + [다른 키워드로 검색] + [인기 항목 보기]
- (C) 자동으로 홈으로 돌아감
- (D) 에러 페이지를 보여줌

정답: (B) — "결과 없음"만 보여주면 막다른 길입니다. 대안 행동(다른 검색, 인기 항목)을 제시하면 사용자가 서비스 안에 머물 수 있습니다.

### 질문 3: 까다로운 사용자 대응

"처음 앱을 다운받은 사용자가 '이 앱이 뭐 하는 건지 모르겠어요'라고 합니다. 가장 좋은 대응은?"

- (A) 사용 설명서를 PDF로 제공한다
- (B) 첫 화면에서 핵심 기능을 3단계로 안내한다 (온보딩)
- (C) 고객센터 전화번호를 보여준다
- (D) 회원가입부터 시킨다

정답: (B) — 처음 온 사용자에게는 "이 앱은 이런 걸 해요" → "한번 해보세요" → "잘했어요!"라는 짧은 온보딩이 가장 효과적입니다. 설명서를 읽을 사용자는 거의 없습니다.
