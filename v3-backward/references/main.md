## EXPLAIN

### "레고 완성 사진" 전략

레고를 조립할 때 가장 먼저 하는 일은 무엇일까요?

설명서를 읽는 것? 부품을 분류하는 것? 아닙니다. **완성 사진을 보는 것**입니다.

완성된 모습을 알아야 조립하면서 "지금 어디쯤 왔는지" 감을 잡을 수 있습니다. 성(城)을 만들고 있다는 걸 알아야, 지금 끼우는 블록이 탑인지 벽인지 이해가 되죠.

서비스 빌딩도 마찬가지입니다. 코드를 한 줄도 모르는 상태에서 처음부터 만들면 막막합니다. 그래서 우리는 **반대로** 합니다:

1. 이미 완성된 서비스를 먼저 본다
2. 직접 써본다 (사용자처럼!)
3. 코드가 어떻게 구성되어 있는지 구경한다
4. "내 서비스도 이런 구조로 만들면 되겠구나" 하고 감을 잡는다

이것이 **Working Backward** 방식입니다. 완성에서 시작으로 거꾸로 가는 전략이에요. 아마존(Amazon)도 이 방식으로 신제품을 기획합니다.

### 왜 완성된 걸 먼저 봐야 할까?

피아노를 배울 때를 떠올려보세요. 악보 읽는 법부터 배우면 재미가 없습니다. 하지만 선생님이 먼저 멋진 곡을 연주해주면? "나도 저렇게 하고 싶다!"는 마음이 생기죠.

완성된 서비스를 먼저 보면:
- **"이 정도면 나도 만들 수 있겠다"** — 생각보다 단순한 구조에 안심이 됩니다
- **내 서비스와 비교할 수 있다** — "이 기능은 내 서비스에도 필요하겠네"
- **코드 구조가 미리 눈에 익는다** — 나중에 내가 만들 때 "아, 그 구조구나" 연결됩니다

### 샘플 앱: 독서 모임 서비스

우리가 살펴볼 샘플 앱은 **독서 모임 서비스**입니다. Claude에게 직접 만들어달라고 해서 체험할 겁니다.

기능은 간단합니다:
- 회원가입/로그인
- 책 목록 보기
- 책 추가/수정/삭제
- 읽은 책에 감상 남기기

"이게 끝이야?" 싶으시죠? 네, 이게 끝입니다. 하지만 이 간단한 앱 안에 서비스의 **핵심 구조가 전부** 들어있습니다. 배달의민족도, 당근마켓도, 넷플릭스도 결국 이 구조의 확장입니다.

- 배달의민족: 음식 목록 보기 + 주문 추가 + 리뷰 남기기
- 당근마켓: 물건 목록 보기 + 물건 등록 + 채팅하기
- 독서 모임: 책 목록 보기 + 책 추가 + 감상 남기기

패턴이 보이시나요? **목록 → 추가 → 상세 → 연결된 활동**. 이 패턴이 모든 서비스의 뼈대입니다.

### 코드 구조 = 건물의 층별 안내도

Rails 프로젝트의 코드는 정해진 구조로 정리되어 있습니다. 마치 건물에 층별 안내도가 있는 것처럼요:

| 폴더 | 역할 | 건물 비유 |
|------|------|----------|
| `app/models/` | 데이터 구조 정의 | 지하 창고 — 물건을 어떻게 보관할지 정함 |
| `app/views/` | 화면에 보이는 부분 | 인테리어 — 손님이 보는 공간 |
| `app/controllers/` | 요청을 처리하는 로직 | 안내 데스크 — 손님 요청을 받아서 처리 |
| `config/routes.rb` | URL 경로 설정 | 건물 입구의 층별 안내판 |
| `db/` | 데이터베이스 파일 | 실제 창고 (물건이 저장되는 곳) |

지금은 이 구조를 **외울 필요가 전혀 없습니다**. "아, 이런 식으로 나뉘어 있구나" 정도만 느끼면 됩니다. 나중에 내 서비스를 만들면 자연스럽게 익숙해집니다.

## EXECUTE

### 단계 1: 샘플 앱 만들기

Claude에게 이렇게 요청하세요:
```
sample-bookclub이라는 Rails 샘플 프로젝트를 만들어줘.
독서 모임 서비스야.

기능:
- 회원가입/로그인 (Rails 8 내장 인증)
- 책 목록 보기, 추가, 수정, 삭제
- 각 책에 감상(리뷰) 남기기
- 로그인한 사람만 책 추가/감상 남기기 가능
- 홈페이지에 서비스 소개

Tailwind CSS로 깔끔하게 디자인해줘.
샘플 데이터도 3개 정도 넣어줘.
```

Claude가 프로젝트를 생성합니다. 에디터 왼쪽에 수많은 파일이 나타나는 것을 확인하세요.

> 시간이 좀 걸릴 수 있습니다 (3~5분). Claude가 파일을 하나씩 만들고, 데이터베이스를 설정하고, 샘플 데이터를 넣는 과정입니다. 에러가 나면 당황하지 마세요 — 에러 메시지를 Claude에게 그대로 보여주세요.

### 단계 2: 서버 실행 — "서비스가 살아있다!"

Claude에게 이렇게 요청하세요:
```
이 프로젝트를 실행해줘. 브라우저에서 볼 수 있게 해줘.
```

Claude가 필요한 설정(데이터베이스 생성 등)을 하고 서버를 시작합니다.

이제 브라우저(크롬, 사파리 등)를 열고 주소창에 이렇게 입력하세요:

```
http://localhost:3000
```

**localhost** — 이 단어를 기억하세요. "내 컴퓨터에서만 보이는 비밀 주소"입니다. 옆에 앉은 사람도, 같은 와이파이를 쓰는 사람도 이 주소로는 아무것도 볼 수 없습니다. 오직 여러분의 컴퓨터에서만 보이는 비밀 서비스입니다.

화면이 나타나면... 축하합니다! **이것이 완성된 웹 서비스의 모습**입니다.

> 에러가 나면: 에러 메시지를 복사해서 Claude에게 보여주세요. "이런 에러가 나왔어"라고 말하면 됩니다. Claude가 고쳐줍니다.

### 단계 3: 사용자처럼 체험하기 — 서비스 투어

이제 여러분은 이 서비스의 **첫 번째 사용자**입니다. 실제 사용자처럼 앱을 써봅시다:

**투어 코스:**

1. **홈페이지 둘러보기**: 서비스 소개가 보이나요? "시작하기" 같은 버튼이 있나요?

2. **회원가입하기**: 가입 페이지로 가서 이메일과 비밀번호를 입력하세요. (가짜 이메일도 OK — test@test.com 같은 거)

3. **로그인 확인**: 가입 후 자동으로 로그인되었나요? 네비게이션 바가 바뀌었나요?

4. **책 목록 보기**: 샘플로 넣어둔 책들이 보이나요? 몇 권인가요?

5. **책 추가하기**: 여러분이 좋아하는 책 1권을 추가해보세요.

6. **감상 남기기**: 아무 책이나 클릭해서 감상을 남겨보세요.

7. **수정/삭제**: 방금 추가한 책을 수정해보세요. 그리고 삭제도 해보세요.

8. **로그아웃**: 로그아웃하면 어떤 것들이 안 보이게 되나요?

**체험하면서 메모해보세요:**
- "이 기능은 내 서비스에도 필요하겠다" → 어떤 기능?
- "이건 내 서비스에서는 다르게 하고 싶다" → 어떻게?
- "이 부분이 좋은데, 비슷하게 만들고 싶다" → 어떤 부분?

> 이 메모가 나중에 여러분의 서비스를 만들 때 큰 도움이 됩니다.

### 단계 4: 코드 구조 탐색 — 건물 안 들여다보기

이제 사용자 시점에서 벗어나, **건물 내부**를 들여다봅니다.

Claude에게 이렇게 요청하세요:
```
이 프로젝트의 코드 구조를 비개발자도 이해할 수 있게 설명해줘.
건물에 비유해서 각 폴더가 무슨 역할을 하는지 알려줘.

특히:
- 방금 회원가입할 때 어떤 파일이 동작한 건지
- 책 목록을 볼 때 어떤 파일이 화면을 만든 건지
- 책을 추가할 때 데이터가 어디에 저장된 건지

흐름을 따라가면서 설명해줘.
```

Claude가 프로젝트 구조를 안내합니다. 에디터 왼쪽의 파일 탐색기에서 직접 폴더를 클릭해보세요.

- `app/views/` 폴더를 열어보세요 → 여기에 화면(HTML)이 있습니다
- `app/models/` 폴더를 열어보세요 → 여기에 데이터 구조가 있습니다
- `app/controllers/` 폴더를 열어보세요 → 여기에 처리 로직이 있습니다

파일을 클릭해서 코드를 봐도 이해가 안 되는 게 정상입니다. **구조만 느끼면 됩니다.** "아, 화면은 views에 있고, 데이터는 models에 있구나."

### 단계 5: 내 서비스와 매핑하기 — "이 정도면 나도 만들 수 있겠다"

이제 가장 중요한 단계입니다. 이 독서 모임 앱의 구조를 **여러분의 서비스**에 대입해봅니다.

Claude에게 이렇게 요청하세요:
```
내 서비스는 [C2에서 정리한 서비스 설명]이야.
핵심 기능은 [C2에서 정한 Must 기능들]이야.

이 독서 모임 앱의 구조를 참고해서,
내 서비스가 어떤 구조가 될지 매핑해줘.

예를 들어:
- 독서 모임의 "책" → 내 서비스의 "[핵심 콘텐츠]"
- 독서 모임의 "감상" → 내 서비스의 "[사용자 활동]"
- 독서 모임의 "회원가입" → 내 서비스에서도 동일

이런 식으로 1:1 대응표를 만들어줘.
```

이 매핑을 보면 깨달음이 옵니다:

> "독서 모임에서 '책'을 관리하는 것이나, 내 서비스에서 '[핵심 콘텐츠]'를 관리하는 것이나, 구조가 **똑같네**?"

맞습니다. 서비스의 겉모습은 다르지만, 안의 구조는 놀랍도록 비슷합니다. **이것이 "나도 만들 수 있겠다"의 근거**입니다.

### 단계 6: 매핑 결과 저장

Claude에게 이렇게 요청하세요:
```
지금 만든 매핑 내용을 my-service-mapping.md 파일로 저장해줘.
나중에 내 서비스를 만들 때 참고할 거야.
```

### 단계 7: 샘플 앱 정리

샘플 앱은 참고용이었으니, 구경을 마쳤으면 정리합니다:

```
서버를 멈춰줘.
```

> 꿀팁: 샘플 앱 폴더를 삭제하지 않아도 됩니다. 나중에 "이 부분 어떻게 되어 있었지?" 할 때 다시 볼 수 있으니까요. 다만 내 진짜 서비스와 헷갈리지 않도록 폴더 이름에 'sample-'이 붙어있는 걸 기억하세요.
