## EXPLAIN

### API 키를 코드에 직접 적으면 어떤 일이 생기나

먼저 짧은 가상 시나리오 하나.

> 어떤 비개발자가 처음 OpenAI API 키를 발급받고 흥분해서 본인 코드에 이렇게 적었다.
> ```python
> client = OpenAI(api_key="sk-proj-abc123real-key-here-xyz...")
> ```
> 그리고 코드를 GitHub에 push했다. 24시간 뒤, OpenAI에서 이메일이 왔다. "이 시간 동안 1,200달러어치 사용. 결제 카드 청구 예정."
>
> 누가 썼나? 본인이 아니다. **GitHub을 자동으로 스캔하는 봇**이 본인 키를 1초 만에 찾아내서, 본인 키로 GPT-4를 마구 호출한 것.

이건 가상 시나리오지만 실제로 자주 일어나요. GitHub에서 API 키 노출은 **24시간 안에 99% 발견**되고, 발견되면 자동으로 도용돼요.

코드에 직접 키를 적으면 안 되는 이유가 그래서예요.

- 코드를 GitHub에 올리면 비밀이 그대로 인터넷 공개 (해커 봇이 자동 스캔)
- 개발 환경과 운영 환경에서 다른 키를 써야 할 때 코드 수정 필요
- 팀원이 각자 본인 키를 쓰려면 코드를 분리해야 함

### 해결: 환경변수로 분리

**환경변수**는 코드와 분리해서 관리하는 값이에요. 키를 별도 파일(`.env`)에 두고, 코드에서는 변수 이름만 참조해요.

```ruby
# ❌ 나쁜 예
client = Anthropic.new(api_key: "sk-ant-real-key-...")

# ✅ 좋은 예
client = Anthropic.new(api_key: ENV["ANTHROPIC_API_KEY"])
```

`.env` 파일은 `.gitignore`에 등록되어 git이 자동으로 무시해요. 즉, GitHub에 올라가지 않아요. 본인 PC 안에만 있어요.

비유하면 **집 열쇠**예요. 열쇠를 명함에 적어서 길에 뿌리는 사람은 없죠? 열쇠는 본인 주머니에만, 꺼낼 때만. `.env`가 그 ‘본인 주머니’예요.

### .env vs Rails credentials — 두 가지 옵션

Rails는 환경변수 관리에 두 가지 옵션을 제공해요. 비교:

| | `.env` (dotenv) | Rails credentials |
|---|---|---|
| 파일 | `.env` (평문) | `config/credentials.yml.enc` (암호화) |
| git 커밋? | 안 됨 (gitignore) | 됨 (암호화돼서 OK) |
| 키 관리 | 각자 본인 PC만 | `master.key` 1개 공유 필요 |
| 추천 시나리오 | 1인 / 빠른 시작 / 작은 팀 | 큰 팀 / 배포 환경 분리 필요 |

**본 코스 권장: `.env`** — 1인 비개발자에게는 충분하고 가장 빨라요. 팀이 커지거나 배포 환경 분리가 필요해지면 그때 credentials로 옮겨도 늦지 않아요.

### LLM이 뭐고, 왜 키가 필요한가

**LLM**(Large Language Model) = Claude, GPT 같은 ‘대형 언어 모델’이에요. 본인 서비스에 ‘AI 기능’을 붙인다는 건, 사실상 본인 서비스가 LLM을 호출해서 결과를 받아오는 거예요.

LLM은 무료가 아니에요. 한 번 호출할 때마다 작은 비용(보통 1원 미만 ~ 수십 원)이 들어요. 그래서 LLM 회사(Anthropic, OpenAI)는 본인을 식별하려고 **API 키**를 발급해요. 본인 키로 호출하면 본인 카드에 청구돼요. 그래서 키가 새 나가면 큰일 나는 거예요.

본 코스는 **Anthropic Claude**를 메인으로 권장. Claude는 한국어가 자연스럽고 긴 문맥을 잘 다뤄요. OpenAI는 이미지 생성·음성·임베딩 같은 보조 영역에서 유용.

```

### Turn 6: 본인 서비스에 AI 기능 붙이는 사례 3가지

```

#### 사례 ③ 자동 답변 초안 (Draft Generation)

> "상황과 맥락을 보고, 사람이 검토할 답변 초안을 AI가 만들어 줌."

```

- SNS 댓글 → 친근한 톤 / 정중한 톤 초안 2개 동시
```

> "이 3가지가 가장 빠르고 비용 효율 좋아요. 더 화려한 것(이미지 생성, 음성 등)은 다음 챕터에서 다뤄요."

## 자주 막히는 지점

**".env에 키를 추가했는데 코드에서 못 읽어요"**
→ 서버 재시작 필요. `bin/dev`가 도는 터미널에서 `Ctrl + C` → `bin/dev` 다시. `.env`는 서버가 시작될 때 1회만 읽혀요.

**"실수로 .env를 git에 올렸어요"**
→ 즉시 다음 3가지 동시에:
1. 그 키를 발급처(Anthropic / OpenAI / Google 등)에서 폐기 + 새로 발급
2. Claude Code에 "이 파일을 git 이력에서 완전히 제거해 줘. .env 파일." 한 줄 (BFG Repo-Cleaner 또는 git filter-branch 사용)
3. `git push --force`로 GitHub의 이력도 덮어쓰기

**"GitHub에서 ‘leaked secret detected’ 이메일 왔어요"**
→ GitHub Secret Scanning이 잡은 거예요. 위와 같이 즉시 키 폐기 + 재발급 + 이력 정리. 발급처에서도 자동으로 키를 비활성화한 경우가 많아요.

**"무료 크레딧이 너무 빨리 다 떨어져요"**
→ Claude/GPT 모델을 작은 모델로 바꿔요 (예: claude-3-5-haiku, gpt-4o-mini). 1~10배 저렴. 또는 호출당 max_tokens를 작게(예: 500) 잡아 반환 글자 수를 줄여요.

**".env.example은 뭐예요?"**
→ `.env`의 ‘템플릿’이에요. 실제 값은 빈 문자열(또는 `your_key_here`)로 두고 git에 같이 올려요. 다른 사람이 본인 코드를 받았을 때 ‘아, 이런 키들이 필요하구나’ 알 수 있어요. 본인 키가 노출되지는 않아요.
