## EXPLAIN

### AI의 치명적 약점: 기억상실

Claude는 똑똑합니다. 코드도 짜고, 오류도 고치고, 설명도 잘합니다.

하지만 치명적인 약점이 하나 있습니다: **새 대화를 시작하면 이전 대화를 전부 잊습니다.**

영화 "메멘토"를 아시나요? 주인공이 매일 아침 기억을 잃어서, 중요한 것들을 몸에 문신으로 새겨둡니다. Claude도 비슷합니다. 아무리 긴 대화를 나눠도, 대화창을 닫는 순간 모든 기억이 사라집니다.

여러분이 C1에서 열정적으로 서비스 아이디어를 설명하고, C2에서 꼼꼼히 기획서를 만들었어도 — 새 대화를 열면 Claude는 이렇게 말합니다:

> "어떤 서비스를 말씀하시는 건가요?"

... 허탈하죠?

### 해결책: CLAUDE.md — "새 직원 온보딩 문서"

회사에 새 직원이 입사하면 어떻게 하나요? 회사 소개 문서, 프로젝트 현황, 업무 규칙 등을 담은 **온보딩 문서**를 줍니다. 새 직원이 이 문서를 읽으면 "아, 이 회사가 이런 일을 하는구나, 지금 이 프로젝트를 하고 있구나" 바로 파악할 수 있죠.

**CLAUDE.md**가 바로 그 온보딩 문서입니다.

프로젝트 폴더에 CLAUDE.md 파일을 놓으면, Claude Code가 새 대화를 시작할 때 **이 파일을 가장 먼저 읽습니다.** 자동으로요. 별도 설정 없이요.

비유를 정리하면:
- 기억을 잃는 주인공 → Claude (매 대화마다 리셋)
- 몸에 새긴 문신/메모 → **CLAUDE.md** (매 대화 시작 시 자동으로 읽음)
- 새 직원 온보딩 문서 → **CLAUDE.md** (프로젝트의 모든 맥락을 담은 문서)

### CLAUDE.md에 뭘 적나요?

새 직원에게 알려줘야 할 것을 생각하면 됩니다:

**필수 항목 (최소한 이 세 가지):**
1. **내 서비스가 뭔지** — 한 줄 설명
2. **핵심 기능** — C2에서 정한 Must 기능 목록
3. **현재 진행 상태** — 어디까지 했는지

**있으면 더 좋은 것:**
- 대상 사용자 (C1 페르소나)
- 기술 스택 (Ruby on Rails, Tailwind CSS 등)
- 서비스의 톤앤매너 (친근한? 전문적인?)
- 주의사항이나 규칙 ("User 모델은 건드리지 마" 같은)

### CLAUDE.md가 있을 때와 없을 때

**CLAUDE.md가 없으면** — 매번 이렇게 시작해야 합니다:
```
나는 [서비스명]을 만들고 있어.
대상 사용자는 [누구]야.
핵심 기능은 A, B, C야.
기술은 Rails 쓰고 있어.
지금 A까지 완성했고 B를 할 차례야.
디자인은 Tailwind CSS 쓰고 있어.
아, 그리고 데이터베이스 구조는...
```

매번 이걸 타이핑한다고 생각해보세요. 10줄이 넘는 설명을 매 대화마다? 지치겠죠.

**CLAUDE.md가 있으면:**
```
B 기능 이어서 만들자.
```

한 줄이면 됩니다. Claude가 나머지를 CLAUDE.md에서 읽으니까요.

이것이 CLAUDE.md의 힘입니다. **여러분의 시간을 아껴주고, Claude의 답변 품질을 높여줍니다.**

## EXECUTE

### 단계 1: AI 기억상실 직접 체험 — "이것이 문제입니다"

먼저 문제를 직접 느껴봅시다. 새 대화를 시작합니다.

> 중요: 반드시 **새 대화**여야 합니다. 기존 대화에서 하면 Claude가 기억하고 있어서 문제를 체감할 수 없습니다.

Claude에게 이렇게 물어보세요:
```
내 서비스의 핵심 기능이 뭐야?
```

Claude의 반응을 보세요:

> "어떤 서비스를 말씀하시는 건가요? 서비스에 대해 알려주시면 도움을 드릴 수 있습니다."

...이것이 문제입니다. C1에서 며칠에 걸쳐 정한 서비스 아이디어, C2에서 꼼꼼히 만든 기획서 — Claude는 아무것도 모릅니다.

한 가지 더 물어보세요:
```
대상 사용자가 누구야?
```

역시 모릅니다. 이 좌절감을 기억하세요. 이 문제를 지금 해결할 겁니다.

### 단계 2: C1/C2 산출물 확인

CLAUDE.md를 쓰기 전에, C1과 C2에서 만든 것들을 확인합니다.

Claude에게 이렇게 요청하세요:
```
현재 폴더에 있는 md 파일들을 보여줘.
C1에서 만든 기획서와 C2에서 만든 설계서를 찾아줘.
```

C1 기획서에서 가져올 것:
- 서비스 이름
- 가치 선언문 (한 줄 소개)
- 대상 사용자 (페르소나)

C2 설계서에서 가져올 것:
- Must 기능 목록
- 서비스 유형
- 화면 흐름

이 내용들이 CLAUDE.md의 재료가 됩니다.

### 단계 3: CLAUDE.md 함께 작성

Claude에게 이렇게 요청하세요:
```
내 C1 기획서와 C2 설계서를 읽고,
프로젝트 루트에 CLAUDE.md 파일을 만들어줘.

아래 구조로 작성해줘:

# [서비스이름]

## 서비스 소개
- 한 줄 설명: [C1 가치 선언문]
- 대상 사용자: [C1 페르소나]
- 서비스 유형: [C2에서 정한 유형]

## 핵심 기능
1. [C2 Must 기능 1]
2. [C2 Must 기능 2]  
3. [C2 Must 기능 3]

## 기술 스택
- Ruby on Rails 8
- Tailwind CSS
- SQLite

## 현재 상태
- C3 학습 중, 아직 코드 작성 전

## 규칙
- 한국어로 대화
- 비개발자도 이해할 수 있게 설명
- 코드 변경 시 항상 이유를 설명
```

Claude가 프로젝트 루트(최상위 폴더)에 CLAUDE.md 파일을 생성합니다.

### 단계 4: CLAUDE.md 내용 확인

에디터에서 CLAUDE.md 파일을 직접 클릭해서 열어봅니다.

확인할 것:
- 서비스 이름이 맞나요?
- 한 줄 설명이 C1에서 정한 것과 일치하나요?
- 핵심 기능이 C2에서 정한 것과 같나요?
- 빠진 내용은 없나요?

수정이 필요하면:
```
CLAUDE.md에서 [수정할 부분]을 [이렇게] 바꿔줘.
```

### 단계 5: 기억 테스트 — "마법의 순간"

이제 진짜 마법을 체험합니다.

**새 대화를 시작합니다.** (이것이 핵심입니다 — 반드시 새 대화!)

그리고 단계 1에서 했던 것과 **똑같은 질문**을 합니다:
```
내 서비스의 핵심 기능이 뭐야?
```

Claude의 답변을 보세요:

이번에는 정확하게 대답합니다! 서비스 이름도, 기능 목록도, 대상 사용자도 — 다 알고 있습니다.

**같은 질문, 완전히 다른 답변.** 달라진 것은 딱 하나 — CLAUDE.md 파일의 존재입니다.

> 이것이 CLAUDE.md의 힘입니다.

한 가지 더 테스트해보세요:
```
대상 사용자가 누구야?
```

정확히 대답하죠? 아까는 "어떤 서비스요?" 했던 Claude가, 이제는 여러분의 프로젝트를 완벽히 이해하고 있습니다.

### 단계 6: CLAUDE.md 업데이트 연습

CLAUDE.md는 한 번 쓰고 끝이 아닙니다. 프로젝트가 진행되면 **현재 상태도 업데이트**해야 합니다.

Claude에게 이렇게 요청하세요:
```
CLAUDE.md의 현재 상태를 업데이트해줘.
"v3-memory 완료. CLAUDE.md 작성 완료."로 바꿔줘.
```

이 업데이트는 **앞으로 매 레슨이 끝날 때마다** 합니다. 그래야 Claude가 "지금 어디까지 진행했는지" 항상 알 수 있습니다.

### 단계 7: CLAUDE.md 업데이트 체크리스트

앞으로 CLAUDE.md를 언제 업데이트해야 할까요? 체크리스트를 만들어둡시다:

- 레슨을 완료했을 때 → "현재 상태" 업데이트
- 새 기능을 추가했을 때 → "핵심 기능" 아래에 구현 상태 표시
- 새 모델(데이터 테이블)을 만들었을 때 → "데이터 구조" 섹션 추가
- 중요한 결정을 했을 때 → "규칙"에 추가
- 에러가 반복될 때 → "주의사항"에 추가

Claude에게 이렇게 요청하세요:
```
CLAUDE.md에 "업데이트 시점"이라는 섹션을 추가해줘.
위의 체크리스트 내용을 넣어줘.
```

이 체크리스트가 있으면, "아 CLAUDE.md 업데이트해야 하는데..." 하고 빼먹는 일이 줄어듭니다.

> 꿀팁: CLAUDE.md 업데이트가 귀찮게 느껴질 수 있습니다. 하지만 생각해보세요. 매번 10줄씩 배경 설명을 타이핑하는 것과, 1줄만 업데이트하는 것. 어느 쪽이 더 귀찮을까요? CLAUDE.md는 "미래의 나"를 위한 투자입니다.
