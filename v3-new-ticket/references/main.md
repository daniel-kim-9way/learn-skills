## EXPLAIN

### 왜 본인이 직접 티켓을 만들어야 하나

빌드킷에는 PLAN(Ch.2) 시점에 정해 둔 티켓들이 미리 들어 있어요. 그건 "출발선의 계획"이에요. 그런데 BUILD를 진행하면서 본인 머릿속에는 새로운 아이디어가 계속 떠올라요.

- "써 보니 이 화면에 ○○ 버튼이 하나 더 있어야겠다"
- "카톡 공유 기능 하나만 더 있으면 좋겠다"
- "방금 본 그 버그, 잊기 전에 적어 둬야 하는데"
- "이 기능에 AI 자동 요약을 붙이면 어떨까"

이런 생각이 떠올랐을 때 머릿속에만 두면 잊혀요. **티켓으로 옮겨 두면** AI 팀이 본인 대신 처리해 줘요. 티켓 만들기는 무겁지 않아요. 한 줄 명령이면 됩니다.

이번 스킬에서 본인은 **첫 신규 티켓 1개**를 만들어 봅니다. 한 번 해 보면 그 다음부터는 자연스러워져요.

### AI 팀 에이전트 4종류 — 누구에게 무엇을 시키나

본인 빌드킷에는 보통 다음 4종류의 에이전트가 들어 있어요. 각각 다른 전문가로 생각하세요. 한 사람이 다 한다고 생각하면 결과 품질이 떨어져요.

| 에이전트 | 슬러그 | 전문 영역 | 언제 배정 |
|---|---|---|---|
| Rails 개발자 | `02-rails-dev` | 백엔드·DB·API·테스트 코드(TDD) | 기능 구현·버그 수정·DB 변경·API 추가 |
| UX/UI 디자이너 | `ux-designer` | 화면 레이아웃·디자인 시스템·접근성·반응형 | 새 화면·디자인 polish·반응형 깨짐 수정 |
| 리뷰어 | `03-review-agent` | 코드 품질·테스트 커버리지·N+1 쿼리·성능 | 큰 기능 구현 후 코드 검토·리팩토링 |
| 보안 전문가 | `04-security-agent` | 인증·세션·취약점·환경변수·외부 API 키 | 로그인·결제·API 키 관련 작업·출시 직전 점검 |

비유하면 **건물을 짓는 4명의 전문가**예요. 디자이너가 도면을 그리고, 개발자가 골조를 세우고, 리뷰어가 시공 검수를 하고, 보안 전문가가 도어록·CCTV를 점검해요. 본인은 건축주(client)고요.

### 티켓 = 작업 명령서

티켓은 한 명에게 던지는 짧은 작업 명령서예요. 좋은 티켓은 다음 3가지가 명확해요.

1. **무엇을** — 한 줄로 결과물을 묘사 (예: "마이페이지에 즐겨찾기 섹션")
2. **어디에** — 어느 화면·어느 모델인지 (예: "User#mypage 화면")
3. **어떻게** — 눈에 보이는 결과 기준 (예: "카드 그리드 + 각 카드에 X 버튼")

너무 추상적("좋게 만들어줘") 또는 너무 큼("전체 디자인 다시")은 모두 나쁜 티켓이에요. **분해**하세요.

## EXECUTE

### Turn 1: 어떤 기능을 추가하고 싶나요?

> "지금 본인 빌드킷에 ‘이게 있었으면 좋겠다’고 떠오르는 기능이 있나요? 한 줄로 적어 주세요."

> "거창할 필요 없어요. 다음 같은 것도 다 OK예요:"

```
- "사용자가 책을 검색할 수 있게 해 줘"
- "카페 카드에 '오늘 영업 중' 뱃지를 보여 줘"
- "마이페이지에 본인이 저장한 항목 목록을 만들어 줘"
- "회원가입 후 환영 이메일 한 통 보내 줘"
- "랜딩 페이지에 카톡 공유 버튼 하나만"
```

학생이 머뭇거리면:
> "본인 SaaS의 메인 화면에 들어갔다고 상상해 보세요. ‘아, 여기 ○○만 있었으면’이라는 생각, 한 번이라도 든 게 있나요? 그게 첫 신규 티켓 후보예요."

학생이 답하면 → Turn 2로.

만약 학생 답이 너무 큰 덩어리면(예: "추천 시스템 통째로 만들어 줘"):
> "오, 좋은 방향이에요. 그런데 그건 한 티켓으로는 좀 큰 덩어리예요. 본인이 가장 빨리 보고 싶은 ‘첫 단추’ 한 가지로 좁혀 볼까요? 예를 들어 ‘인기 카페 Top 5 목록 화면’ 정도로요."

### Turn 2: 작은 단위로 쪼개기

학생의 아이디어를 받아서 **2~5개 작은 단위**로 함께 분해해요.

> "한 티켓 = 한 가지 결과물이에요. 본인이 적어 준 ‘[학생 답]’을 작은 단위로 쪼개 봅시다. 제가 후보를 보여 줄게요. 본인이 가장 먼저 보고 싶은 1개를 골라 주세요."

예시 — 학생이 "사용자가 책을 검색할 수 있게"라고 했다면:

```
후보 단위:
1. 검색 폼(상단 검색창 + 버튼) — 화면만
2. 검색 API(키워드로 책 데이터 가져오기) — 백엔드
3. 검색 결과 표시 화면 (카드 리스트)
4. 페이지네이션 (한 페이지 10개씩)
```

> "이 4개가 모두 모여야 ‘책 검색 기능’이 완성돼요. 첫 신규 티켓으로 1번을 추천해요. 가장 작고, 결과를 눈으로 즉시 볼 수 있어요. 1번으로 갈까요? 아니면 다른 번호?"

학생이 다른 번호를 고르면 그 번호 기준으로 다음 단계 진행.

학생이 "다 한 번에 하면 안 돼요?"라고 하면:
> "한 번에 다 시키는 것도 가능은 해요. 그런데 결과가 한꺼번에 쏟아지면 본인이 검토하기 어려워요. 한 단위씩 끝내고 다음으로 가는 게 결과 품질이 더 좋아요. 첫 티켓 끝나면 같은 흐름으로 다음 3개도 금방이에요."

---

### Turn 3: 어떤 에이전트가 필요한가요?

> "이 티켓에는 4명의 에이전트 중 누구를 부를까요? 본인이 고르기 전에, 일단 4명을 한 번 더 떠올려 봅시다."

```
👨‍💻 02-rails-dev (Rails 개발자) — 백엔드 + DB + 테스트 코드
🎨 ux-designer (UX·UI 디자이너) — 화면 레이아웃 + 디자인 시스템
🔍 03-review-agent (리뷰어) — 코드 품질 + 성능 점검
🛡 04-security-agent (보안 전문가) — 인증·키·취약점 점검
```

> "본인 티켓이 ‘[학생이 고른 단위]’라면, 가장 먼저 누구를 부르는 게 자연스러울까요?"

학생 답에 따라 가이드:

**학생이 "ux-designer"를 골랐고 티켓이 새 화면이라면:**
> "정확해요. 새 화면은 디자이너부터. 디자이너가 와이어프레임 + 1차 UI를 만들면, 그 위에 02-rails-dev가 백엔드를 붙여요. 한 티켓에 두 명을 ‘순서대로’ 배정할 수 있어요."

**학생이 "02-rails-dev"를 골랐고 티켓이 백엔드/API/모델이라면:**
> "맞아요. 데이터 다루는 일은 개발자부터. 큰 기능이면 끝나고 03-review-agent에게 검토 부탁해요."

**학생이 헷갈려 하면 — 종류별 가이드 표:**

| 본인 티켓 종류 | 먼저 | 다음(선택) |
|---|---|---|
| 새 화면 추가 | ux-designer | 02-rails-dev |
| 버그 수정 | 02-rails-dev | 03-review-agent |
| DB 컬럼 추가 | 02-rails-dev | 03-review-agent |
| 로그인·세션·결제 관련 | 02-rails-dev | 04-security-agent |
| 디자인만 다듬기 | ux-designer | — |
| API 키 추가/관리 | 02-rails-dev | 04-security-agent |

> "본인 티켓에 맞춰 ‘먼저’ + ‘다음’ 한 명씩 골라 주세요."

학생이 골랐으면 → Turn 4로.

### Turn 4: vuild.kr 보드에 실제 티켓 만들기

> "이제 진짜로 티켓을 만들어요. 두 가지 방법이 있어요. 본인이 편한 쪽으로."

**방법 A — Claude Code 한 줄 (가장 빠름):**

> "Antigravity 하단 Claude Code 채팅에 다음을 그대로 붙여 넣어 보세요:"

```
/ticket [본인이 정한 제목]

설명: [한 줄 무엇을 + 어디에 + 어떻게]
우선순위: medium
배정: [먼저 에이전트 슬러그] → [다음 에이전트 슬러그]
```

예시 (즐겨찾기 섹션):

```
/ticket 마이페이지에 즐겨찾기 섹션 추가

설명: 사용자가 카페를 즐겨찾기로 저장할 수 있고, 마이페이지에서 카드 그리드 형태로 목록을 보고, 각 카드의 X 버튼으로 삭제 가능.
우선순위: medium
배정: ux-designer → 02-rails-dev
```

**방법 B — vuild.kr 보드에서 직접:**

> "브라우저에서 본인 보드(`vuild.kr/projects/{본인ID}/build/board`) 열기 → 우상단 ‘새 티켓’ 버튼 → 제목·설명·우선순위·에이전트 입력 → 저장."

> "어느 쪽으로 했든, 만들면 보드에 새 카드가 ‘백로그’ 컬럼에 등장해요. ID(예: T-42)가 부여돼요. 그 ID 기억해 두세요. 다음 단계에서 써요."

학생이 만들었으면 → Turn 5로. 못 만들었으면 어디서 막혔는지 듣고 풀어 주기.

### Turn 5: 진행 + 결과 검토

> "티켓이 만들어졌으면 이제 실행. 다음 명령으로 에이전트가 작업을 시작해요:"

```
/backlog [방금 만든 ID]
```

> "예: `/backlog T-42`. 그러면 본인이 배정한 에이전트(예: ux-designer)가 자동으로 호출돼서, 작업하고, 결과를 review 상태로 올려 줘요. 보통 5~15분 걸려요. 큰 티켓은 더 걸려요."

> "에이전트가 작업하는 동안 본인은 보드에서 진행 상태를 볼 수 있어요. 카드가 ‘백로그 → 진행 중 → 리뷰 대기’ 순으로 이동해요."

> "리뷰 대기에 도착하면 본인이 검토해요. 다음 4가지를 봐요:"

```
✅ 1. 본인 의도대로 동작하나? (브라우저에서 본인 손으로 한 번 써 보기)
✅ 2. 본인이 시킨 ‘무엇/어디/어떻게’가 다 들어 있나?
✅ 3. 다른 화면이 깨지지 않았나?
✅ 4. 코드 변경(diff)을 한 번 훑어 봐도 이상한 데가 없나?
```

> "다 OK면 보드에서 ‘완료(done)’로 옮기거나, Claude Code에 `/review T-42` 한 줄. 그러면 자동으로 done 처리돼요."

> "본인이 직접 만든 첫 신규 티켓이 done이 되는 순간 — 빌드킷이 본인의 것이 됐다는 신호예요."

### Turn 6: 결과가 마음에 안 들면? — 피드백 루프

> "현실적으로 첫 결과가 한 번에 OK인 경우는 드물어요. 자주 있는 상황:"

```
- "디자이너가 너무 많이 바꿨어요" — 본인이 안 시킨 다른 화면도 손댐
- "내가 원한 건 카드 그리드인데 리스트로 만들었어요"
- "버튼은 있는데 클릭해도 동작 안 해요"
```

> "이럴 땐 ‘다시 시키기’예요. 같은 티켓에 코멘트로 피드백을 추가해요. 패턴:"

```
[티켓 ID] 검토 결과: ○○ 부분만 다시 해 줘.
원래 요청: [본인이 시킨 거 한 번 더 명시]
실제 결과와 차이: [구체적으로]
다시 해 줄 것: [무엇을 / 어떻게]
```

예시:

```
T-42 검토 결과: 카드 디자인만 다시 해 줘.
원래 요청: 카드 그리드 + 각 카드에 X 버튼
실제 결과와 차이: 리스트 형태 + 삭제 버튼 안 보임
다시 해 줄 것: 카드 그리드(3열) + 우상단 X 아이콘 버튼 추가
```

> "이걸 Claude Code에 붙이면 같은 에이전트가 그 부분만 다시 작업해 줘요. 처음부터 다시 만들 필요 없어요."

> "그래도 안 되면 다른 에이전트에게 한 번 맡겨 봐요. 디자이너가 자꾸 헷갈리면 02-rails-dev에게 ‘디자인은 그대로 두고 X 버튼 동작만 붙여 줘’."

### Turn 7: 산출물 저장 + 다음 단계

> "이번 스킬에서 본인이 한 일을 한 페이지로 정리해 두세요. 다음 학습 + 회고에 도움이 돼요."

본인이 학생의 답변을 모아 `learn-outputs/ch4-7-new-ticket.md` 파일을 **실제로 생성**한다. 형식:

```markdown
# Ch.4-7 — 첫 신규 티켓

## 티켓 명세
- **제목**: [학생이 정한 제목]
- **ID**: [예: T-42]
- **설명 (무엇/어디/어떻게)**:
  - 무엇을: [한 줄]
  - 어디에: [한 줄]
  - 어떻게: [한 줄]
- **우선순위**: [low / medium / high]
- **배정 에이전트**: [먼저: 슬러그] → [다음: 슬러그]

## 진행 + 검토 노트
- 작업 시간: 약 [학생이 답한 시간]
- 한 번에 통과? [네 / 아니오]
- 다시 시킨 내용 (있으면): [한 줄]
- 최종 결과 한 줄 평: [학생의 한 줄]

## 다음에 만들 티켓 후보 (잊지 않게)
1. [학생이 Turn 2에서 분해한 나머지 단위 중 1]
2. [나머지 단위 중 2]
3. [나머지 단위 중 3]
```

저장 후:

> "축하해요. 본인 빌드킷에 본인이 직접 추가한 첫 티켓이 동작했어요."

> "이제부터 ‘이게 있었으면’이라고 떠오를 때마다 같은 흐름으로 추가하면 돼요. 한 줄 → 분해 → 에이전트 배정 → 실행 → 검토. 5단계."

> "다음 레슨은 4-8 DB + 회원가입 — 구글/카카오 OAuth예요. 본인이 만든 신규 티켓을 동작하려면 회원가입이 필요한 경우가 많아요. 거기로 자연스럽게 이어져요."

> "다음 스킬: `/v3-database` + `/v3-auth` (둘이 한 세트)"

## 자주 막히는 지점

**"에이전트 슬러그를 잘못 적었어요"**
→ Claude가 자동으로 비슷한 이름의 에이전트를 찾아 줘요. 정확히 일치하지 않아도 팀리드(메인 Claude)가 알아서 배정. 너무 걱정 마세요.

**"디자이너가 너무 많이 바꿨어요"**
