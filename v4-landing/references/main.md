## EXPLAIN

### 랜딩 페이지: 서비스의 영화 예고편

랜딩 페이지는 처음 방문한 사람이 "착지(land)"하는 페이지입니다. 카카오톡에서 링크를 클릭하거나, 인스타그램에서 프로필 링크를 누르거나, 네이버에서 검색해서 처음 보는 그 화면입니다.

역할은 하나입니다: **낯선 사람을 고객으로 바꾸기.**

랜딩 페이지를 **영화 예고편**이라고 생각하면 이해가 쉽습니다.

영화 예고편은 2분 안에 해야 할 일이 있습니다:
1. "이게 무슨 영화야?" — 한 눈에 장르와 분위기를 알려줌
2. "왜 봐야 해?" — 궁금증을 유발하는 갈등 보여줌
3. "어떻게 해결돼?" — 해결의 실마리를 살짝 보여줌
4. "진짜 대박이네" — 핵심 볼거리를 나열
5. "지금 예매하러 가자!" — 행동을 유도

랜딩 페이지도 정확히 같은 구조입니다.

### 5섹션 구조: 설득의 심리학

잘 만들어진 랜딩 페이지는 사람의 심리를 따라 설계됩니다:

```
[Hero]     "이게 뭔데?"     → 3초 안에 답한다
   ↓
[Problem]  "맞아, 나도..."  → 고객의 고통을 콕 짚는다
   ↓
[Solution] "이렇게 해결돼"  → 우리가 어떻게 해결하는지
   ↓
[Features] "구체적으로..."  → 핵심 기능 3개를 시각적으로
   ↓
[CTA]      "지금 시작해"    → 하나의 버튼, 하나의 메시지
```

각 섹션을 자세히 봅시다.

### Hero 섹션: 3초의 승부

사용자가 페이지에 도착하면 **3초** 안에 결정합니다. "더 볼까?" 아니면 "뒤로 가기."

Hero 섹션은 스크롤 없이 보이는 첫 화면입니다. 여기서 승부가 납니다.

**Hero 작성 공식:**
> **[누구를 위한] + [어떤 문제를 해결하는] + [서비스]**

좋은 예:
- "바쁜 1인 사업자를 위한 자동 고객 관리 도구"
- "혼자 공부하는 수험생의 학습 계획을 AI가 잡아드립니다"
- "매일 뭘 먹을지 고민하는 직장인을 위한 맞춤 식단 추천"

나쁜 예:
- "최고의 플랫폼에 오신 것을 환영합니다" (뭘 하는 건지 모르겠음)
- "혁신적인 솔루션" (구체적이지 않음)
- "Welcome to our service" (한국어 서비스인데 영어?)

**핵심 포인트**: 전문 용어 없이, 초등학생도 "아, 이런 서비스구나" 알 수 있게.

### Problem 섹션: 공감이 먼저다

기능을 소개하기 전에 **공감**이 먼저입니다.

"우리 서비스 좋아요!"라고 외치기 전에, "당신이 이런 문제 때문에 힘들다는 거 알아요"라고 먼저 다가가야 합니다.

사람은 자기 문제를 정확히 짚어주는 사람을 신뢰합니다. 병원에서 의사가 "여기가 아프시죠?"라고 정확히 짚으면 "이 의사 잘하네" 싶은 것처럼요.

**Problem 작성 공식:**
> "[대상 사용자]는 [이런 상황]에서 [이런 고통]을 겪습니다."

예시:
- "혼자 공부하면 계획을 세우지만, 3일이면 무너집니다"
- "냉장고 재료는 있는데, 뭘 만들어야 할지 매일 고민입니다"
- "고객 문의가 쌓이는데, 하나씩 수동으로 답하느라 본업을 못 합니다"

읽는 사람이 "어, 이거 나 이야기잖아?" 하고 고개를 끄덕이면 성공입니다.

### Solution 섹션: 우리가 해결합니다

Problem에서 공감을 얻었으면, 이제 **해결책**을 보여줍니다.

여기서 중요한 것은 **기능 나열이 아니라 결과**입니다.

나쁜 예: "AI 기반 추천 알고리즘을 탑재했습니다"
좋은 예: "냉장고 사진을 찍으면 10초 안에 레시피가 나옵니다"

사용자는 기술에 관심이 없습니다. **"내 문제가 어떻게 해결되는데?"**에 관심이 있습니다.

### Features 섹션: 핵심 기능 3개

기능은 3~4개만 보여줍니다. 10개를 나열하면 하나도 기억에 남지 않습니다.

각 기능은 이렇게 표현합니다:
- **아이콘 또는 이모지** — 시각적 앵커
- **한 줄 제목** — "뭘 하는지"
- **한 줄 설명** — "왜 좋은지"

카드 형태로 나란히 배치하면 시각적으로 정돈됩니다.

### CTA 섹션: 하나의 버튼, 하나의 메시지

CTA(Call to Action)는 "행동 촉구"입니다. 페이지의 마지막에서 사용자에게 **딱 하나만** 요청합니다.

- "지금 무료로 시작하기"
- "3분 만에 체험해보기"
- "먼저 써보고 결정하세요"

버튼이 2개 이상이면 사용자는 고민합니다. 고민하는 순간 이탈합니다. **하나의 버튼, 하나의 메시지.**

버튼 아래에 안심 문구를 넣으면 전환율이 올라갑니다:
- "신용카드 없이 시작" / "가입 없이 체험" / "30초면 끝"

### C1 서비스 메시지 활용하기

C1에서 정리한 문서들이 여기서 빛을 발합니다:

| C1 문서 | 랜딩 페이지 활용 |
|---------|----------------|
| `my-service-identity.md` 가치 선언문 | → Hero 섹션의 한 줄 설명 |
| `my-service-plan-v0.1.md` 페르소나 | → Hero의 "[누구]를 위한" |
| `my-service-plan-v0.1.md` 문제 정의 | → Problem 섹션 |
| `my-service-identity.md` 핵심 가치 | → Solution 섹션 |
| `my-service-design-v1.0.md` Must 기능 | → Features 섹션 |

C1에서 "왜 만드는가"를 고민한 덕분에 랜딩 페이지 작성이 훨씬 수월합니다. 그때의 고민이 지금 결실을 맺습니다.

## EXECUTE

### 단계 1: Hero 섹션 만들기

먼저 C1에서 만든 서비스 메시지를 활용합니다.

Claude Code에서:
```
my-service-identity.md와 my-service-plan-v0.1.md를 읽고,
랜딩 페이지의 Hero 섹션을 만들어줘.

포함할 것:
- 서비스 한 줄 설명 (가치 선언문 기반, 전문 용어 없이)
- 서브타이틀로 핵심 가치 1문장
- CTA 버튼 ("시작하기" 또는 "무료로 체험하기")
- 버튼 아래 안심 문구 ("가입 없이 바로 체험" 등)
- Tailwind CSS로 깔끔하게
- 배경은 심플하게 (그라데이션 또는 단색)

3초 안에 "이 서비스가 뭔지" 알 수 있게 해줘.
```

> **꿀팁**: Hero 섹션이 마음에 안 들면 "다른 버전으로 써줘"라고 하면 됩니다. 2~3개 버전을 보고 고르세요.

### 단계 2: Problem + Solution 섹션

```
Problem 섹션과 Solution 섹션을 만들어줘.

Problem 섹션:
- 타겟 사용자가 공감할 수 있는 불편함 3가지
- 각 불편함에 아이콘/이모지 + 제목 + 설명
- "맞아, 나 이거 불편했어!" 반응이 나올 수 있게
- my-service-plan-v0.1.md의 문제 정의를 참고해줘

Solution 섹션:
- Problem의 각 불편함에 대응하는 해결책
- "이 서비스를 쓰면 이렇게 달라집니다"
- 기술 설명이 아닌 결과 중심으로
- 시각적으로 Before/After 느낌

Tailwind CSS, 깔끔한 레이아웃으로.
```

### 단계 3: Features + CTA 섹션

```
Features 섹션과 CTA 섹션을 만들어줘.

Features 섹션:
- 핵심 기능 3개를 카드 형태로
- 각 카드: 아이콘 + 제목 + 한 줄 설명
- 모바일에서는 1열, PC에서는 3열
- my-service-design-v1.0.md의 Must 기능 참고

CTA 섹션:
- 강한 행동 촉구 문구
- 큰 버튼 하나 (메인 색상)
- 버튼 아래 안심 문구
- "지금 시작하세요" 같은 짧고 강한 메시지
- 배경색을 살짝 다르게 해서 시선 유도
```

### 단계 4: 전체 흐름 점검

완성된 랜딩 페이지를 처음부터 끝까지 천천히 스크롤합니다.

**처음 보는 사람의 입장에서** 자문합니다:
- Hero: "이 서비스가 뭔지 3초 안에 알겠어?" ✅/❌
- Problem: "맞아, 나도 이게 불편해" 느낌이 드는가? ✅/❌
- Solution: "오, 이렇게 해결되는구나" 이해가 되는가? ✅/❌
- Features: "구체적으로 이런 게 되는구나" 그려지는가? ✅/❌
- CTA: "한번 써볼까?" 마음이 드는가? ✅/❌

❌인 부분이 있으면 구체적으로 피드백합니다:
```
Problem 섹션에서 공감이 잘 안 돼. 
[이런 상황]을 겪는 사람 입장에서 더 와닿게 써줘.
```

### 단계 5: 모바일 반응형 확인

브라우저에서 개발자 도구를 엽니다 (F12):
- 모바일 뷰 전환 (Ctrl+Shift+M)
- 위에서 아래로 천천히 스크롤

확인할 것:
- Hero 문구가 잘 보이는지 (너무 작지 않은지)
- Problem/Solution 카드가 세로로 쌓이는지
- Features 카드가 1열로 정렬되는지
- CTA 버튼이 충분히 크고 누르기 쉬운지
- 좌우 스크롤이 생기지 않는지

문제가 있으면:
```
모바일에서 [어떤 부분]이 [어떻게] 깨져 보여. 반응형으로 수정해줘.
```

### 단계 6: 최종 다듬기

마지막으로 전체적인 인상을 다듬습니다:

```
랜딩 페이지 전체를 최종 점검해줘.
- 섹션 간 전환이 자연스러운지
- 색상 테마가 일관된지
- 여백이 충분한지
- 글이 너무 길지 않은지 (핵심만 남기기)
- 모바일에서도 깔끔한지
```

> **막혔을 때**: "잘 모르겠는데 전문적으로 보이게 해줘"라고만 해도 Claude가 개선해줍니다. 구체적 피드백이 더 좋지만, 막연해도 괜찮습니다.
