## EXPLAIN

### 완벽은 완료의 적이다

서비스를 만들다 보면 이런 생각이 듭니다:
- "이 기능만 더 넣으면..."
- "디자인이 아직 좀 아쉬운데..."
- "버그가 하나 더 있는 것 같아서..."
- "좀 더 다듬고 나서..."

이러다가 오픈을 **영원히** 미루는 분들이 정말 많습니다.

하지만 진실은 이렇습니다: **실제로 써보지 않으면 무엇이 진짜 중요한지 알 수 없습니다.**

페이스북 초창기 슬로건: **"Done is better than perfect"** (완료가 완벽보다 낫다)

인스타그램도 처음에는 기능이 13개밖에 없었습니다. 카카오톡도 처음에는 문자 기능밖에 없었습니다. 모두 **먼저 내놓고, 사용자 반응을 보며 개선**했습니다.

첫 고객 10명이 "이게 불편해요"라고 하면 그때 고치면 됩니다. 혼자 상상하는 "완벽"보다 실제 사용자의 피드백이 100배 가치 있습니다.

### 오픈 전 반드시 확인할 5가지

무한히 준비할 수 없으니, **이 5가지만** 확인하면 오픈해도 됩니다:

| # | 항목 | 확인 방법 |
|---|------|----------|
| 1 | **핵심 기능이 작동한다** | 주요 시나리오를 처음부터 끝까지 한 번 더 테스트 |
| 2 | **결제가 된다** (해당시) | 테스트 모드에서 성공/실패 모두 확인 |
| 3 | **개인정보 처리방침** | 법적 의무. Claude에게 작성 요청 |
| 4 | **연락처가 있다** | 사용자가 문제를 알려줄 방법 (이메일, 폼) |
| 5 | **보안 체크리스트 통과** | v4-security에서 10개 항목 완료 |

**나중에 해도 되는 것**:
- 완벽한 디자인 → 첫 사용자 피드백 후 개선
- 모든 기능 완성 → 핵심만 있으면 충분
- SEO 최적화 → 첫 사용자는 직접 공유로 모음
- 다국어 지원 → 한국어 사용자부터

### 공유 최적화: 카카오톡에서 예쁘게 보이기

한국에서 서비스를 공유하는 첫 번째 경로는 **카카오톡**입니다.

카카오톡이나 SNS에 링크를 보내면 **미리보기**가 나옵니다. 제목, 설명, 이미지가 깔끔하게 나오면 클릭률이 2~3배 올라갑니다. 빈 박스만 나오면 "뭐야 이거" 하고 무시합니다.

이 미리보기를 결정하는 것이 **Meta Tags**와 **OG(Open Graph) Image**입니다.

```html
<!-- 이 코드가 카카오톡 미리보기를 결정합니다 -->
<meta property="og:title" content="서비스 이름 - 한 줄 설명" />
<meta property="og:description" content="누구를 위한 어떤 서비스인지 60자 이내" />
<meta property="og:image" content="https://내서비스.com/og-image.png" />
<meta property="og:url" content="https://내서비스.com" />
```

| 태그 | 역할 | 예시 |
|------|------|------|
| **og:title** | 카카오톡에서 보이는 제목 | "북메모리 - AI 독서 노트" |
| **og:description** | 제목 아래 설명 | "책을 읽고 핵심만 기억하세요" |
| **og:image** | 미리보기 이미지 | 서비스 스크린샷 또는 로고 |
| **og:url** | 링크 주소 | https://bookmemory.kr |

**OG 이미지 권장 사이즈**: 1200 x 630 픽셀 (가로가 넓은 직사각형)

이걸 설정하지 않으면 카카오톡에서 공유했을 때 빈 박스만 나옵니다. 설정하면 제목 + 설명 + 이미지가 깔끔하게 나와서 "오, 이거 뭐야?" 하고 클릭하게 됩니다.

### favicon: 브라우저 탭의 작은 아이콘

브라우저 탭에 보이는 작은 아이콘이 favicon입니다. 기본 Rails 아이콘 대신 내 서비스 로고를 넣으면 "전문 서비스" 느낌이 납니다.

### 모바일 최종 점검

한국 웹 트래픽의 60% 이상이 모바일입니다. 카카오톡으로 공유하면 받는 사람도 **폰에서 클릭**합니다.

모바일에서 확인할 것:
1. 페이지가 깨지지 않고 잘 보이는지
2. 버튼이 손가락으로 누를 수 있는 크기인지
3. 입력창에 텍스트 입력이 되는지
4. 가로 스크롤이 생기지 않는지
5. 핵심 기능이 모바일에서도 동작하는지

### LAUNCH의 진짜 의미

오픈은 끝이 아닙니다. **새로운 시작**입니다.

```
오픈 → 실제 사용자 → 피드백 → 개선 → 더 많은 사용자 → 더 많은 피드백 → ...
```

이 사이클을 시작하는 것이 LAUNCH입니다. 오늘 이후로 여러분의 서비스는 **살아있는 것**이 됩니다. 사용자가 쓰고, 피드백을 주고, 그에 따라 진화합니다.

## EXECUTE

### 단계 1: Meta Tags + OG Image 설정

Claude Code에서:
```
내 서비스의 Meta Tags와 OG Image를 설정해줘.

설정할 것:
- og:title — 서비스 이름 + 한 줄 설명
- og:description — 서비스가 뭘 하는지 60자 이내
- og:image — 서비스 대표 이미지 URL
  (이미지가 없으면 간단한 OG 이미지를 HTML로 만들거나,
   서비스 스크린샷을 public 폴더에 넣어줘)
- og:url — 서비스 배포 URL
- og:type — website
- og:locale — ko_KR
- favicon 설정
- 모바일 viewport 메타 태그 확인
- Twitter Card 메타 태그도 추가

카카오톡과 인스타그램에서 공유했을 때 미리보기가 깔끔하게 나오게 해줘.
```

설정 후 확인하는 방법:
1. https://developers.kakao.com/tool/debugger/sharing 에서 URL 입력
2. 미리보기가 어떻게 나오는지 확인
3. 마음에 안 들면 수정 후 "캐시 초기화" 클릭

> **꿀팁**: 카카오 디버거에서 한 번 확인하면 실제 카카오톡 공유 시 어떻게 보이는지 정확히 알 수 있습니다. 이 단계를 건너뛰면 "어, 미리보기가 이상하게 나와" 하고 후회하게 됩니다.

### 단계 2: 개인정보 처리방침 추가

사용자 정보를 수집하는 서비스라면 **법적으로** 개인정보 처리방침이 필요합니다.

```
내 서비스의 개인정보 처리방침 페이지를 만들어줘.

포함할 것:
- 수집하는 개인정보 항목 (이메일, 이름 등 실제 수집하는 것만)
- 수집 목적
- 보관 기간
- 제3자 제공 여부
- 개인정보 관리 책임자 연락처
- 한국 개인정보보호법 기준

푸터에 "개인정보 처리방침" 링크를 추가해줘.
```

### 단계 3: 연락처 / 문의 방법 추가

사용자가 문제를 알려줄 방법이 있어야 합니다.

```
서비스에 문의/피드백 받을 수 있는 방법을 추가해줘.

방법 중 택 1:
- 푸터에 이메일 주소 표시
- 간단한 문의 폼 페이지
- 카카오톡 오픈채팅 링크

가장 간단한 방법으로.
```

### 단계 4: 모바일 최종 점검

실제 폰에서 배포된 URL에 접속합니다:

1. 폰의 브라우저에서 배포 URL 입력
2. 랜딩 페이지부터 시작 → 천천히 스크롤
3. 가입 → 로그인 → 핵심 기능 → 결제(있다면) 순서로 테스트
4. 글자 크기, 버튼 크기, 레이아웃 확인

문제가 있으면:
```
모바일에서 [문제 설명]. 
실제 폰에서 테스트했을 때 발견한 문제야. 수정해줘.
```

### 단계 5: 런치 카운트다운

모든 준비가 끝났습니다.

**5가지 체크리스트 최종 확인:**
- [ ] 핵심 기능이 작동한다
- [ ] 보안 체크리스트를 통과했다
- [ ] Meta Tags가 설정되었다
- [ ] 모바일에서 잘 보인다
- [ ] 연락처/문의 방법이 있다

모두 체크되었나요?

잠시 멈추고 이 순간을 느껴봅니다.

당신은:
- 아이디어를 생각했고 (C1)
- 기획서를 만들었고 (C2)
- 코드를 작성했고 (C3)
- 디자인하고, 보안 점검하고, 배포했습니다 (C4)

그리고 지금, **세상에 공유하려 합니다.**

```
 ___________________________
|                           |
|    L A U N C H            |
|    C O U N T D O W N      |
|                           |
|    3... 서비스가 작동합니다  |
|    2... 보안 점검 완료      |
|    1... 세상이 기다립니다    |
|                           |
|    🚀  L A U N C H !      |
|___________________________|
```

### 단계 6: 첫 공유 메시지 작성

이제 세상에 알릴 차례입니다.

**카카오톡용 메시지:**
```
내 서비스를 처음 공유할 카카오톡 메시지를 써줘.

포함할 것:
- 서비스 이름과 한 줄 설명
- 누구를 위한 서비스인지
- 서비스 URL
- "한번 써보고 솔직한 피드백 주면 정말 고마울 것 같아" 느낌
- 너무 광고 같지 않게, 친구한테 보내는 느낌으로
- 짧게 (카카오톡이니까 3~4줄)
```

**인스타그램/SNS용 메시지:**
```
내 서비스를 인스타그램이나 SNS에 올릴 게시물을 써줘.

포함할 것:
- 서비스 스토리 (왜 만들게 되었는지)
- 어떤 문제를 해결하는지
- 서비스 URL
- 피드백 요청
- 해시태그 5~10개
- 진정성 있는 톤
```

### 단계 7: 런치 발표문 작성

좀 더 공식적인 발표문도 만들어둡니다:

```
내 서비스의 런치 발표문을 써줘.

용도: 블로그, 커뮤니티 (디스콰이엇, 프로덕트 헌트, 관련 카페)

구조:
1. 왜 이 서비스를 만들었는지 (배경 스토리)
2. 어떤 문제를 해결하는지
3. 핵심 기능 3가지
4. 서비스 URL
5. 앞으로의 계획
6. 피드백 요청

길이: 읽는 데 2분 이내
톤: 진정성 있고, 겸손하지만 자신감 있게
```

### 단계 8: 첫 공유

메시지를 복사해서 **지금 바로** 공유합니다.

먼저 보낼 대상:
1. **가족** — 무조건 응원해줄 사람
2. **친한 친구 2~3명** — 솔직한 피드백을 줄 사람
3. **관련 커뮤니티** — 실제 타겟 사용자가 있는 곳

> **꿀팁**: 첫 공유가 가장 떨립니다. "별거 아닌데..." 하는 생각이 들 수 있습니다. 하지만 **아이디어에서 실제 서비스를 만들어서 배포한 것**은 정말 대단한 일입니다. 자신감을 가지세요.

> **막혔을 때**: "아직 부족해서..." 하고 공유를 미루고 싶다면, 딱 1명에게만 보내세요. 가장 가까운 사람 1명. 그 1명의 반응이 "계속 가도 돼"라는 신호가 됩니다.
