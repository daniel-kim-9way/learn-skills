## EXPLAIN

### 내 사용자를 보호하는 방패

보안이라고 하면 "해킹", "유출", "사이버 공격" 같은 무서운 단어가 떠오릅니다.

관점을 바꿔봅시다.

보안은 **내가 만든 서비스를 믿고 가입한 사용자들을 보호하는 방패**입니다. 이름, 이메일, 비밀번호를 맡긴 사람들의 **신뢰를 지키는 것**입니다. 이것은 기술 문제가 아니라 **책임**입니다.

그리고 좋은 소식이 있습니다.

> **Rails 같은 현대 프레임워크는 대부분의 기본 보안을 자동으로 처리합니다.**

전문 해커를 막으려면 전문가가 필요합니다. 하지만 흔한 공격의 90%는 **기본만 지키면** 막을 수 있습니다. 그 "기본"을 오늘 함께 점검합니다.

여러분이 해야 할 일은 두 가지입니다:
1. **실수하지 않는 것** (API 키를 소스 코드에 안 넣기 등)
2. **기본 설정을 올바르게 유지하는 것** (Rails가 해주는 보호를 끄지 않기)

### 보안 위협 5가지: 일상 비유로 이해하기

전문 용어가 무섭게 들리지만, 일상에서 벌어지는 상황과 같습니다.

**1. 환경 변수 노출 — "열쇠를 문 앞에 두기"**

집 열쇠를 현관 매트 밑에 두면 누구든 들어올 수 있습니다. API 키나 비밀번호를 소스 코드에 직접 넣으면, GitHub에 올릴 때 전 세계에 공개됩니다.

실제 사례: 한 개발자가 AWS 키를 GitHub에 실수로 올렸더니 3시간 만에 봇이 발견해서 수백만 원어치의 서버를 임의로 생성했습니다.

**방어**: `.env` 파일에 보관하고, `.gitignore`에 추가

**2. 인증 우회 — "자물쇠 없는 문"**

관리자 페이지에 로그인 체크를 깜빡하면, URL을 아는 누구나 접근할 수 있습니다. 마치 귀중품 보관소 문에 자물쇠를 안 건 것과 같습니다.

실수 예시: `/admin` 페이지를 만들었는데 `before_action :authenticate_user!`를 빠뜨린 경우

**방어**: 모든 비공개 페이지에 로그인 확인 코드 추가

**3. XSS (Cross-Site Scripting) — "댓글에 악성 코드 심기"**

사용자가 댓글에 일반 텍스트가 아닌 JavaScript 코드를 넣으면, 다른 사용자의 브라우저에서 그 코드가 실행될 수 있습니다. 로그인 정보를 훔치거나, 악성 사이트로 이동시킬 수 있습니다.

일상 비유: 학교 게시판에 "이 쪽지를 읽으면 비밀번호를 말해야 한다"라고 적어놓은 것

**방어**: Rails는 기본적으로 사용자 입력을 "이스케이프" 처리합니다. `<script>` 태그가 코드가 아닌 일반 텍스트로 보이게 됩니다. 이 기본 설정을 건드리지 않으면 안전합니다.

**4. SQL 인젝션 — "빈칸에 악의적인 명령어 넣기"**

로그인 입력란에 아이디 대신 SQL 명령어를 넣어서 데이터베이스를 조작하는 공격입니다. "아이디: admin' OR 1=1 --" 같은 걸 넣으면 모든 계정으로 로그인될 수 있습니다.

일상 비유: 택배 수취인란에 "모든 택배를 나에게 보내라"라고 적는 것

**방어**: Rails의 ActiveRecord가 자동으로 막아줍니다. 다만 `User.where("name = '#{params[:name]}'")` 같이 직접 SQL을 쓰면 위험합니다. `User.where(name: params[:name])`으로 쓰면 안전합니다.

**5. CSRF (사이트 간 요청 위조) — "위조 편지"**

다른 사이트에서 사용자 몰래 우리 서비스에 요청을 보내는 공격입니다. 사용자가 악성 사이트를 방문했을 때, 그 사이트가 우리 서비스에 "비밀번호 변경" 요청을 대신 보내는 식입니다.

일상 비유: 내 이름을 도용해서 은행에 위조 편지를 보내는 것

**방어**: Rails가 자동으로 CSRF 토큰으로 방어합니다. 모든 폼에 "이건 진짜 요청이다"라는 비밀 도장을 찍어서 위조를 구분합니다.

### 10개 보안 체크리스트

서비스를 배포하기 전에 **반드시** 확인할 10가지입니다:

| # | 항목 | 왜 중요한가 | Rails 자동? |
|---|------|-----------|-----------|
| 1 | .env 파일이 .gitignore에 있는가 | API 키, 비밀번호 유출 방지 | 아니오 |
| 2 | 인증 보호: 로그인 필요한 페이지가 보호되는가 | 비인가 접근 차단 | 아니오 |
| 3 | HTTPS 강제 (production) | 데이터 도청 방지 | 설정 필요 |
| 4 | 관리자 페이지 접근 제한 | 관리 기능 보호 | 아니오 |
| 5 | Strong Parameters 설정 | 허용하지 않은 데이터 변조 방지 | 수동 |
| 6 | XSS 방어 (이스케이프 처리) | 악성 스크립트 실행 방지 | 자동 |
| 7 | CSRF 방어 (토큰 확인) | 위조 요청 차단 | 자동 |
| 8 | Raw SQL 미사용 | SQL 인젝션 방지 | 자동* |
| 9 | 비밀번호 암호화 | 데이터 유출 시 피해 최소화 | 자동* |
| 10 | Production 오류 메시지 숨김 | 시스템 정보 노출 방지 | 자동 |

자동*: 올바른 방법을 사용했을 때만 자동. 직접 SQL을 쓰거나, 비밀번호를 직접 저장하면 보호가 안 됩니다.

**이 체크리스트를 통과하면 대부분의 흔한 공격을 막을 수 있습니다.** 100% 완벽한 보안은 없지만, 이 기본만 지켜도 "쉬운 먹잇감"에서 벗어납니다. 대부분의 공격자는 쉬운 대상부터 노립니다.

### Claude가 자동으로 검사하는 방법

일일이 코드를 뒤질 필요 없습니다. Claude에게 "보안 체크리스트를 실행해줘"라고 하면 전체 코드를 스캔하고 결과를 알려줍니다.

각 항목에 대해:
- ✅ **통과** — 이 항목은 안전합니다
- ⚠️ **주의** — 위험하지는 않지만 개선하면 좋습니다
- ❌ **즉시 수정** — 지금 바로 고쳐야 합니다

❌ 항목이 있으면 Claude에게 "이거 수정해줘"라고 하면 바로 고쳐줍니다.

> **중요**: 이 보안 점검은 **배포 전에 반드시** 해야 합니다. "나중에 하지" 하면 보안 사고 후에 후회하게 됩니다. 자물쇠를 안 잠그고 여행을 가는 것과 같습니다.

## EXECUTE

### 단계 1: 10개 보안 체크리스트 자동 검사

Claude Code에서 아래 프롬프트를 실행합니다:

```
내 서비스의 보안 체크리스트를 실행해줘.
아래 10가지 항목을 하나씩 확인하고 결과를 알려줘:

1. 환경 변수: .env 파일이 .gitignore에 있는지
2. 인증 보호: 로그인 없이 접근하면 안 되는 페이지가 보호되는지
3. HTTPS: Production에서 HTTPS가 강제 적용되는지 (config/environments/production.rb)
4. 관리자 페이지: /admin이 있다면 인증으로 보호되는지
5. 파라미터 검증: Strong Parameters가 올바르게 설정됐는지 (permit에 불필요한 항목 없는지)
6. XSS 방어: 사용자 입력을 HTML에 출력할 때 이스케이프 처리되는지 (raw/html_safe 사용 여부)
7. CSRF 방어: ApplicationController에 protect_from_forgery가 있는지
8. SQL 인젝션: Raw SQL 쿼리를 직접 쓰는 곳이 없는지
9. 비밀번호 암호화: 비밀번호가 평문 저장되지 않는지 (has_secure_password 또는 Devise)
10. 에러 메시지: Production에서 상세 오류가 노출되지 않는지

각 항목: ✅ 통과 / ⚠️ 주의 / ❌ 즉시 수정 필요
문제가 있으면 구체적으로 어디가 문제인지, 어떻게 고치면 되는지 알려줘.
```

### 단계 2: 문제 항목 즉시 수정

❌ 항목이 있으면 **즉시** 수정합니다:

```
[항목 번호]번 항목이 ❌야. 지금 바로 수정해줘.
수정 전후를 비교해서 보여줘.
```

⚠️ 항목도 가능하면 수정합니다:

```
[항목 번호]번 항목이 ⚠️야. 개선해줘.
```

**모든 항목이 ✅가 될 때까지 반복합니다.**

> **에러가 나면**: 보안 수정 중 기존 기능이 깨질 수 있습니다. 수정 후에 기본 기능(로그인, 핵심 기능)이 잘 동작하는지 확인하세요. 문제가 있으면 "보안 수정 후 [이 기능]이 안 돼. 확인해줘"라고 하면 됩니다.

### 단계 3: 추가 보안 강화 (선택)

기본 10개를 통과했으면, 한 단계 더 강화할 수 있습니다:

```
추가 보안도 점검해줘:
- Content Security Policy (CSP) 헤더 설정
- Referrer-Policy 헤더 설정
- 로그인 실패 횟수 제한 (brute force 방어)
- 세션 타임아웃 설정
```

이건 선택사항입니다. 기본 10개만 통과해도 충분합니다.

### 단계 4: 보안 확인서 생성

모든 항목 통과 후:

```
보안 체크리스트 결과를 요약해줘.
10개 항목 모두 통과했으면 "보안 점검 완료" 확인서를 만들어줘.

포함할 것:
- 점검 날짜
- 점검 항목 10개와 결과
- "이 서비스는 기본 보안 점검을 완료했습니다"
```

이 확인서는 대단한 문서가 아닙니다. **"나는 기본 보안을 점검하고 서비스를 오픈한다"**는 자기 자신과의 약속입니다.

대부분의 보안 사고는 "나중에 하지" 하다가 일어납니다. 당신은 오늘 그 "나중에"를 앞당겼습니다.

> **꿀팁**: 보안 점검은 배포할 때마다 하는 것이 좋습니다. 새 기능을 추가하면 보안 구멍이 생길 수 있습니다. Claude에게 "보안 체크리스트 다시 돌려줘"라고 하면 30초면 끝납니다.
