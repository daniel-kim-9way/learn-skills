## EXPLAIN

### 엘리베이터 피치: 30초 안에 마음을 움직이기

**엘리베이터 피치**란, 엘리베이터를 타는 30초 동안 우연히 만난 투자자에게 서비스를 설명하는 능력입니다.

왜 30초인가? 사람의 집중력은 짧습니다. 30초 안에 "이거 재밌네"라고 느끼지 못하면 관심은 사라집니다. 이메일 소개도, 네트워킹 행사도, 카카오톡 공유 메시지도 — 핵심은 같습니다. **짧은 시간 안에 "더 알고 싶다"는 마음을 만드는 것.**

### 30초 피치 구조

```
"[공감 질문] 혹시 [이런 경험] 있으신가요?

 [해결책] 저는 [이런 사람들]을 위해 [서비스 이름]을 만들었습니다.

 [핵심 기능] [한 가지 기능]으로 [결과/혜택]을 드립니다.

 [행동 촉구] 한번 써보시겠어요?"
```

**좋은 피치 예시:**
> "혹시 책을 읽고 나서 내용을 금방 잊어버리는 경험 있으신가요? 저는 독서광들을 위해 '북메모리'를 만들었습니다. AI가 핵심 내용을 정리하고 복습 알림을 보내서, 읽은 책을 오래 기억할 수 있게 도와드립니다. 한번 써보시겠어요?"

**나쁜 피치 예시:**
> "저는 AI 기반 독서 관리 플랫폼을 만들었습니다. NLP 기술을 활용한 텍스트 요약 기능과 스페이스드 리피티션 알고리즘을 적용한..."

차이가 느껴지시나요? 첫 번째는 **감정**에 호소하고, 두 번째는 **기술**을 나열합니다.

### 스토리가 데이터를 이긴다

같은 서비스를 두 가지로 소개할 때:

**데이터형**: "독서 앱, AI 요약 기능 포함, 월 활성 사용자 100명"
**스토리형**: "책을 읽고 나서 90%를 잊어버리는 고통을 해결합니다"

같은 서비스인데 **스토리가 훨씬 강력합니다.** 듣는 사람이 자기 경험을 투영하기 때문입니다.

"나도 책 읽고 다 잊어버리는데..." → "이거 써보고 싶다"

이 연결이 자연스럽게 일어납니다.

피치를 만들 때 기억할 것:
1. **공감으로 시작** — "이런 경험 있으세요?"
2. **문제를 짚기** — 기능이 아니라 해결하는 고통
3. **결과를 말하기** — 기술이 아니라 혜택
4. **쉬운 말로** — 전문 용어 금지

### Showcase에서의 첫인상

Vuild Showcase는 서비스 빌더들이 만든 서비스를 공유하는 공간입니다. 다른 빌더, 잠재 사용자, 멘토, 투자자들이 둘러봅니다.

서비스 목록에서 클릭을 받으려면 **제목과 한 줄 설명**이 전부입니다.

사람들은 5~10개를 훑어보고 흥미로운 것만 클릭합니다. 눈길을 잡는 시간은 **2초**입니다.

| | 나쁜 예 | 좋은 예 |
|--|--------|--------|
| 제목 | 독서 관리 앱 | 북메모리 |
| 한 줄 설명 | 독서 관리를 도와주는 앱입니다 | 책 읽고 90%를 잊어버리는 분들을 위한 AI 독서 노트 |

좋은 한 줄 설명의 공식:
> **[대상]을 위한 [핵심 가치/해결하는 문제]**

이 공식으로 쓰면 "누구를 위한 건지"와 "왜 필요한지"가 2초 안에 전달됩니다.

### C1부터 C4까지: 당신의 여정

잠시 멈추고 여기까지 온 길을 돌아봅시다.

**C1 — 발견과 정의**
아이디어를 발견했습니다. "이런 게 있으면 좋겠다"는 생각을 "이 사람들이 이 문제 때문에 힘들다"로 바꿨습니다. 서비스 이름을 짓고, 가치 선언문을 만들었습니다.

**C2 — 설계와 기획**
막연한 아이디어를 구체적인 기획서로 만들었습니다. 페르소나를 설정하고, 기능을 정의하고, 화면 흐름을 설계했습니다. "이렇게 만들면 되겠다"는 확신이 생겼습니다.

**C3 — 구현**
Claude Code를 설치하고, Rails 프로젝트를 만들고, 실제로 동작하는 서비스를 만들었습니다. 처음으로 `localhost:3000`에서 내 서비스가 돌아가는 것을 봤습니다.

**C4 — LAUNCH**
디자인을 입히고, 랜딩 페이지를 만들고, 보안을 점검하고, 인터넷에 배포했습니다. 그리고 세상에 공유했습니다.

몇 주 전만 해도 **"코딩을 모르는 나"**였습니다.

지금은 **실제 서비스를 만들어서 인터넷에 올리고, URL을 공유하는 사람**입니다.

이것은 작은 변화가 아닙니다. **패러다임의 전환**입니다.

"아이디어가 있는데 개발자가 없어서..." 라는 말을 더 이상 하지 않아도 됩니다. 이제는 "아이디어? 만들면 되지"입니다.

### C5~C8 독립 트랙

C1~C4는 **필수 여정**이었습니다. 아이디어 → 기획 → 구현 → 런치. 이 순서를 따라야 했습니다.

여기서부터는 다릅니다. **독립 트랙**입니다. 관심 있는 것을 골라서, 순서 상관없이, 필요할 때 학습합니다.

| 트랙 | 핵심 | 이런 분에게 |
|------|------|-----------|
| **C5: 성장** | 사용자 분석, A/B 테스트, 마케팅 기초 | "사용자를 더 모으고 싶다" |
| **C6: 수익화** | 구독 모델, 프리미엄 기능, 가격 전략 | "이걸로 돈을 벌고 싶다" |
| **C7: 팀빌딩** | 협업 도구, 코드 리뷰, 프로젝트 관리 | "팀을 만들어 함께 하고 싶다" |
| **C8: 스케일링** | 성능 최적화, 인프라 확장, 모니터링 | "사용자가 많아져서 서버가 느리다" |

지금 당장 안 해도 됩니다. 서비스를 운영하다가 필요해지면 돌아오세요.

> "사용자가 안 늘어" → C5
> "이걸로 수입을 만들고 싶어" → C6
> "혼자 하기 힘들어" → C7
> "사용자가 많아져서 느려" → C8

## EXECUTE

### 단계 1: 엘리베이터 피치 작성

Claude에게:
```
내 서비스의 30초 엘리베이터 피치를 써줘.

구조:
1. 공감 질문으로 시작 ("혹시 ~하신 경험 있으세요?")
2. 서비스 이름 + 대상 소개
3. 핵심 기능 한 가지 + 결과/혜택
4. 행동 촉구 ("한번 써보시겠어요?")

주의:
- 전문 용어 금지 (API, AI, 알고리즘 같은 단어 쓰지 마)
- 기술 설명이 아니라 해결하는 고통 중심으로
- 자연스럽게 말할 수 있는 구어체로
- 3가지 버전을 만들어줘
```

3가지 중 가장 마음에 드는 것을 골라 **소리 내어 읽어봅니다.**

시간을 재봅니다. 30초 안에 끝나나요?

너무 길면:
```
이 피치가 30초가 넘어. 더 짧게 줄여줘.
핵심만 남기고 나머지는 빼줘.
```

> **꿀팁**: 피치를 외울 필요는 없습니다. 핵심 구조만 기억하세요: 공감 → 소개 → 혜택 → 촉구. 자연스럽게 말할 수 있을 때까지 2~3번 읽어보면 됩니다.

### 단계 2: Vuild Showcase 등록 준비

등록 전에 필요한 것을 준비합니다.

**한 줄 설명 작성:**
```
내 서비스의 Showcase 한 줄 설명을 3가지 버전으로 써줘.

공식: "[대상]을 위한 [핵심 가치/해결하는 문제]"

조건:
- 60자 이내
- 전문 용어 없이
- "~하는 분들을 위한" 형태
- 클릭하고 싶게
```

**대표 이미지 준비:**
서비스의 가장 매력적인 화면을 스크린샷으로 찍습니다.
- 랜딩 페이지 Hero 섹션 또는
- 핵심 기능이 동작하는 화면

### 단계 3: Vuild Showcase 등록

1. https://vuild.kr/showcase 접속
2. "내 서비스 등록" 클릭
3. 정보 입력:
   - **서비스 이름**: 내 서비스 이름
   - **한 줄 설명**: 위에서 고른 버전 (60자 이내)
   - **서비스 URL**: 배포된 URL
   - **대표 이미지**: 준비한 스크린샷
   - **카테고리**: 가장 적합한 것 선택
   - **만든 사람**: 이름 또는 닉네임
4. "등록" 클릭

**등록 완료!** 이제 다른 빌더들이 여러분의 서비스를 볼 수 있습니다.

### 단계 4: 여정 회고

Claude와 함께 여정을 돌아봅니다:
```
우리가 이 서비스를 만들면서의 여정을 정리해줘.

포함할 것:
- C1에서 처음 쓴 아이디어와 지금 서비스의 비교
- 만들면서 가장 어려웠던 순간들
- 가장 뿌듯했던 순간들
- 처음 계획과 달라진 점
- 앞으로 개선하고 싶은 것 3가지

감성적이면서도 구체적으로 써줘.
```

이 회고는 여러분만의 **성장 기록**입니다. 나중에 다시 읽으면 "내가 이때 이랬구나" 하는 소중한 자산이 됩니다.

### 단계 5: "당신은 서비스 빌더입니다"

아이디어에서 시작해서 실제 서비스를 만들고, 배포하고, 세상에 공유했습니다.

---

**당신은 서비스 빌더입니다.**

---

이 문장의 무게를 느껴보세요.

"나는 코딩을 모르는 사람"이 아닙니다.
"나는 아이디어를 실제 서비스로 만들 수 있는 사람"입니다.

이제 코딩은 낯선 것이 아닙니다. Claude와 함께 **무엇이든** 만들 수 있습니다.

다음 아이디어가 떠오르면? **또 만들면 됩니다.** 이번에는 훨씬 빠르게.

C1에서 처음 쓴 문장을 기억하시나요? "이런 서비스가 있으면 좋겠다"고 적었던 그 순간을. 그 문장이 지금 인터넷에 살아있는 서비스가 되었습니다.

그것이 **당신이 해낸 일**입니다.

---

### 단계 6: C5~C8 독립 트랙 살펴보기

지금 당장 시작할 필요는 없습니다. 서비스를 운영하면서 필요를 느낄 때 돌아오세요.

하지만 미리 맛보기를 원한다면:

```
내 서비스의 현재 상태를 보고,
C5(성장), C6(수익화), C7(팀빌딩), C8(스케일링) 중
가장 먼저 필요할 것 같은 트랙과 이유를 알려줘.
```

이 답변이 앞으로의 방향을 잡는 데 도움이 될 겁니다.
