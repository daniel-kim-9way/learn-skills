## EXPLAIN

### 숫자의 두 얼굴: 허영 지표 vs 실행 지표

서비스를 운영하면 숫자가 쏟아집니다. 방문자 수, 페이지뷰, 가입자 수... 숫자가 올라가면 기분이 좋습니다. 하지만 여기 함정이 있습니다.

**모든 숫자가 의사결정에 도움이 되는 건 아닙니다.**

숫자에는 두 종류가 있습니다:

**허영 지표 (Vanity Metrics)** — 보기엔 좋지만, "그래서 뭘 해야 해?"에 답을 못 주는 숫자

| 허영 지표 | 왜 위험한가 |
|-----------|-------------|
| 총 누적 가입자 수 | 탈퇴한 사람, 안 쓰는 사람 포함. 숫자만 올라감 |
| 총 페이지뷰 | 한 사람이 100번 방문해도 100. 실제 사용자 수 모름 |
| SNS 좋아요 수 | 좋아요 누른 사람이 서비스를 쓰지는 않음 |
| 앱 다운로드 수 | 깔기만 하고 안 열어보는 사람 대부분 |

**실행 지표 (Actionable Metrics)** — "이걸 보고 무엇을 할지" 판단할 수 있는 숫자

| 실행 지표 | 왜 중요한가 |
|-----------|-------------|
| DAU (일간 활성 사용자) | 매일 실제로 서비스를 쓰는 사람 |
| 핵심 기능 사용률 | 메인 기능을 쓰는 비율 = 서비스 가치 증명 |
| 전환율 | 방문 → 가입, 가입 → 사용 각 단계 전환 |
| 이탈률 | 다시 안 오는 사람 비율 = 문제 신호 |

**예시로 비교:**
- "우리 서비스 가입자 500명이에요!" → 좋아 보이지만, 이 중 지금도 쓰는 사람이 10명이면?
- "우리 서비스 DAU 30명이에요!" → 매일 30명이 쓴다면, 진짜 가치를 제공하고 있는 것

> **꿀팁**: "이 숫자가 올라가면(또는 내려가면) 나는 뭘 다르게 할까?"라고 자문해보세요. 답이 안 나오면 허영 지표입니다. 답이 나오면 실행 지표입니다.

### DAU, WAU, MAU — 활성 사용자의 3형제

서비스가 얼마나 "살아있는지" 보여주는 가장 기본적인 지표입니다.

| 지표 | 뜻 | 계산 방법 | 좋은 기준 (초기 서비스) |
|------|-----|-----------|------------------------|
| **DAU** | Daily Active Users | 오늘 서비스에 접속한 사용자 수 | 매일 꾸준하면 좋음 |
| **WAU** | Weekly Active Users | 이번 주 한 번이라도 접속한 사용자 수 | 가입자의 30% 이상 |
| **MAU** | Monthly Active Users | 이번 달 한 번이라도 접속한 사용자 수 | 가입자의 50% 이상 |

이 숫자들 사이의 비율도 중요합니다:

- **DAU / MAU 비율** = "스티키니스(점착도)"
  - DAU 10명 / MAU 100명 = 10% → 한 달에 한 번 정도만 옴 (보통)
  - DAU 30명 / MAU 100명 = 30% → 거의 매일 옴 (좋음!)
  - DAU 50명 / MAU 100명 = 50% → 습관처럼 매일 옴 (대단함!)

**한국 서비스 예시:**
- 당근마켓: DAU/MAU 약 50% (매일 중고 물건 보는 습관)
- 배달의민족: DAU/MAU 약 25% (일주일에 1~2번 주문)
- 여행 서비스: DAU/MAU 약 5% (여행 갈 때만 씀 — 정상)

> **막혔을 때**: "우리 서비스는 DAU가 너무 낮아요." 서비스 특성에 따라 다릅니다. 매일 쓸 필요가 없는 서비스(세무 신고, 이사 견적 등)는 DAU가 낮아도 정상입니다. 이런 경우 MAU와 "재방문율"이 더 중요한 지표입니다.

### 퍼널 분석: 어디서 사용자가 새고 있나?

퍼널(funnel, 깔때기)은 사용자가 최종 목표까지 가는 과정을 단계별로 나눈 것입니다.

```
방문 (100명)
  │
  ├─ 90명 이탈 (90%)  ← "첫 화면이 뭔지 모르겠음"
  │
  ▼
가입 (10명)
  │
  ├─ 5명 이탈 (50%)   ← "가입했는데 뭘 해야 할지 모르겠음"
  │
  ▼
핵심 기능 사용 (5명)
  │
  ├─ 3명 이탈 (60%)   ← "한 번 써봤는데 굳이..."
  │
  ▼
결제 (2명)
  │
  └─ 전체 전환율: 2%
```

**퍼널의 핵심 원리:**
- 각 단계에서 사람이 빠져나갑니다 (이건 정상)
- 문제는 **어디서 가장 많이 빠지는가**입니다
- **가장 많이 새는 곳**을 먼저 막아야 합니다

위 예시에서는 "방문 → 가입" 단계에서 90%가 이탈합니다. 이건 랜딩 페이지나 가입 절차에 문제가 있다는 신호입니다.

**단계별 이탈률 해석 가이드:**

| 단계 | 정상 이탈률 | 경고 수준 | 대표적 원인 |
|------|-------------|-----------|-------------|
| 방문 → 가입 | 70~85% | 90% 이상 | 서비스가 뭔지 이해 못함, 가입이 복잡 |
| 가입 → 핵심 기능 사용 | 40~60% | 70% 이상 | 온보딩 부재, 첫 경험 설계 미흡 |
| 핵심 기능 → 재방문 | 50~70% | 80% 이상 | 가치를 못 느낌, 알림 부재 |
| 재방문 → 결제 | 80~95% | 98% 이상 | 가격 문제, 무료로 충분 |

> **꿀팁**: 퍼널은 **아래에서 위로** 개선하세요. "방문자를 더 늘리자(광고)" 보다 "가입한 사람이 핵심 기능을 더 쓰게 하자"가 먼저입니다. 새는 바가지에 물을 더 부어봤자 소용없습니다.

### Rails에서 데이터 확인하는 법

데이터를 보는 방법을 알아야 의사결정을 할 수 있습니다. 터미널에서 Rails console을 열어봅시다.

```bash
rails console
```

Ahoy가 설치되어 있다면 이런 것들을 확인할 수 있습니다:

```ruby
# 오늘 방문자 수 (DAU)
Ahoy::Visit.where(started_at: Time.zone.today.all_day).count

# 이번 주 방문자 수 (WAU)
Ahoy::Visit.where(started_at: 1.week.ago..Time.zone.now).
  select(:user_id).distinct.count

# 가장 많이 본 페이지 Top 5
Ahoy::Event.where(name: "페이지 방문").
  group(:properties).order("count_all DESC").limit(5).count

# 특정 이벤트 횟수 (예: 가입 완료)
Ahoy::Event.where(name: "가입 완료").count

# 요일별 방문 패턴
Ahoy::Visit.group_by_day_of_week(:started_at).count
```

> **막혔을 때**: "rails console이 뭐예요?" 걱정하지 마세요. Claude에게 "이번 주 방문자 수 확인하는 명령어 알려줘"라고 물어보면 됩니다. 코드를 외울 필요 없습니다. 중요한 건 **"어떤 질문을 할지"** 아는 것입니다.

### 데이터 기반 의사결정: 3단계 프로세스

데이터를 본다고 끝이 아닙니다. 데이터로 **결정**을 내려야 합니다.

**1단계: 질문 만들기**
- "왜 가입 후 핵심 기능을 안 쓸까?"
- "어떤 페이지에서 사람들이 가장 많이 떠날까?"
- "결제한 사용자와 안 한 사용자의 차이는?"

**2단계: 데이터로 답 찾기**
- 가입 후 24시간 이내 핵심 기능 사용률: 15%
- 이탈이 가장 많은 페이지: 대시보드 (로딩 5초)
- 결제 사용자: 평균 7일 이상 사용 / 비결제: 평균 1일

**3단계: 행동 결정하기**
- "대시보드 로딩 속도를 개선하자" (이탈 원인 제거)
- "가입 후 바로 핵심 기능으로 안내하자" (온보딩 추가)
- "7일 무료 체험 후 결제 안내하자" (결제 패턴 활용)

> **꿀팁**: "데이터가 부족해서 분석할 게 없어요." 사용자가 5명이어도 분석할 수 있습니다. 5명 중 3명이 같은 지점에서 이탈하면 그건 매우 강력한 신호입니다. 통계적 유의성보다 **패턴**을 찾으세요.

- 허영 지표: 총 누적 가입자 수는 탈퇴하거나 더 이상 사용하지 않는 사람까지 포함합니다. 숫자는 계속 올라가니 기분은 좋지만, "그래서 뭘 해야 하지?"에 답을 주지 못합니다. DAU, 리텐션율, 기능 사용률은 즉각적인 행동 판단이 가능한 실행 지표입니다.
- 퍼널 개선: 가입까지는 10%가 전환하지만, 핵심 기능까지 가는 비율이 20%로 매우 낮습니다. 이건 온보딩이 없거나, 핵심 기능을 찾기 어렵다는 뜻입니다. 방문자를 아무리 늘려도(광고비만 증가) 핵심 기능까지 이어지지 않으면 의미 없습니다. "새는 바가지"를 먼저 막아야 합니다.
- DAU/MAU 비율: 이 비율을 "스티키니스(점착도)"라고 부릅니다. 30% 이상이면 좋은 서비스, 50% 이상이면 사용자의 일상에 자리 잡은 서비스입니다. 카카오톡이 DAU/MAU 80% 이상인 것처럼, 높을수록 사용자가 매일 찾아오는 습관적인 서비스입니다.
