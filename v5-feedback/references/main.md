## EXPLAIN

### 왜 피드백이 서비스 성장의 핵심인가?

서비스를 만들고 배포했다면, 이제 시작입니다. 진짜 성장은 **사용자의 목소리를 듣고 개선하는 과정**에서 일어납니다.

많은 서비스가 실패하는 이유는 기능이 부족해서가 아닙니다. **사용자가 원하는 것과 만든 것이 다르기 때문**입니다.

### 피드백 루프란?

피드백 루프는 이런 순환 구조입니다:

1. **측정** — 사용자가 어떻게 서비스를 사용하는지 데이터 수집
2. **수집** — 사용자에게 직접 의견을 받음
3. **분석** — 패턴을 찾고 개선점 도출
4. **실행** — 가장 중요한 것부터 개선
5. **반복** — 다시 1번으로 돌아감

이 루프가 빠를수록 서비스가 빠르게 성장합니다.

### ICE 프레임워크: 무엇부터 개선할까?

피드백이 쌓이면 "뭐부터 할까?"가 고민됩니다. ICE 프레임워크로 판단합니다:

- **I (Impact)** — 이 개선이 얼마나 큰 영향을 줄까? (1~10점)
- **C (Confidence)** — 이 개선이 성공할 확률은? (1~10점)
- **E (Ease)** — 얼마나 쉽게 구현할 수 있을까? (1~10점)

**ICE 점수 = (I + C + E) / 3**

점수가 높은 것부터 실행합니다.

### 분석 도구: Ahoy

Ahoy는 Rails에서 사용하기 쉬운 분석 gem입니다. 설치하면 자동으로:
- 어떤 페이지를 방문했는지 (visits)
- 어떤 버튼을 클릭했는지 (events)
- 사용자가 어디서 왔는지 (referrer)

를 기록해줍니다. Google Analytics보다 훨씬 간단하고, 데이터가 내 DB에 저장됩니다.

## EXECUTE

### 단계 1: Ahoy 분석 도구 설치

Claude에게 요청합니다:

```
내 Rails 서비스에 Ahoy 분석 도구를 설치해줘.
- 페이지 방문 추적
- 주요 버튼 클릭 추적 (회원가입, 핵심 기능 사용 등)
- 간단한 대시보드에서 데이터를 볼 수 있게
```

### 단계 2: 피드백 폼 만들기

Claude에게 요청합니다:

```
사용자 피드백을 수집할 수 있는 간단한 폼을 만들어줘.
- 피드백 유형 (버그 신고 / 기능 제안 / 일반 의견)
- 내용 작성란
- 만족도 (1~5 별점)
- 관리자만 볼 수 있는 피드백 목록 페이지
```

### 단계 3: ICE 프레임워크로 우선순위 정하기

서비스를 돌아보며 개선할 점 5가지를 떠올려봅니다. Claude에게 도움을 요청합니다:

```
내 서비스의 개선 아이디어를 ICE 프레임워크로 우선순위를 매겨줘.
아이디어:
1. [첫 번째 개선 아이디어]
2. [두 번째 개선 아이디어]
3. [세 번째 개선 아이디어]
4. [네 번째 개선 아이디어]
5. [다섯 번째 개선 아이디어]
```

### 단계 4: 개선 계획서 작성

Claude에게 요청합니다:

```
지금까지 분석한 내용을 바탕으로 my-feedback-plan.md 파일을 만들어줘.
포함할 내용:
- 현재 서비스 상태 요약
- 수집 중인 피드백 채널 목록
- ICE 점수 기반 개선 우선순위 표
- 다음 2주간 실행할 개선 항목 3가지
```
