## EXPLAIN

### "감이 아니라 데이터로" — 마인드셋 전환

서비스를 배포하셨습니다. 축하드립니다! 하지만 지금부터가 진짜 시작입니다.

많은 서비스가 이 시점에서 멈춥니다. 만든 사람은 "좋은 서비스"라고 생각하는데, 사용자는 조용히 떠납니다. 왜일까요?

**"내 생각"과 "사용자의 행동"은 놀라울 정도로 다릅니다.**

예를 들어봅시다:
- 내 생각: "이 기능이 핵심이야" → 현실: 사용자의 80%가 그 기능을 한 번도 안 씀
- 내 생각: "디자인이 예쁘면 사람들이 오겠지" → 현실: 가입 버튼을 못 찾아서 이탈
- 내 생각: "기능을 더 추가해야 해" → 현실: 기존 기능을 제대로 쓰는 사람이 없음

이런 차이를 메우는 유일한 방법이 **피드백**입니다. 감으로 결정하는 대신, 데이터와 사용자 의견을 보고 결정합니다.

> **꿀팁**: "나는 이 서비스의 사장이 아니라 탐정이다"라고 생각하세요. 탐정은 증거를 수집하고, 증거를 바탕으로 판단합니다. 감으로 범인을 잡지 않습니다.

### 피드백 루프: 서비스 성장 엔진

피드백 루프는 서비스를 **살아있게** 만드는 엔진입니다. 한 번 만들고 끝나는 게 아니라, 계속 돌아가야 합니다.

```
    ┌─────────────────────────────────────┐
    │                                     │
    ▼                                     │
 1. 측정하기                              │
 (사용자가 뭘 하는지 데이터 수집)          │
    │                                     │
    ▼                                     │
 2. 수집하기                              │
 (사용자에게 직접 물어보기)                │
    │                                     │
    ▼                                     │
 3. 분석하기                           5. 반복
 (패턴을 찾고 문제점 발견)                 │
    │                                     │
    ▼                                     │
 4. 실행하기                              │
 (가장 중요한 것 하나만 개선)              │
    │                                     │
    └─────────────────────────────────────┘
```

**핵심**: 이 루프가 빠를수록 서비스가 빠르게 좋아집니다. 한 달에 한 바퀴 도는 서비스와 일주일에 한 바퀴 도는 서비스는 3개월 뒤 완전히 다른 서비스가 됩니다.

### ICE 프레임워크: "뭐부터 할까?"를 해결하는 도구

피드백이 쌓이면 개선 아이디어가 10개, 20개 쏟아집니다. 이때 가장 위험한 함정이 있습니다:

**"다 중요해 보여서 아무것도 못 하는 상태"**

ICE 프레임워크는 이 문제를 단순하게 해결합니다.

| 항목 | 뜻 | 질문 | 점수 |
|------|-----|------|------|
| **I** (Impact) | 영향력 | "이걸 개선하면 사용자에게 얼마나 큰 변화가 있을까?" | 1~10 |
| **C** (Confidence) | 확신도 | "이 개선이 실제로 효과가 있을 확률은?" | 1~10 |
| **E** (Ease) | 용이성 | "Claude에게 요청하면 얼마나 빨리 만들 수 있을까?" | 1~10 |

**ICE 점수 = (I + C + E) / 3**

한국어 서비스 예시로 살펴봅시다:

**예시 서비스: 동네 맛집 추천 서비스**

| 개선 아이디어 | I | C | E | ICE 점수 |
|---------------|---|---|---|----------|
| 가입 후 첫 화면에 추천 맛집 3개 바로 보여주기 | 9 | 8 | 7 | **8.0** |
| 카카오 로그인 추가 | 8 | 9 | 6 | **7.7** |
| "이 맛집 가봤어요" 후기 기능 | 7 | 6 | 5 | **6.0** |
| 맛집 지도 기능 | 8 | 7 | 3 | **6.0** |
| AI 기반 맞춤 추천 알고리즘 개선 | 9 | 4 | 2 | **5.0** |

보이시나요? "AI 추천 알고리즘 개선"은 Impact가 높지만, Confidence와 Ease가 낮아서 우선순위가 낮습니다. 반면 "첫 화면에 추천 맛집 3개 보여주기"는 세 항목 모두 높아서 **먼저 해야 합니다.**

> **꿀팁**: Ease 점수를 매길 때 "Claude에게 요청해서 30분 안에 가능한가?"를 기준으로 삼으세요. Claude Code를 쓰는 여러분에게 대부분의 기능 구현은 Ease 6점 이상입니다!

### Ahoy: 내 서비스에 눈을 달아주는 도구

Ahoy는 Rails에서 가장 쉽게 쓸 수 있는 분석 도구입니다. Google Analytics와 비교해봅시다:

| 비교 항목 | Google Analytics | Ahoy |
|-----------|-----------------|------|
| 설치 난이도 | 복잡한 태그 설정 | gem 하나 추가 |
| 데이터 저장 위치 | 구글 서버 (미국) | **내 DB** |
| 커스터마이징 | 제한적 | 자유자재 |
| Rails 통합 | 별도 설정 필요 | 자연스러운 통합 |
| 가격 | 무료 (데이터를 구글에 제공) | **완전 무료** |

Ahoy가 자동으로 기록해주는 것들:
- **방문 기록 (visits)** — 누가, 언제, 어디서 들어왔는지
- **이벤트 (events)** — 어떤 버튼을 클릭했는지, 어떤 기능을 사용했는지
- **유입 경로 (referrer)** — 네이버에서 왔는지, 카카오톡에서 왔는지
- **기기 정보** — 모바일인지, PC인지

**데이터가 내 DB에 저장된다**는 것이 가장 큰 장점입니다. Rails console에서 직접 데이터를 조회하고 분석할 수 있습니다.

### 주간 개선 사이클: 매주 금요일 30분

서비스 성장은 거창한 게 아닙니다. **매주 금요일 30분**만 투자하면 됩니다.

```
[금요일 주간 개선 사이클]

1단계 (10분): 숫자 확인
   - 이번 주 방문자 수
   - 신규 가입자 수
   - 핵심 기능 사용 횟수
   → "지난주보다 나아졌나?"

2단계 (10분): 피드백 확인
   - 피드백 폼에 들어온 의견
   - 주변 사람들의 반응
   - 내가 직접 써보면서 느낀 불편함
   → "사용자가 뭘 원하나?"

3단계 (10분): 다음 주 할 일 1개 정하기
   - ICE 점수 가장 높은 것 선택
   - Claude에게 구현 요청할 내용 정리
   → "다음 주엔 이것만 한다"
```

> **막혔을 때**: "이번 주는 딱히 개선할 게 없는데?"라고 느끼면, 직접 서비스를 처음부터 끝까지 써보세요. 가입부터 핵심 기능 사용까지. 반드시 불편한 점이 하나는 보입니다. 그게 이번 주의 개선 대상입니다.

## EXECUTE

### 단계 1: Ahoy 분석 도구 설치

Claude에게 요청합니다:

```
내 Rails 서비스에 Ahoy 분석 도구를 설치해줘.

1. Gemfile에 ahoy_matey gem 추가
2. ahoy:install 제너레이터 실행
3. 마이그레이션 실행
4. 추적할 이벤트 설정:
   - 회원가입 완료
   - 핵심 기능 사용 (내 서비스의 메인 기능)
   - 결제 페이지 진입
5. ApplicationController에 방문 추적 설정

주의: 개인정보(이메일, 이름 등)는 추적하지 않도록 해줘.
```

설치가 끝나면 확인합니다:

```
Ahoy가 잘 설치됐는지 확인해줘.
- rails console에서 Ahoy::Visit.count, Ahoy::Event.count 확인
- 테스트 이벤트 하나 발생시켜서 기록되는지 확인
```

> **꿀팁**: Ahoy 설치 후 며칠 동안은 데이터가 쌓이는 걸 지켜보세요. 최소 1주일치 데이터가 있어야 의미 있는 분석이 가능합니다.

### 단계 2: 피드백 폼 만들기

사용자가 의견을 보낼 수 있는 간단한 폼을 만듭니다.

Claude에게 요청합니다:

```
사용자 피드백을 수집하는 기능을 만들어줘.

피드백 모델:
- category: 버그 신고 / 기능 제안 / 불편한 점 / 칭찬  (select)
- content: 내용 (textarea, 필수)
- rating: 만족도 1~5점 (별점 클릭)
- page_url: 어느 페이지에서 보냈는지 (자동 수집)

사용자 화면:
- 모든 페이지 오른쪽 하단에 "의견 보내기" 플로팅 버튼
- 클릭하면 모달로 피드백 폼 열림
- 로그인 안 한 사용자도 보낼 수 있게

관리자 화면:
- /admin/feedbacks 에서 피드백 목록 확인
- 카테고리별 필터링
- 평균 만족도 표시
- 각 피드백에 "처리 완료" 체크 가능
```

> **막혔을 때**: "피드백 폼을 만들었는데 아무도 안 보내요." 정상입니다! 처음에는 주변 지인 3~5명에게 직접 부탁하세요. "우리 서비스 한번 써보고, 이 버튼 눌러서 솔직하게 의견 보내줘" 이렇게요. 5명의 피드백이면 충분히 좋은 인사이트를 얻을 수 있습니다.

서비스를 돌아보며 개선할 점 5가지를 떠올려봅니다. 사용자 피드백 + 본인이 직접 사용하면서 느낀 불편함을 합쳐서 5가지를 정합니다.

Claude에게 요청합니다:

```

4. [예: 카카오 로그인 지원]

각 아이디어에 대해:
- Impact (1~10): 이유 설명
- Confidence (1~10): 이유 설명
- Ease (1~10): 이유 설명
- ICE 점수 계산
- 최종 우선순위 표로 정리

내 서비스 정보: [서비스 간단 설명]
```

정답:
- 1번째 질문 → "Confidence (확신도)"
- 2번째 질문 → "수집 → 분석 → 실행 → 반복의 순환을 빠르게 돌리는 것"
- 3번째 질문 → "데이터가 내 DB에 저장되어 자유롭게 분석 가능하다"

해설:
- ICE의 C: Confidence는 "이 개선이 실제로 효과가 있을 확률"을 의미합니다. 데이터에 기반한 개선은 Confidence가 높고, "이렇게 하면 좋지 않을까?" 감에 의존한 개선은 Confidence가 낮습니다. 예를 들어 "5명의 사용자가 가입이 어렵다고 했으니 가입 절차를 줄이자"는 Confidence 8~9점이고, "AI 추천을 넣으면 좋지 않을까?"는 Confidence 3~4점입니다.
- 피드백 루프: 핵심은 **"루프(순환)"**입니다. 한 번에 완벽하게 하는 게 아니라, 작은 개선을 빠르게 반복하는 것이 서비스 성장의 비결입니다. 린(Lean) 스타트업 방법론의 핵심이기도 합니다. 매주 1개씩 개선하면 1년에 52번 개선됩니다.
- Ahoy의 장점: 데이터가 구글 서버가 아닌 내 DB에 저장됩니다. 이는 Rails console에서 `Ahoy::Event.where(name: "가입 완료").count`처럼 직접 조회할 수 있다는 뜻입니다. 내 서비스에 맞는 맞춤 분석이 가능합니다.
