## EXPLAIN

### 왜 "만들었습니다"만으로는 안 되는가

면접에서 "프로젝트 소개해주세요"라는 질문을 받으면, 대부분 이렇게 답합니다:

> "Rails로 웹서비스를 만들었습니다. AI 기능이 있고, 결제도 됩니다."

면접관의 속마음: *"그래서... 뭘 어쨌다는 거지?"*

이건 **기술 나열**이지 **스토리**가 아닙니다. 기술 나열은 누구나 합니다. 차별점은 **"이 기술로 어떤 문제를 해결했는가"**입니다.

같은 프로젝트를 이렇게 말하면 어떨까요?

> "소상공인들이 재고를 엑셀로 관리하다 실수가 잦다는 문제를 발견하고, AI가 자동으로 발주를 추천하는 서비스를 만들었습니다. 베타 사용자 15명이 사용 중이고, 재고 실수가 평균 40% 줄었습니다."

느낌이 완전히 다르죠? 이 차이를 만드는 것이 **STAR-M 프레임워크**입니다.

### STAR-M 프레임워크: 프로젝트를 임팩트 있게 정리하기

STAR는 원래 면접 답변 프레임워크입니다. 여기에 **M(Metric/Meaning)**을 추가해서 서비스 빌딩 경험에 맞게 만들었습니다.

| 단계 | 뜻 | 질문 | 예시 |
|------|-----|------|------|
| **S** (Situation) | 상황 | 어떤 문제가 있었나? | "소상공인들이 엑셀로 재고 관리를 하다 월 평균 5건의 실수 발생" |
| **T** (Task) | 과제 | 내가 뭘 하기로 했나? | "AI 기반 재고 관리 SaaS를 기획부터 배포까지 1인 개발" |
| **A** (Action) | 행동 | 구체적으로 뭘 했나? | "Rails + Claude API로 자동 발주 추천, Toss 결제 연동, 6주 개발" |
| **R** (Result) | 결과 | 어떤 결과가 나왔나? | "베타 사용자 15명, 재고 실수 40% 감소, 월 구독 매출 발생" |
| **M** (Metric/Meaning) | 의미 | 왜 이게 중요한가? | "비개발자도 AI를 활용해 실제 문제를 해결할 수 있음을 증명" |

**각 단계 작성 팁:**

**S (Situation) — "공감"을 이끌어내세요**
- 나쁜 예: "재고 관리 서비스를 만들었습니다"
- 좋은 예: "동네 치킨집 사장님이 매주 2시간씩 엑셀에 재고를 입력하고 있었습니다"
- 포인트: 구체적인 사람, 구체적인 고통을 그려야 합니다

**T (Task) — "범위"를 보여주세요**
- 나쁜 예: "웹사이트를 만들었습니다"
- 좋은 예: "기획, 디자인, 개발, 배포, 운영까지 전 과정을 1인으로 담당했습니다"
- 포인트: "혼자서 전부 해냈다"는 것이 Vuild 경험의 최대 장점입니다

**A (Action) — "선택의 이유"를 설명하세요**
- 나쁜 예: "Rails와 Claude API를 사용했습니다"
- 좋은 예: "빠른 프로토타이핑을 위해 Rails를, 자연어 처리를 위해 Claude API를 선택했습니다"
- 포인트: 기술 나열이 아니라 "왜 이 기술을 선택했는지"가 중요합니다

**R (Result) — "숫자"로 말하세요**
- 나쁜 예: "좋은 서비스를 만들었습니다"
- 좋은 예: "베타 사용자 15명 확보, 재고 실수 40% 감소, NPS 8.2점"
- 포인트: 숫자가 없어도 "측정하려 했다"는 자세 자체가 차별점입니다

**M (Metric/Meaning) — "성장"을 보여주세요**
- 나쁜 예: "좋은 경험이었습니다"
- 좋은 예: "비개발자도 AI 도구를 활용하면 6주 만에 실제 서비스를 만들 수 있다는 것을 증명했습니다"
- 포인트: 이 경험이 나의 가치관/성장과 어떻게 연결되는지 보여줍니다

### Before/After: 숫자의 힘

"좋은 서비스를 만들었습니다"보다 "사용자의 업무 시간을 주당 3시간 줄였습니다"가 **100배 강력합니다.**

Before/After를 정리하는 틀:

```
[Before/After 비교표 예시]

┌────────────────────────────────────────────────┐
│  Before (서비스 없이)    →    After (서비스 사용) │
├────────────────────────────────────────────────┤
│  재고 입력: 주 2시간    →    자동 입력: 주 10분  │
│  발주 실수: 월 5건      →    실수: 월 3건       │  
│  재고 파악: 엑셀 뒤적임 →    대시보드에서 즉시   │
│  발주 판단: 감으로      →    AI 추천 기반       │
└────────────────────────────────────────────────┘
```

**숫자로 보여줄 수 있는 카테고리:**

| 카테고리 | 예시 지표 | 측정 방법 |
|----------|-----------|-----------|
| **사용자** | 가입자 수, DAU, NPS | Ahoy 데이터, 피드백 폼 |
| **기술** | 개발 기간, 페이지 수, API 수 | 개발 일지, git 히스토리 |
| **비즈니스** | 매출, 전환율, 이탈률 | 결제 데이터, 분석 대시보드 |
| **효율** | 시간 절약, 에러 감소 | 사용자 인터뷰, 비교 측정 |

> **꿀팁**: 숫자가 작아도 괜찮습니다. "사용자 5명"이 부끄러울 수 있지만, "0명에서 5명을 만들었다"는 것은 의미 있는 성과입니다. 면접관이 보는 것은 숫자의 크기가 아니라 **"측정하고 개선하려는 자세"**입니다.

> **막혔을 때**: "숫자가 진짜 하나도 없어요." 다음을 측정해보세요: 개발 기간(첫 커밋부터 배포까지 며칠?), 코드량(git log로 확인), 기능 수(Route 수), 페이지 수. 이것만으로도 "6주간 15개 페이지, 5개 API 엔드포인트를 포함한 풀스택 서비스 개발"이라는 임팩트 있는 문장을 만들 수 있습니다.

### 이력서에 서비스 경험 넣는 법

이력서에서 프로젝트 경험을 어떻게 표현하느냐에 따라 인상이 완전히 달라집니다.

**나쁜 예시:**
```
프로젝트: 맛집 추천 서비스
기술 스택: Ruby on Rails, PostgreSQL, Tailwind CSS, Claude API
기간: 2024.03 ~ 2024.05
내용: 사용자에게 맛집을 추천하는 웹 서비스 개발
```

**좋은 예시:**
```
프로젝트: 동네맛집 — AI 기반 개인 맛집 큐레이션 서비스

[문제] 매일 "뭐 먹지?" 고민하는 직장인의 점심 선택 시간 평균 15분

[해결] Claude API를 활용한 취향 기반 맛집 추천 엔진 개발
- 기획 → 개발 → 배포 → 운영 전 과정 1인 수행 (6주)
- Rails 8 + Hotwire로 SPA 수준의 반응형 UI 구현
- Toss Payments 결제 연동, 3-Tier 구독 모델 설계
- Ahoy 기반 사용자 행동 분석 파이프라인 구축

[성과]
- 베타 사용자 20명 확보, DAU 8명 유지
- 사용자의 점심 선택 시간 15분 → 3분으로 단축 (사용자 인터뷰)
- NPS 8.5점, 피드백 기반 주간 개선 사이클 운영

배포: https://my-app.com | GitHub: github.com/me/my-app
```

어떤 이력서를 더 자세히 읽고 싶으세요?

### 면접 답변 템플릿

면접에서 자주 나오는 질문별 답변 구조입니다:

**Q: "이 프로젝트에 대해 설명해주세요."**
```
[30초 버전 — 엘리베이터 피치]
"[타겟 사용자]의 [구체적 문제]를 해결하기 위해 
[핵심 기술]로 [서비스]를 만들었습니다. 
현재 [숫자 성과]를 달성했습니다."

예시:
"바쁜 직장인의 점심 선택 고민을 해결하기 위해
AI 맛집 추천 서비스를 만들었습니다.
베타 사용자 20명이 활발하게 사용 중이고,
점심 선택 시간을 평균 12분 줄였습니다."
```

**Q: "기술적으로 가장 어려웠던 부분은?"**
```
[구조]
1. 어려웠던 상황 설명 (1줄)
2. 시도했던 방법과 왜 안 됐는지 (1~2줄)
3. 최종 해결 방법과 이유 (2~3줄)
4. 이 경험에서 배운 점 (1줄)

예시:
"Claude API 응답 시간이 5초 이상 걸려서 UX가 나빴습니다.
처음에는 응답을 캐싱하려 했는데, 개인화 추천이라 캐싱이 어려웠습니다.
Turbo Streams를 활용해 '추천 중...' 로딩 UI를 먼저 보여주고,
백그라운드에서 응답이 오면 실시간으로 업데이트하는 방식으로 해결했습니다.
이 경험으로 '체감 속도'와 '실제 속도'가 다르다는 걸 배웠습니다."
```

**Q: "왜 이 기술 스택을 선택했나요?"**
```
[구조]
1. 요구사항 (1줄)
2. 비교한 대안 (1줄)
3. 선택 이유 (2줄)

예시:
"빠른 프로토타이핑이 필요했고, 프론트와 백엔드를 혼자 해야 했습니다.
Next.js + API 서버 분리도 고려했지만,
Rails + Hotwire가 풀스택을 1인이 운영하기에 가장 효율적이라 판단했습니다.
실제로 6주 만에 MVP를 배포할 수 있었고, 이 판단이 맞았다고 생각합니다."
```

**Q: "개발자로서 어떻게 성장하고 싶나요?"**
```
[Vuild 경험을 활용한 답변]
"이번 서비스 빌딩 경험에서 '사용자 문제를 해결하는 것'이
개발의 본질이라는 걸 깨달았습니다.
기술은 문제 해결의 도구이고, 
어떤 문제를 풀 것인지가 더 중요하다고 생각합니다.
앞으로도 기술력을 키우되, 
항상 '이걸로 누구의 어떤 문제를 해결하는가'를 
중심에 두는 개발자가 되고 싶습니다."
```

> **꿀팁**: 면접 답변은 **외우지 마세요**. 대신 STAR-M 구조만 기억하세요. 어떤 질문이 와도 "상황 → 과제 → 행동 → 결과 → 의미" 순서로 말하면 자연스럽게 좋은 답변이 됩니다.

### LinkedIn / 원티드 프로필 최적화

포트폴리오가 있어도 아무도 안 보면 의미가 없습니다. 프로필을 최적화해서 사람들이 **찾아오게** 만들어야 합니다.

**LinkedIn 프로필 체크리스트:**

| 항목 | 나쁜 예 | 좋은 예 |
|------|---------|---------|
| 헤드라인 | "학생" | "AI를 활용한 서비스 빌더 / Rails Full-stack" |
| 요약 | (비어있음) | STAR-M 요약 + 서비스 링크 |
| 경험 | "개인 프로젝트" | "AI 맛집 추천 서비스 기획/개발/운영" |
| 기술 | 나열만 | 기술 + 활용 맥락 함께 |

**원티드 프로필 체크리스트:**

| 항목 | 팁 |
|------|-----|
| 한 줄 소개 | "[서비스명] 만든 사람. 문제를 발견하고 해결하는 것을 좋아합니다." |
| 이력/경험 | 서비스 경험을 프로젝트가 아닌 "경험"으로 작성 |
| 포트폴리오 | 배포 URL + GitHub + 성과 숫자 |
| 기술 스택 | 실제 사용한 기술만, 수준과 함께 |

**핵심: "나는 이런 문제를 해결하는 사람입니다"라는 메시지가 일관되게 전달되어야 합니다.**

> **막혔을 때**: "프로필에 뭘 써야 할지 모르겠어요." STAR-M으로 정리한 내용을 그대로 프로필에 넣으면 됩니다. 결국 하나의 프로젝트를 여러 형태(이력서, 면접, 프로필)로 변환하는 것뿐입니다.

### 커리어 활용 방향: 4가지 길

Vuild에서 쌓은 서비스 빌딩 경험은 다양한 커리어 방향에 활용됩니다:

```
[서비스 빌딩 경험의 4가지 활용]

        ┌──────────────────────┐
        │   서비스 빌딩 경험    │
        └──────────┬───────────┘
                   │
     ┌─────────────┼─────────────┐
     │             │             │
     ▼             ▼             ▼
┌─────────┐ ┌──────────┐ ┌──────────┐
│ 창업/    │ │ 취업     │ │ 프리랜서 │
│ 사이드   │ │ 포트폴리오│ │          │
│ 프로젝트 │ │          │ │          │
└─────────┘ └──────────┘ └──────────┘
  "서비스를    "혼자 기획~    "실제 서비스
   계속       배포~운영      운영 경험이
   성장시키기" 까지 한 경험"  신뢰도를 높임"

                              ┌──────────┐
                              │ 사내      │
                              │ 프로젝트  │
                              └──────────┘
                              "AI 활용 역량
                               증명으로
                               리더십 기회"
```

어떤 길을 가든, **"0에서 1을 만들어본 경험"**은 강력한 차별점입니다.

## EXECUTE

### 단계 1: STAR-M으로 프로젝트 정리

Claude에게 요청합니다:

```
내 서비스 프로젝트를 STAR-M 프레임워크로 정리해줘.

내 서비스 정보:
- 서비스명: [이름]
- 주요 기능: [핵심 기능 설명]
- 타겟 사용자: [누구를 위한 서비스인지]
- 개발 기간: [시작일 ~ 현재]
- 현재 상태: [사용자 수, 매출 등]

STAR-M 각 항목을:
1. 한 줄 버전 (30자)
2. 세 줄 버전 (이력서용)  
3. 상세 버전 (면접/포트폴리오용)
으로 각각 작성해줘.

숫자가 없는 부분은 정성적으로, 
측정 가능한 부분은 정량적으로 표현해줘.
```

### 단계 2: Before/After 숫자 정리

Claude에게 요청합니다:

```
내 서비스 프로젝트의 Before/After를 정리해줘.

카테고리별로:

1. 사용자 관련
   - 서비스 없이 (Before): [문제 상황]
   - 서비스 사용 (After): [개선된 상황]

2. 기술 관련
   - 개발 기간: ___일
   - 총 페이지 수: ___개
   - API 엔드포인트 수: ___개
   - 사용 기술: [스택]

3. 비즈니스 관련
   - 가입자 수: ___명
   - 활성 사용자: ___명
   - 매출: ___원 (있다면)

깔끔한 비교표로 만들어줘.
숫자가 없는 항목은 git 히스토리나 
서비스 데이터에서 뽑을 수 있는 방법도 알려줘.
```

### 단계 3: 면접 답변 준비

Claude에게 요청합니다:


해설:
- STAR-M의 M: 기존 STAR 프레임워크에 M을 추가했습니다. M은 Metric(지표)과 Meaning(의미) 두 가지를 포함합니다. 단순히 "무엇을 했다"를 넘어 "숫자로 증명한 결과"와 "이 경험이 왜 의미 있는가"를 설명합니다. 이것이 지원자의 깊이와 성장 마인드셋을 보여주는 핵심입니다.
- Before/After: 기술 나열은 누구나 합니다. "Rails, PostgreSQL, Tailwind"를 쓴 사람은 수천 명입니다. 차별점은 "이 기술로 어떤 문제를 해결했는가"입니다. Before/After 숫자는 구체적 증거가 되고, 스토리는 읽는 사람의 기억에 남습니다. 이것이 서비스 빌딩 경험의 진짜 가치입니다.
- 면접 답변: 어려운 점이 없었다고 하면 "경험이 얕다"고 느끼고, 너무 어려웠다고만 하면 "해결 능력이 부족하다"고 느낍니다. "문제 → 시도 → 해결 → 배움"의 구조는 **성장하는 개발자**라는 인상을 줍니다. 실패한 시도를 언급하는 것도 좋습니다. "처음에 이렇게 했는데 안 돼서, 다시 이렇게 했더니 됐다"는 사고 과정을 보여주는 것입니다.
