## EXPLAIN

### 왜 같은 AI인데 결과가 다를까?

두 사람이 Claude에게 같은 기능을 요청합니다.

**A씨의 요청:**
```
검색 기능 만들어줘
```

**B씨의 요청:**
```
우리 서비스의 레시피 검색 기능을 만들어줘.

Context: 요리 초보 대상 레시피 공유 플랫폼. 현재 Rails 7 + Hotwire 스택.
Constraint: 한글 초성 검색 지원, 검색 결과 실시간 업데이트(Turbo Frame), 3개 이하 재료로 필터링 가능.
Criteria: 검색어 입력 후 300ms 이내 결과 표시, 결과 없을 때 "이런 레시피는 어때요?" 추천 노출.
```

A씨는 3번의 수정 요청 끝에 원하는 결과를 얻었습니다. B씨는 첫 요청에서 90% 완성된 결과를 받았습니다.

차이는 **프롬프트의 명확성**입니다.

### 3C 패턴: Context, Constraint, Criteria

좋은 프롬프트에는 항상 3가지가 담겨 있습니다.

**Context (맥락)**: 지금 상황이 어떤지
- 어떤 서비스인지, 누구를 위한 것인지
- 현재 기술 스택, 기존 코드 구조
- "요리 초보 대상 레시피 공유 플랫폼, Rails 7 + Hotwire"

**Constraint (제약)**: 반드시 지켜야 할 조건
- 기술적 제약: "Turbo Frame 사용", "기존 API 형식 유지"
- 비즈니스 제약: "모바일 우선", "한국어 지원 필수"
- "한글 초성 검색 지원, Turbo Frame으로 실시간 업데이트"

**Criteria (기준)**: 완성의 기준이 무엇인지
- 성능 기준: "300ms 이내 응답"
- UX 기준: "결과 없을 때 대안 추천"
- "검색 결과 300ms 이내, 빈 결과 시 추천 노출"

### 3C가 없으면 생기는 일

| 빠진 요소 | Claude가 하는 일 | 결과 |
|-----------|-----------------|------|
| Context 없음 | 일반적인 구조로 추측 | 내 프로젝트와 안 맞는 코드 |
| Constraint 없음 | 가장 보편적인 방식 선택 | 요구사항 누락, 재작업 |
| Criteria 없음 | "동작하면 완성"으로 판단 | 품질 기준 불명확 |

핵심: Claude는 똑똑하지만 **여러분의 머릿속을 읽을 수 없습니다**. 3C는 여러분의 의도를 전달하는 프레임워크입니다.

### 실전 Before/After 예시

**기능 추가 요청:**

Before:
```
다크 모드 추가해줘
```

After:
```
다크 모드를 추가해줘.

Context: Tailwind CSS 사용 중인 SaaS 대시보드. 현재 라이트 모드만 지원.
Constraint: Tailwind의 dark: 접두사 활용, 시스템 설정 자동 감지 + 수동 토글 버튼, 기존 색상 변수 기반.
Criteria: 페이지 깜빡임 없이 전환, localStorage에 설정 저장, 모든 컴포넌트에 일관 적용.
```

**버그 수정 요청:**

Before:
```
로그인이 안 돼
```

After:
```
로그인 시 500 에러가 발생해.

Context: Devise gem 사용 중, 어제 Rails 7.1에서 7.2로 업그레이드함.
Constraint: Devise 설정은 변경하지 않았고, 회원가입은 정상 동작.
Criteria: 기존 사용자가 이메일/비밀번호로 로그인 시 세션이 정상 생성되어야 함.
```

## EXECUTE

### Step 1: Bad Prompt 체험 (5분)

코드 에디터에서 Claude Code를 열고, 자신의 서비스에 필요한 기능을 **일부러 모호하게** 요청해봅니다:

```
[기능 이름] 만들어줘
```

Claude의 응답을 살펴봅니다. 다음을 관찰하세요:
- Claude가 어떤 가정을 했는가?
- 내가 원하던 것과 얼마나 다른가?
- 어떤 부분이 부족한가?

### Step 2: 3C 패턴 적용 (10분)

같은 기능을 3C 패턴으로 다시 요청합니다:

```
[기능 이름]을 만들어줘.

Context: [서비스 설명, 대상 사용자, 기술 스택]
Constraint: [반드시 지켜야 할 조건 2~3가지]
Criteria: [완성의 기준 2~3가지]
```

두 결과를 비교합니다:
- 어떤 차이가 있는가?
- 3C 중 어떤 요소가 가장 큰 차이를 만들었는가?

### Step 3: 프롬프트 라이브러리 만들기 (10분)

자주 하는 요청 유형별로 3C 템플릿을 만들어 저장합니다.

Claude에게 다음과 같이 요청합니다:

```
내가 자주 하는 요청 유형(기능 추가, 버그 수정, 리팩토링, UI 수정)에 대해
3C 패턴 템플릿을 만들어서 my-prompt-library.md로 저장해줘.

각 템플릿에는:
- 유형별 3C 체크리스트
- 좋은 예시 1개
- 나쁜 예시 1개

내 서비스 컨텍스트:
[서비스 이름, 기술 스택, 대상 사용자 등]
```

### Step 4: Before/After 비교 (마무리)

오늘 작성한 Bad prompt와 Good prompt를 나란히 보며:
- 3C 패턴이 자연스러운지
- 앞으로 어떤 요청에서 가장 활용도가 높을지

정리하고 `my-prompt-library.md`에 오늘의 Before/After 사례를 추가합니다.

## QUIZ

다음 질문에 답해봅니다:

1. 3C 패턴의 세 요소(Context, Constraint, Criteria)를 각각 한 문장으로 설명해보세요.

2. 다음 프롬프트에서 빠진 3C 요소는 무엇인가요?
```
게시판에 댓글 기능을 추가해줘. 대댓글은 1단계까지만 지원하고, Turbo Stream으로 실시간 업데이트 해줘.
```

3. 왜 모호한 프롬프트도 "어느 정도" 동작하는 결과를 만들어내는데, 그래도 3C를 써야 할까요?

정답:
- 1번: Context는 "지금 상황과 배경", Constraint는 "반드시 지켜야 할 제약 조건", Criteria는 "완성으로 인정하는 기준"
- 2번: Context가 빠져 있음. 어떤 서비스인지, 기술 스택이 무엇인지, 대상 사용자가 누구인지 알 수 없음. Criteria도 부족함 (댓글 로딩 속도, 에러 처리 기준 등)
- 3번: Claude가 "보편적 추측"으로 채운 부분이 내 서비스와 맞지 않을 확률이 높기 때문. 재작업 비용이 3C 작성 비용보다 훨씬 큼. 특히 프로젝트가 커질수록 맥락 불일치의 누적 비용이 기하급수적으로 증가함.
