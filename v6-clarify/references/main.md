## EXPLAIN

### 왜 같은 AI인데 결과가 다를까?

두 사람이 동시에 Claude Code를 열고 똑같은 기능을 요청합니다.

**A씨의 요청:**
```
검색 기능 만들어줘
```

**B씨의 요청:**
```
우리 서비스의 레시피 검색 기능을 만들어줘.

Context: 요리 초보 대상 레시피 공유 플랫폼. 현재 Rails 7 + Hotwire 스택.
Constraint: 한글 초성 검색 지원, 검색 결과 실시간 업데이트(Turbo Frame), 3개 이하 재료로 필터링 가능.
Criteria: 검색어 입력 후 300ms 이내 결과 표시, 결과 없을 때 "이런 레시피는 어때요?" 추천 노출.
```

A씨는 3번의 수정 요청 끝에 원하는 결과를 얻었습니다. 총 소요시간 25분.
B씨는 첫 요청에서 90% 완성된 결과를 받았습니다. 총 소요시간 8분.

차이는 코딩 실력이 아닙니다. **프롬프트의 명확성**입니다.

이 차이가 하루에 5번 반복되면? 한 달이면? A씨는 수정 요청에만 수십 시간을 쏟고, B씨는 그 시간에 새로운 기능을 만들고 있습니다.

### 3C 패턴: Context, Constraint, Criteria

좋은 프롬프트에는 항상 세 가지가 담겨 있습니다. 하나라도 빠지면 Claude는 **추측**으로 채울 수밖에 없습니다.

#### Context (맥락) — "지금 상황이 어떤지"

Claude에게 작업 배경을 알려줍니다. 의사에게 증상만 말하는 게 아니라 생활 습관, 복용 약, 과거 병력도 함께 말하는 것과 같습니다.

포함할 것:
- **서비스 설명**: 어떤 서비스인지, 누구를 위한 것인지
- **기술 환경**: 사용 중인 기술 스택, 기존 코드 구조
- **현재 상태**: 이미 구현된 것, 진행 중인 것

예시:
```
Context: 요리 초보 대상 레시피 공유 플랫폼. 
현재 Rails 7 + Hotwire 스택, 사용자 500명, 레시피 2000개 등록.
검색은 아직 없고, 카테고리 필터만 있는 상태.
```

#### Constraint (제약) — "반드시 지켜야 할 조건"

Claude가 자유롭게 해석하면 안 되는 부분을 명시합니다. 건축가에게 "예쁜 집 지어줘"가 아니라 "3층 이하, 방 3개, 주차장 포함"이라고 말하는 것입니다.

포함할 것:
- **기술적 제약**: "Turbo Frame 사용", "기존 API 형식 유지", "jQuery 금지"
- **비즈니스 제약**: "모바일 우선", "한국어 지원 필수", "무료 플랜에서도 사용 가능"
- **범위 제약**: "이번에는 백엔드만", "UI 수정 없이 로직만"

예시:
```
Constraint: 
- 한글 초성 검색 지원 (ㅂㅂ → 불백볶음밥)
- Turbo Frame으로 페이지 새로고침 없이 실시간 업데이트
- 3개 이하 재료로 필터링 가능
- 기존 Recipe 모델 구조 변경하지 않기
```

#### Criteria (기준) — "완성이란 무엇인지"

"잘 동작하면 됐지"는 기준이 아닙니다. Claude에게 합격선을 명확히 알려줍니다.

포함할 것:
- **성능 기준**: "300ms 이내 응답", "이미지 1MB 이하"
- **UX 기준**: "결과 없을 때 추천 노출", "로딩 중 스피너 표시"
- **품질 기준**: "테스트 포함", "에러 메시지 한국어로"

예시:
```
Criteria:
- 검색어 입력 후 300ms 이내에 결과 표시
- 결과 없을 때 "이런 레시피는 어때요?" 추천 3개 노출
- 검색 결과에 레시피 썸네일 + 소요시간 + 난이도 표시
- 한글/영어 혼합 검색 시에도 정상 동작
```

### 3C가 없으면 생기는 일

| 빠진 요소 | Claude가 하는 일 | 결과 |
|-----------|-----------------|------|
| Context 없음 | 가장 보편적인 구조로 추측 | 내 프로젝트와 안 맞는 코드, 처음부터 재작업 |
| Constraint 없음 | 가장 일반적인 방식 선택 | 핵심 요구사항 누락, "이것도 추가해줘" 반복 |
| Criteria 없음 | "동작하면 완성"으로 판단 | 품질 불균일, "이건 좀 아닌데..." 반복 |
| 3개 다 없음 | 전부 추측으로 채움 | 처음부터 다시 설명하느라 시간 낭비 |

핵심: Claude는 매우 똑똑하지만 **여러분의 머릿속을 읽을 수 없습니다**. 3C는 여러분의 의도를 정확하게 전달하는 프레임워크입니다.

### 실전 Before/After 5쌍

#### 1. 기능 추가 요청

**Bad:**
```
다크 모드 추가해줘
```
Claude는 "어떤 프로젝트?", "CSS 프레임워크?", "시스템 설정 연동?" 전부 추측합니다.

**Good:**
```
다크 모드를 추가해줘.

Context: Tailwind CSS 사용 중인 SaaS 대시보드. 현재 라이트 모드만 지원. 
사용자 설정 페이지(/settings)가 이미 있음.

Constraint: 
- Tailwind의 dark: 접두사 활용
- 시스템 설정 자동 감지 + 수동 토글 버튼
- 기존 색상 변수(colors.css) 기반으로 다크 색상 정의

Criteria: 
- 모드 전환 시 페이지 깜빡임 없음
- localStorage에 사용자 선택 저장
- 모든 기존 컴포넌트에 일관 적용 (미적용 컴포넌트 0개)
```

#### 2. 버그 수정 요청

**Bad:**
```
로그인이 안 돼
```
"안 된다"만으로는 10가지 원인이 가능합니다. Claude는 가장 흔한 원인부터 추측합니다.

**Good:**
```
로그인 시 500 에러가 발생해.

Context: Devise gem 사용 중, 어제 Rails 7.1에서 7.2로 업그레이드함. 
회원가입은 정상 동작. 기존 사용자만 로그인 실패.

Constraint: 
- Devise 설정(devise.rb)은 변경하지 않았음
- 에러 로그: "BCrypt::Errors::InvalidHash"
- 테스트 환경에서는 정상 동작

Criteria: 기존 사용자가 이메일/비밀번호로 로그인 시 세션이 정상 생성되어야 함.
```

#### 3. UI 수정 요청

**Bad:**
```
메인 페이지 이쁘게 해줘
```
"이쁘다"의 기준이 사람마다 다릅니다.

**Good:**
```
메인 페이지 히어로 섹션을 개선해줘.

Context: B2B SaaS 랜딩 페이지. 타겟은 30~40대 스타트업 대표.
현재 히어로 섹션에 텍스트만 있고, CTA 버튼 클릭률이 2%로 낮음.

Constraint:
- Tailwind CSS만 사용 (외부 라이브러리 추가 금지)
- 히어로 이미지는 /public/images/hero.webp 사용
- 모바일 375px에서도 텍스트 잘림 없이 표시

Criteria:
- 핵심 메시지가 3초 이내에 읽힘 (15단어 이내)
- CTA 버튼이 스크롤 없이 바로 보임 (above the fold)
- 로딩 시간 LCP 2.5초 이내
```

#### 4. 데이터베이스 관련 요청

**Bad:**
```
게시판 만들어줘
```

**Good:**
```
QnA 게시판의 데이터 모델과 기본 CRUD를 만들어줘.

Context: 온라인 강의 플랫폼. 수강생이 질문하고 강사가 답변하는 구조.
이미 User 모델(role: student/instructor)과 Course 모델이 있음.

Constraint:
- 질문은 특정 Course에 속해야 함 (course_id 필수)
- 답변은 강사(instructor role)만 작성 가능
- 첨부 파일은 Active Storage 사용, 이미지만 허용
- 대댓글 없이 1단 답변만 지원

Criteria:
- 질문 목록: 최신순/답변대기순 정렬 가능
- 답변 완료된 질문에 "해결됨" 배지 표시
- 내 질문만 필터링하는 기능 포함
- Fixture 기반 모델 테스트 포함
```

#### 5. 성능 개선 요청

**Bad:**
```
페이지가 느려
```

**Good:**
```
강의 목록 페이지(/courses)의 로딩 속도를 개선해줘.

Context: 강의 300개, 각 강의에 수강생 수/평점/카테고리 표시.
현재 페이지 로드 시간 약 4초. Rails 서버 로그에서 SQL 쿼리가 312개 찍힘.

Constraint:
- 기존 UI 레이아웃 유지 (HTML 구조 변경 최소화)
- 페이지네이션은 Pagy gem 사용
- 캐싱은 아직 적용하지 않은 상태

Criteria:
- SQL 쿼리 수 20개 이내로 감소 (N+1 해결)
- 페이지 로드 시간 1초 이내
- 개선 전/후 쿼리 로그 비교 가능하도록 설명 포함
```

### "모호한 요청 → 명확한 지시" 변환 공식

모든 요청에 3C를 풀로 쓸 필요는 없습니다. 상황에 따라 무게를 조절합니다.

**빠른 변환 공식:**
```
[모호한 요청] + "~인 상황에서" (Context) + "~를 지키면서" (Constraint) + "~하게" (Criteria)
```

예시 변환:
- "알림 기능 추가해줘" → "Rails Action Cable 프로젝트에서(C), 실시간 브라우저 알림을 지키면서(C), 읽지 않은 알림 수가 뱃지로 표시되게(C) 알림 기능 추가해줘"
- "코드 정리해줘" → "users_controller.rb가 300줄인 상황에서(C), 기존 테스트가 전부 통과하는 걸 지키면서(C), 각 메서드가 20줄 이내가 되게(C) 코드 정리해줘"
- "배포해줘" → "Railway에 연결된 Rails 앱 상황에서(C), main 브랜치만 배포하면서(C), 배포 후 health check 통과가 확인되게(C) 배포해줘"

### 3C를 더 강력하게: 고급 테크닉

#### 부정 조건 추가하기

"하지 말 것"을 명시하면 Claude가 잘못된 방향으로 가는 걸 막습니다.

```
Constraint:
- jQuery 사용하지 않기 (Stimulus만)
- 새로운 gem 추가하지 않기
- 기존 마이그레이션 수정하지 않기 (새 마이그레이션으로)
```

#### 우선순위 표시하기

모든 조건이 동등하지 않을 때:

```
Criteria (우선순위 순):
1. [필수] 기존 테스트 전부 통과
2. [필수] 모바일에서 깨지지 않음
3. [선호] 애니메이션 부드럽게 전환
4. [선호] 다크 모드 지원
```

#### 참고 자료 첨부하기

디자인이나 구조를 설명할 때:

```
Context: 
참고할 UI: Notion의 사이드바 (접힘/펼침 구조)
참고할 코드: app/views/layouts/application.html.erb 의 기존 사이드바
```

### 프롬프트 라이브러리: 자주 쓰는 패턴 저장

매번 3C를 처음부터 쓰지 않아도 됩니다. **자주 하는 유형의 템플릿**을 만들어두면 빈칸만 채우면 됩니다.

**기능 추가 템플릿:**
```
[기능 이름]을 만들어줘.

Context: [서비스 한 줄 설명]. [기술 스택]. [관련 기존 기능].
Constraint: [기술 제약 1], [비즈니스 제약 1], [범위 제약 1].
Criteria: [성능 기준], [UX 기준], [품질 기준].
```

**버그 수정 템플릿:**
```
[증상 한 줄 설명].

Context: [관련 기술/gem]. [최근 변경 사항]. [동작하는 부분].
Constraint: [에러 메시지/로그]. [재현 조건]. [변경하면 안 되는 것].
Criteria: [정상 동작 기준]. [검증 방법].
```

**리팩토링 템플릿:**
```
[대상 파일/모듈]을 리팩토링해줘.

Context: [현재 상태 — 줄 수, 복잡도 등]. [이 코드의 역할].
Constraint: [유지할 것 — 인터페이스, 테스트 등]. [변경 범위].
Criteria: [목표 상태 — 줄 수, 구조 등]. [성공 기준 — 테스트 통과 등].
```

**UI 수정 템플릿:**
```
[페이지/컴포넌트]의 [수정할 부분]을 개선해줘.

Context: [현재 상태]. [대상 사용자]. [디자인 시스템/CSS 프레임워크].
Constraint: [레이아웃 제약]. [반응형 조건]. [접근성 요구].
Criteria: [시각적 기준]. [인터랙션 기준]. [성능 기준].
```

이 템플릿들을 `my-prompt-library.md`에 저장해두고, 요청할 때마다 복사해서 빈칸만 채우면 됩니다.


## EXECUTE

### Step 1: Bad Prompt 체험하기 (5분)

코드 에디터에서 Claude Code를 열고, 자신의 서비스에 필요한 기능을 **일부러 모호하게** 요청합니다:

```
[기능 이름] 만들어줘
```

Claude의 응답을 주의 깊게 관찰합니다. 다음을 메모하세요:

- Claude가 어떤 **가정**을 했는가? (기술 스택, UI 구조, 데이터 형식 등)
- 내가 원하던 것과 **얼마나 다른가?** (0% ~ 100%)
- **어떤 정보가 빠졌기 때문에** 차이가 생겼는가?
- Claude가 "확인 질문"을 했다면, 그 질문이 3C 중 어디에 해당하는가?

이 결과를 삭제하지 마세요. Step 2 결과와 비교할 겁니다.

### Step 2: 3C 패턴으로 같은 기능 다시 요청하기 (10분)

이번에는 같은 기능을 3C 패턴으로 다시 요청합니다:

```
[기능 이름]을 만들어줘.

Context: [내 서비스 설명, 대상 사용자, 기술 스택, 현재 상태]
Constraint: [기술적 제약 2~3가지, 비즈니스 제약 1~2가지]
Criteria: [성능 기준, UX 기준, 품질 기준 각 1개 이상]
```

작성이 어렵다면 Claude에게 도움을 요청합니다:

```
방금 "[기능 이름] 만들어줘"라고 요청했더니 내가 원하는 것과 달랐어.
이 요청을 3C 패턴(Context/Constraint/Criteria)으로 다시 작성하는 걸 도와줘.
하나씩 질문해서 내 의도를 정확하게 파악해줘.
```

두 결과를 나란히 비교합니다:
- 어떤 차이가 있는가?
- 3C 중 **어떤 요소**가 가장 큰 차이를 만들었는가?
- 절약된 수정 횟수는 몇 번인가?

### Step 3: 프롬프트 라이브러리 만들기 (7분)

자주 하는 요청 유형별로 3C 템플릿을 만들어 저장합니다. Claude에게 다음과 같이 요청합니다:

```
내가 자주 하는 요청 유형에 대해 3C 패턴 템플릿을 만들어서 
my-prompt-library.md로 저장해줘.

포함할 유형:
1. 기능 추가
2. 버그 수정
3. UI 수정
4. 리팩토링

각 템플릿에는:
- 빈칸 채우기 형식의 3C 체크리스트
- Good 예시 1개 (내 서비스 맥락으로)
- Bad 예시 1개 (같은 요청의 모호한 버전)
- "이것만은 빠뜨리지 마세요" 체크 항목

내 서비스 컨텍스트:
[서비스 이름, 기술 스택, 대상 사용자]
```

### Step 4: Before/After 비교 및 저장 (3분)

오늘 작성한 Bad prompt와 Good prompt를 나란히 정리합니다:

```
오늘 실습한 Before/After를 my-prompt-library.md 맨 아래에 추가해줘.

## 오늘의 실습 기록
### Before (모호한 요청)
[Step 1에서 쓴 프롬프트]
→ 결과: [어떤 문제가 있었는지]

### After (3C 패턴)
[Step 2에서 쓴 프롬프트]  
→ 결과: [어떻게 개선됐는지]

### 배운 점
[3C 중 가장 효과적이었던 요소와 이유]
```

앞으로 새로운 Before/After 사례가 생길 때마다 이 문서에 추가하면, 자신만의 프롬프트 노하우가 쌓입니다.


## QUIZ

다음 질문에 답해봅니다:

1. 3C 패턴의 세 요소(Context, Constraint, Criteria)를 각각 한 문장으로 설명해보세요.

2. 다음 프롬프트에서 빠진 3C 요소는 무엇인가요?
```
게시판에 댓글 기능을 추가해줘. 대댓글은 1단계까지만 지원하고, 
Turbo Stream으로 실시간 업데이트 해줘.
```

3. 왜 모호한 프롬프트도 "어느 정도" 동작하는 결과를 만들어내는데, 그래도 3C를 써야 할까요?

4. 다음 Bad 프롬프트를 3C 패턴으로 변환해보세요:
```
알림 기능 추가해줘
```

5. 3C 패턴에서 "부정 조건(하지 말 것)"을 추가하면 좋은 이유는 무엇인가요?

정답:
- 1번: Context는 "지금 상황과 배경 — 어떤 서비스이고, 기술 스택이 무엇이고, 현재 어떤 상태인지", Constraint는 "반드시 지켜야 할 제약 조건 — 기술적/비즈니스적으로 변경 불가능한 것", Criteria는 "완성으로 인정하는 기준 — 성능, UX, 품질 측면에서 합격선"
- 2번: Context가 빠져 있음. 어떤 서비스인지, 기술 스택이 무엇인지, 대상 사용자가 누구인지, 기존 게시판 구조가 어떤지 알 수 없음. Criteria도 부족함 — 댓글 로딩 속도, 에러 처리 기준, 빈 댓글 방지 등의 완성 기준이 없음. Constraint(대댓글 1단계, Turbo Stream)만 있는 상태.
- 3번: Claude가 "보편적 추측"으로 채운 부분이 내 서비스와 맞지 않을 확률이 높기 때문. 재작업 비용(3번 수정 x 매번 5분 = 15분)이 3C 작성 비용(3분)보다 훨씬 큼. 특히 프로젝트가 커질수록 맥락 불일치의 누적 비용이 기하급수적으로 증가함.
- 4번: 예시 답변 — "실시간 브라우저 알림 기능을 추가해줘. Context: Rails + Action Cable 기반 온라인 강의 플랫폼, 수강생 1000명. 현재 알림 없이 이메일만 발송 중. Constraint: Action Cable 사용, 알림 유형은 새 강의/댓글 답변/공지 3종, 모바일 반응형 필수. Criteria: 알림 발생 후 2초 이내 브라우저 표시, 읽지 않은 알림 수 뱃지 표시, 7일 지난 알림 자동 정리."
- 5번: Claude는 최신 트렌드나 가장 보편적인 방식을 선택하는 경향이 있음. "jQuery 쓰지 마", "새 gem 추가하지 마"처럼 금지 조건을 명시하면 Claude가 잘못된 방향으로 가는 것을 사전에 차단할 수 있음. 특히 기존 프로젝트에 코드를 추가할 때, 기존 컨벤션과 다른 방식을 막는 데 매우 효과적.
