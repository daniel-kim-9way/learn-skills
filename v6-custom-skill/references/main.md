## EXPLAIN

### 스킬이란? — "나만의 레시피북"

매일 아침 카페에서 같은 커피를 주문한다고 생각해보세요.

**스킬 없이**: 매번 "에스프레소 2샷, 우유 150ml, 60도로 데워서, 시나몬 살짝, 컵은 머그로, 텀블러 할인 적용해주세요"
**스킬 있으면**: "내 카페라떼요" (바리스타가 레시피를 이미 알고 있으니까)

Claude Code에서 스킬은 **재사용 가능한 지시서**입니다. 한 번 만들어두면 `/스킬이름`으로 바로 실행할 수 있습니다. 같은 설명을 반복하지 않아도 됩니다.

---

### 왜 스킬이 필요한가?

Claude Code를 쓰다 보면 이런 상황이 반복됩니다:

```
"저번에도 이렇게 요청했는데, 또 같은 설명을 하고 있네..."
"매번 비슷한 형식으로 요청하는데, 템플릿이 있으면 좋겠다..."
"이 작업은 매주 하는 건데, 한 번에 끝나면 좋겠다..."
```

이런 반복을 **자동화**하는 것이 스킬입니다.

스킬 없이 반복하면:
- 매번 같은 맥락을 설명하느라 시간 소모
- 요청할 때마다 미묘하게 다른 결과 (일관성 없음)
- 좋은 프롬프트를 만들어도 다음에 또 처음부터

스킬을 만들면:
- `/스킬이름` 한 번으로 즉시 실행
- 항상 같은 품질의 결과 (일관성 확보)
- 한 번 만들고 계속 개선 (복리 효과)

---

### Template + STUB 패턴 상세 설명

스킬은 두 부분으로 구성됩니다. **빈칸 채우기 시험**을 떠올리면 이해가 쉽습니다.

#### Template (뼈대) — 변하지 않는 구조

모든 스킬에 공통인 구조입니다. 이 구조는 바꾸지 않습니다.

```markdown
---
name: STUB_NAME
description: "STUB_DESCRIPTION"
---

# STUB_TITLE

## 이 스킬이 하는 일
STUB_PURPOSE

## 실행 조건
STUB_TRIGGER

## 실행 단계
1. STUB_STEP_1
2. STUB_STEP_2
3. STUB_STEP_3

## 출력 형식
STUB_OUTPUT_FORMAT

## 주의사항
STUB_CAUTION
```

#### STUB (빈칸) — 상황에 맞게 채우는 부분

각 STUB은 여러분의 구체적인 상황에 맞게 채웁니다:

| STUB | 역할 | 예시 |
|------|------|------|
| `STUB_NAME` | 스킬 이름 (슬래시 명령어) | `daily-report` |
| `STUB_DESCRIPTION` | 한 줄 설명 (스킬 목록에 표시) | "매일 아침 작업 요약 보고서 생성" |
| `STUB_TITLE` | 스킬 제목 | "Daily Report 생성" |
| `STUB_PURPOSE` | 이 스킬이 해결하는 문제 | "매일 git log를 보고 수동으로 보고서를 쓰는 시간을 절약" |
| `STUB_TRIGGER` | 언제 이 스킬을 쓰는지 | "매일 오전 스탠드업 전" |
| `STUB_STEP_1~3` | 구체적인 실행 단계 | "어제의 git commit 목록을 가져와서 요약" |
| `STUB_OUTPUT_FORMAT` | 결과물 형식 | "마크다운 표, 3개 섹션" |
| `STUB_CAUTION` | 주의할 점, 하지 말 것 | "개인 브랜치 커밋은 제외" |

**핵심 원리**: Template는 뼈대, STUB은 살. 뼈대가 있으면 빈 화면 앞에서 "뭘 써야 하지?" 고민할 필요가 없습니다.

### 반복 작업 식별하기 — 스킬 후보 찾는 법

좋은 스킬의 3가지 조건:

**1. 반복성** — 일주일에 2회 이상 하는 작업인가?
```
매일 아침: 어제 작업 요약 → 반복성 ✅
1년에 한 번: 서버 마이그레이션 → 반복성 ❌
```

**2. 구조성** — 매번 비슷한 패턴으로 진행되는가?
```
코드 리뷰: 항상 같은 체크리스트 확인 → 구조성 ✅
브레인스토밍: 매번 완전히 다른 방향 → 구조성 ❌
```

**3. 구체성** — 실행 단계가 명확한가?
```
"git log 확인 → 요약 작성 → 형식에 맞춰 출력" → 구체성 ✅
"좋은 코드 짜줘" → 구체성 ❌ (기준이 모호)
```

**자가 진단 체크리스트:**

내가 지난 한 주 동안...
- [ ] 같은 형식의 요청을 2번 이상 했다
- [ ] "저번에도 이렇게 했는데..." 라고 생각했다
- [ ] 같은 작업을 하면서 이전 채팅을 스크롤해서 찾았다
- [ ] 결과물의 형식이 매번 달라서 통일하고 싶었다

2개 이상 체크했다면, 스킬을 만들 때입니다.

### 스킬 후보 예시 모음

**업무/협업 스킬:**
| 스킬 이름 | 하는 일 | 실행 빈도 |
|----------|---------|----------|

| `pr-checklist` | PR 생성 시 리뷰 체크리스트 자동 작성 | 주 3~5회 |

**개발 스킬:**
| 스킬 이름 | 하는 일 | 실행 빈도 |
|----------|---------|----------|
| `new-feature` | 새 기능 개발 시 보일러플레이트 생성 | 주 1~2회 |
| `bug-report` | 버그 리포트를 정해진 템플릿으로 작성 | 수시 |
| `code-review` | 코드 리뷰 시 체크리스트 기반 검토 | 주 3~5회 |
| `deploy-check` | 배포 전 체크리스트 확인 | 주 1~2회 |

**콘텐츠 스킬:**
| 스킬 이름 | 하는 일 | 실행 빈도 |
|----------|---------|----------|
| `blog-outline` | 블로그 글 구조 잡기 | 주 1회 |
| `changelog` | 변경 사항을 사용자 친화적으로 정리 | 주 1회 |
| `email-draft` | 고객/파트너 이메일 초안 작성 | 수시 |

### 실전 스킬 예시: daily-standup (완성본)

STUB를 전부 채운 완성된 스킬 예시입니다:

```markdown
---
name: daily-standup
description: "매일 아침 스탠드업 준비. 어제 작업 요약, 오늘 할 일 정리."
---

# Daily Standup 준비

## 이 스킬이 하는 일
git log와 이슈 목록을 기반으로 스탠드업 내용을 자동 정리합니다.
매일 아침 10분 걸리던 보고서 작성을 2분으로 줄입니다.

## 실행 조건
매일 오전 스탠드업 회의 전에 실행합니다.

## 실행 단계
1. 어제의 git commit 목록을 가져와서 "완료한 일"로 요약
   - 커밋 메시지에서 핵심 내용만 추출
   - 사소한 커밋(오타 수정, 포맷팅)은 묶어서 "코드 정리"로 표시
2. 현재 브랜치에서 진행 중인 작업을 "진행 중"으로 정리
   - 브랜치 이름에서 작업 내용 추측
   - 마지막 커밋 시간 기준으로 진행률 추정
3. GitHub Issues에서 나에게 할당된 이슈를 "오늘 할 일"로 정리
   - 우선순위 높은 것부터 정렬
   - 마감일이 있으면 함께 표시
4. 어제 커밋 중 실패한 테스트가 있으면 "블로커"로 표시

## 출력 형식
### 📋 [날짜] 스탠드업

#### ✅ 어제 완료
- [커밋 기반 요약, 3~5줄]

#### 🔄 진행 중
- [현재 브랜치 작업, 1~2줄]

#### 📌 오늘 예정
- [이슈 기반 목록, 우선순위 순]

#### 🚧 블로커
- [있으면 작성, 없으면 "없음"]

## 주의사항
- 개인 실험 브랜치(exp/*, scratch/*)의 커밋은 제외
- 주말/공휴일 다음 날은 마지막 근무일 기준으로 조회
```

이 예시에서 주목할 점:
- 각 단계가 **구체적** (무엇을, 어떻게, 어떤 기준으로)
- 예외 상황 처리 포함 (사소한 커밋, 주말 다음 날)
- 출력 형식이 **명확** (복사해서 바로 사용 가능)

### 스킬 작성 시 흔한 실수 5가지

| 실수 | 예시 | 개선 |
|------|------|------|
| 너무 모호한 단계 | "코드를 분석한다" | "app/models/ 폴더의 모든 모델 파일에서 N+1 쿼리 가능성을 찾는다" |
| 출력 형식 미지정 | "결과를 보여준다" | "마크다운 표로, 파일명/줄번호/문제/제안 4컬럼" |
| 너무 많은 단계 | 10단계 이상 | 5단계 이내로, 복잡하면 2개 스킬로 분리 |
| 주의사항 누락 | 없음 | "테스트 파일은 분석 대상에서 제외" |
| 트리거 미지정 | 언제 쓰는지 불명확 | "PR 생성 직전에 실행" |

### 스킬 개선의 사이클

스킬은 한 번에 완벽하게 만들 수 없습니다. 아래 사이클을 반복합니다:

```
작성 (v1) → 테스트 → 부족한 점 발견 → 개선 (v2) → 테스트 → ...
   ↑                                                    │
   └────────── 실전에서 새로운 니즈 발견 ←──────────────┘
```

**v1**: 기본 동작만 되면 OK (완벽하지 않아도 됨)
**v2**: 첫 테스트에서 발견한 문제 수정
**v3**: 실전 사용 후 예외 상황 추가
**v4+**: 팀원 피드백 반영, 출력 형식 다듬기

**중요**: 처음부터 완벽한 스킬을 만들려고 하지 마세요. **쓰면서 개선**하는 것이 핵심입니다.


## EXECUTE

### Step 1: 반복 작업 발견하기 (5분)

지난 일주일을 돌아봅니다. Claude에게 다음과 같이 요청합니다:

```
내가 지난 일주일간 Claude에게 반복적으로 요청한 것 같은 패턴을 찾아줘.
채팅 히스토리를 보고 2회 이상 비슷하게 요청한 것을 알려줘.

없으면, 아래 후보 중에서 내 상황에 맞는 걸 추천해줘:
- 코드 리뷰 체크리스트 생성
- 커밋 메시지 작성
- 버그 리포트 템플릿 작성
- 회의록 정리
- 데일리 작업 요약
- 새 기능 보일러플레이트 생성

내 상황: [간단한 역할/업무 설명]
```

추천받은 것 중 하나를 선택합니다. **가장 자주 하는 것**을 고르세요.

### Step 2: STUB 함께 채우기 — 빈칸 채우기 실습 (15분)

선택한 반복 작업을 스킬로 변환합니다. **핵심: Claude가 질문하고, 내가 답하는 대화형으로 진행합니다.**

Claude에게 요청합니다:

```
[선택한 작업]을 Claude Code 스킬로 만들고 싶어.

아래 Template의 STUB 부분을 함께 채워줘.
한 번에 완성하지 말고, STUB 하나씩 물어봐줘.
내 답변을 듣고 적절한 내용으로 채워줘.

Template:
---
name: STUB_NAME
description: "STUB_DESCRIPTION"
---

# STUB_TITLE

## 이 스킬이 하는 일
STUB_PURPOSE

## 실행 조건
STUB_TRIGGER

## 실행 단계
1. STUB_STEP_1
2. STUB_STEP_2
3. STUB_STEP_3

## 출력 형식
STUB_OUTPUT_FORMAT

## 주의사항
STUB_CAUTION
```

Claude가 물어보면 답합니다. 예시 대화:

```
Claude: "이 스킬의 이름을 뭘로 할까요? /명령어로 실행할 이름이에요."
나: "daily-report"
Claude: "한 줄 설명은요? 스킬 목록에서 보일 설명이에요."
나: "매일 아침 어제 작업 내용을 정리하는 보고서"
Claude: "이 스킬은 구체적으로 어떤 문제를 해결하나요?"
나: "매일 아침 git log를 보면서 수동으로 보고서를 쓰는 데 10분 걸려. 이걸 자동화하고 싶어"
...
```

모든 STUB이 채워지면 Claude가 완성본을 보여줍니다.

### Step 3: 스킬 저장 및 테스트 (10분)

완성된 스킬을 저장합니다:

```
완성된 스킬을 .claude/skills/[스킬이름]/SKILL.md로 저장해줘.
```

저장 후 바로 테스트합니다:

```
/[스킬이름]
```

결과를 꼼꼼히 확인합니다. 다음을 메모하세요:
- 기대한 대로 동작하는가?
- 빠진 정보가 있는가?
- 출력 형식이 사용하기 편한가?
- 불필요한 내용이 포함되어 있는가?
- 예외 상황(데이터 없음 등)에서는 어떻게 동작하는가?

### Step 4: 반복 개선하기 (10분)

테스트 결과를 바탕으로 스킬을 개선합니다:

```
방금 /[스킬이름]을 테스트했는데 이런 부분이 아쉬워:
- [개선점 1: 예 — "커밋 메시지가 너무 세세하게 나와서 요약이 필요해"]
- [개선점 2: 예 — "출력에 날짜가 빠져있어"]
- [개선점 3: 예 — "테스트 관련 커밋은 따로 분류해줬으면 좋겠어"]

스킬 파일을 수정해줘.
```

수정 후 다시 테스트합니다. **2~3회 반복**하면 실용적인 스킬이 완성됩니다.

완벽할 필요 없습니다. 80%만 동작해도 매번 처음부터 작성하는 것보다 훨씬 빠릅니다. 나머지 20%는 쓰면서 다듬으면 됩니다.

**최종 확인:**
```
/[스킬이름]을 한 번 더 실행해서 최종 결과를 확인하고,
만족스러우면 "이 스킬은 준비 완료"라고 알려줘.
개선할 점이 더 있으면 제안해줘.
```

5. 스킬 개선 사이클에서 "처음부터 완벽하게 만들지 말라"는 이유는?

정답:
- 1번: Template는 모든 스킬에 공통인 변하지 않는 구조(뼈대), STUB는 각 스킬의 목적에 맞게 채워 넣는 구체적인 내용(살). Template가 있으면 새 스킬을 빈 화면에서 시작하지 않아도 됨. 빈칸 채우기 시험에서 문제지가 Template, 답이 STUB.
- 2번: (a), (c), (e). (a)는 프로젝트마다 반복되고 구조가 비슷함. (c)는 PR마다 반복되는 체크리스트. (e)는 매주 반복되는 보고서. (b)는 구조성이 없고(매번 완전히 다른 판단), (d)는 반복성이 없음(한 번만 하는 작업).
- 3번: "분석한다"가 너무 모호해서 Claude가 무엇을 어떻게 분석할지 추측해야 함. "app/models/ 폴더의 모든 모델 파일에서 N+1 쿼리 가능성을 찾는다"처럼 대상, 범위, 기준을 구체적으로 써야 일관된 결과가 나옴.
- 4번: 테스트 실행. 실제로 한 번 돌려보고 결과를 확인한 뒤 부족한 부분을 개선해야 함. 스킬은 한 번에 완성되지 않고 "작성 → 테스트 → 개선" 사이클로 다듬어짐.
- 5번: 실제로 사용해봐야 부족한 점을 알 수 있기 때문. 머릿속으로 상상한 것과 실전은 항상 다름. 80%만 동작하는 v1을 빨리 만들고, 실전 피드백으로 20%를 채우는 것이 처음부터 100%를 목표로 시작도 못 하는 것보다 훨씬 효과적.
