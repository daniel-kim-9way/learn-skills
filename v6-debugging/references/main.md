## EXPLAIN

### 에러를 만나면 누구나 당황합니다

화면에 빨간 글씨가 뜨면 심장이 쿵 합니다. "뭘 잘못한 거지?", "큰일 난 건 아니겠지?"

하지만 에러는 **적이 아니라 단서**입니다. 에러 메시지는 "여기가 아파요"라고 말하는 환자와 같습니다. 메시지를 잘 읽으면 원인을 찾을 수 있고, 원인을 알면 고칠 수 있습니다.

이 레슨에서는 에러를 **체계적으로** 다루는 방법을 배웁니다. 감으로 이것저것 고쳐보는 대신, 의사처럼 진단하는 법을 익힙니다.

---

### 에러의 3가지 유형 — 종류를 알면 전략이 달라진다

모든 에러는 크게 3가지로 나뉩니다. 유형에 따라 접근 전략이 완전히 다릅니다.

#### 1. 구문 에러 (Syntax Error) — "문법이 틀렸어요"

코드의 "맞춤법"이 틀린 경우입니다. 한글로 치면 "안녕하세오"처럼 규칙에 맞지 않는 것.

```ruby
# 구문 에러 예시 — 괄호가 안 닫힘
def hello(name
  puts "Hello, #{name}"
end
# → SyntaxError: unexpected end, expecting ')'
```

**특징:**
- 프로그램이 **아예 실행되지 않음** (시작 전에 멈춤)
- 에디터가 빨간 밑줄로 바로 표시해줌
- 에러 메시지가 정확한 위치를 알려줌

**난이도: 가장 쉬움** (에디터와 Claude가 즉시 찾아줌)

**자주 발생하는 구문 에러:**
| 실수 | 예시 | 해결 |
|------|------|------|
| 괄호 안 닫음 | `def hello(name` | `)` 추가 |
| 따옴표 안 닫음 | `"Hello, #{name}` | `"` 추가 |
| 오타 | `defn hello` | `def`로 수정 |
| 들여쓰기 잘못 | Python에서 탭/스페이스 혼용 | 통일 |
| end 누락 | Ruby에서 `do...` 후 `end` 없음 | `end` 추가 |

#### 2. 런타임 에러 (Runtime Error) — "실행 중에 터졌어요"

코드 문법은 맞지만, 실행 중에 예상치 못한 상황이 발생한 경우입니다. 
문법은 맞는 문장인데 실제로는 불가능한 것과 같습니다. "나는 하늘을 날겠다" — 문법 OK, 현실 불가능.

```ruby
# 런타임 에러 예시 — 존재하지 않는 데이터 접근
user = User.find(999)        # ID 999가 DB에 없음
# → ActiveRecord::RecordNotFound: Couldn't find User with 'id'=999

user = nil
user.name                    # nil에서 메서드 호출
# → NoMethodError: undefined method 'name' for nil:NilClass
```

**특징:**
- 코드가 **일단 실행은 됨** (특정 조건에서만 터짐)
- 항상 발생하는 게 아니라 **특정 데이터/상황**에서만 발생할 수 있음
- **스택 트레이스**를 잘 읽어야 원인을 찾을 수 있음

**난이도: 중간** (스택 트레이스 읽는 법을 알아야 함)

**자주 발생하는 런타임 에러:**
| 에러 | 의미 | 흔한 원인 |
|------|------|----------|
| `NoMethodError: undefined method for nil` | nil에서 메서드 호출 | DB에서 못 찾은 데이터 사용 |
| `RecordNotFound` | DB에 해당 레코드 없음 | 삭제된 데이터 접근 |
| `ZeroDivisionError` | 0으로 나누기 | 사용자 입력값 미검증 |
| `TypeError` | 타입 불일치 | 문자열과 숫자 혼합 연산 |
| `NameError` | 정의 안 된 변수/메서드 | 오타 또는 스코프 문제 |

#### 3. 논리 에러 (Logic Error) — "돌아가는데 결과가 틀려요"

프로그램은 **아무 에러 없이 정상 실행**됩니다. 하지만 결과가 틀립니다. 가장 찾기 어려운 에러입니다.

```ruby
# 논리 에러 예시 — 할인 계산이 잘못됨
def discount_price(price, discount_percent)
  price * discount_percent / 100
  # 원래 의도: price - (price * discount_percent / 100)
  # 10000원의 20% 할인 → 8000원이어야 하는데
  # 실제 결과: 10000 * 20 / 100 = 2000 (할인액만 반환)
end
```

**특징:**
- **에러 메시지가 없음** (프로그램은 "정상" 동작)
- 결과를 직접 확인해야만 발견 가능
- 코드의 "의도"와 "실제 동작"의 차이를 이해해야 함

**난이도: 가장 어려움** (에러 메시지 없이 직접 찾아야 함)

**논리 에러 발견 팁:**
- 계산 결과를 직접 손으로 검산해보기
- 경계값(0, 음수, 빈 값, 매우 큰 수) 테스트
- "이 코드가 무엇을 해야 하는지" 먼저 말로 설명하고, 코드와 비교

### 체계적 디버깅 5단계 — 의사처럼 진단하기

에러를 만나면 당황하지 말고 이 순서를 따릅니다. 의사가 환자를 진단하는 과정과 같습니다.

#### 1단계: 재현 (Reproduce) — "언제 아픈가요?"

의사가 가장 먼저 하는 질문입니다. 에러도 마찬가지입니다.

```
확인할 것:
- 에러가 항상 발생하는가? 아니면 가끔?
- 어떤 조건에서 발생하는가? (특정 사용자? 특정 데이터? 특정 시간?)
- 최소 재현 단계는? (어떤 순서로 하면 에러가 나는가?)
```

**왜 중요한가**: 재현할 수 없는 에러는 고칠 수 없습니다. 그리고 고친 후에도 "진짜 고쳐졌는지" 확인할 수 없습니다.

**Claude에게 재현 조건 공유하기:**
```
이런 에러가 발생해:
- 상황: 로그인한 사용자가 프로필 페이지에 접근할 때
- 빈도: 항상 발생 (비로그인 사용자는 정상)
- 최소 재현: 로그인 → /profile 접근 → 500 에러
```

#### 2단계: 읽기 (Read) — "어디가 아픈지 정확히 봅시다"

에러 메시지를 **끝까지** 읽습니다. 대부분의 사람이 이 단계를 건너뛰고 바로 코드를 수정하려 합니다.

**에러 메시지 읽는 순서:**
1. **에러 타입**: 어떤 종류의 에러인가 (NoMethodError, RecordNotFound 등)
2. **에러 메시지**: 구체적으로 뭐가 문제인가
3. **발생 위치**: 어느 파일, 몇 번째 줄인가
4. **스택 트레이스**: 어떤 경로로 여기까지 왔는가

```
NoMethodError: undefined method 'title' for nil:NilClass
^^에러 타입^^  ^^^^^^^^^^에러 메시지^^^^^^^^^^^^^^^^^^
```

이것만 읽어도 "nil인 무언가에서 title을 호출했다"는 걸 알 수 있습니다.

#### 3단계: 격리 (Isolate) — "이 부분이 문제인 것 같습니다"

가설을 2~3개 세우고, 가장 가능성 높은 것부터 검증합니다.

```
가설 예시:
1. DB에서 데이터를 못 찾아서 nil이 됐다 (가능성 높음)
2. 변수명 오타로 다른 변수를 참조하고 있다 (가능성 중간)
3. 데이터는 있지만 title 컬럼이 없다 (가능성 낮음)
```

**격리 방법:**
- 의심 가는 부분 **앞뒤에 출력문** 추가: `puts "user = #{user.inspect}"`
- **한 번에 한 가지만** 변경해서 확인
- 변경 → 확인 → 원복 → 다음 시도

#### 4단계: 수정 (Fix) — "치료합니다"

원인을 확인하고 수정합니다. 수정할 때 주의할 점:

```
나쁜 수정: 에러만 안 나게 감추기
user.title rescue "제목 없음"  # 에러를 숨기면 다른 곳에서 터짐

좋은 수정: 원인을 해결하기
user = Post.find_by(id: params[:id])
if user.nil?
  redirect_to posts_path, alert: "게시글을 찾을 수 없습니다"
  return
end
user.title  # 이제 안전
```

#### 5단계: 검증 (Verify) — "다 나았는지 확인합시다"

수정 후 반드시 확인합니다:

1. **원래 에러가 사라졌는가?** (재현 단계 다시 실행)
2. **다른 곳에서 새 에러가 생기지 않았는가?** (전체 테스트 실행)
3. **같은 에러가 재발하지 않도록** 테스트를 추가했는가?

```
Claude에게 요청:
"이 수정이 다른 곳에 영향을 주지 않는지 확인해줘.
 그리고 같은 에러가 재발하지 않도록 테스트를 추가해줘."
```

### 스택 트레이스 읽는 법 — 비개발자를 위한 해석 가이드

스택 트레이스는 에러가 발생한 **경로를 보여주는 지도**입니다. 처음 보면 무섭지만, 규칙을 알면 읽을 수 있습니다.

**실제 스택 트레이스 예시:**

```
NoMethodError: undefined method 'title' for nil:NilClass

  app/views/posts/show.html.erb:5:in '_app_views_posts_show_html_erb'
  app/controllers/posts_controller.rb:12:in 'show'
  actionpack (7.1.0) lib/action_controller/metal/basic_implicit_render.rb:6
  actionpack (7.1.0) lib/action_controller/metal/rendering.rb:165
  rack (3.0.0) lib/rack/runtime.rb:24
```

**읽는 순서 (위에서 아래로):**

**Step 1 — 첫 줄: 에러의 정체**
```
NoMethodError: undefined method 'title' for nil:NilClass
```
해석: "nil인 무언가에서 title을 호출했다. nil에는 title 메서드가 없어서 에러."

**Step 2 — 내 코드 찾기: `app/`으로 시작하는 줄**
```
app/views/posts/show.html.erb:5          ← 여기! 내 코드, 5번째 줄
app/controllers/posts_controller.rb:12    ← 여기! 내 코드, 12번째 줄
```
`app/`으로 시작하는 줄이 내 코드입니다. 여기가 범인입니다.

**Step 3 — 나머지는 무시해도 됨**
```
actionpack (7.1.0) lib/...   ← Rails 프레임워크 코드 (보통 문제 아님)
rack (3.0.0) lib/...         ← Rack 라이브러리 코드 (보통 문제 아님)
```
이 줄들은 Rails 프레임워크 내부 코드입니다. 99%의 경우 여기가 문제가 아니라, **내 코드에서 잘못된 값을 넘겨준 것**이 문제입니다.

**Step 4 — 핵심 질문 던지기**
```
"show.html.erb의 5번째 줄에서 nil.title을 호출했다"
→ "왜 이 값이 nil인가?"
→ "posts_controller.rb의 12번째 줄에서 이 값을 설정했을 텐데..."
→ "12번째 줄에서 Post.find가 nil을 반환한 건가?"
```

**Claude에게 스택 트레이스 공유하는 템플릿:**
```
이런 에러가 발생했어:

[에러 메시지 + 스택 트레이스 전체 붙여넣기]

질문:
1. 이 에러의 직접적인 원인이 뭘까?
2. 어느 파일의 몇 번째 줄을 확인해야 할까?
3. 가장 가능성 높은 해결 방법은?
```

### git bisect로 "언제부터 망가졌나" 찾기

**상황**: "어제까지 잘 됐는데 오늘 갑자기 안 돼요."

이런 상황에서 가장 강력한 도구가 `git bisect`입니다.

**원리: 이진 탐색**

커밋이 100개 있다고 합시다. 하나하나 확인하면 최대 100번. git bisect는 **절반씩 나눠서** 확인합니다.

```
커밋 100개가 있을 때:
1번째 테스트: 50번 커밋 확인 → 여기까지 정상 → 51~100 사이에 범인
2번째 테스트: 75번 커밋 확인 → 여기서 에러 → 51~75 사이에 범인  
3번째 테스트: 63번 커밋 확인 → 여기까지 정상 → 64~75 사이에 범인
...
7번째 테스트: 범인 커밋 발견!
```

100개 커밋을 **7번**만 테스트하면 범인을 찾습니다.

**사용 방법:**

```bash
# 1. bisect 시작
git bisect start

# 2. 현재 상태: 에러가 있음
git bisect bad

# 3. 정상이었던 커밋 지정 (날짜나 커밋 해시로)
git bisect good abc1234

# 4. git이 중간 커밋으로 이동 → 테스트 실행
#    정상이면: git bisect good
#    에러면:   git bisect bad

# 5. 반복하면 범인 커밋을 찾아줌
# "abc5678 is the first bad commit"

# 6. 원래 상태로 복귀
git bisect reset
```

**비개발자를 위한 비유:**

책에서 특정 페이지를 찾을 때, 처음부터 한 장씩 넘기지 않죠? 책을 반으로 펼쳐서 "너무 앞이네/너무 뒤네"를 판단하면서 좁혀갑니다. git bisect도 같은 원리입니다.

**Claude에게 bisect 도움 받기:**
```
"어제까지 동작하던 [기능 이름]이 오늘 안 돼.
git bisect로 원인 커밋을 찾고 싶어.
단계별로 안내해줘."
```

### 예방 전략: 버그를 만들지 않는 5가지 습관

디버깅의 최고 전략은 **버그를 안 만드는 것**입니다. 100% 예방은 불가능하지만, 발생 빈도를 크게 줄일 수 있습니다.

#### 습관 1: 작게 커밋하기

변경이 작으면 문제의 범위도 작습니다.

```
나쁜 습관: 3시간 동안 작업하고 한 번에 커밋
→ 에러 발생 시 3시간치 코드를 전부 뒤져야 함

좋은 습관: 30분마다, 또는 기능 단위로 커밋
→ 에러 발생 시 최근 30분치 코드만 확인하면 됨
```

**규칙**: "동작하는 상태"에서만 커밋합니다. 절대 "반쯤 만든 상태"에서 커밋하지 않습니다.

#### 습관 2: 변경 후 바로 테스트하기

코드를 수정하면 **바로** 관련 테스트를 실행합니다.

```
수정 → 테스트 → 통과 → 다음 수정
수정 → 테스트 → 실패 → 방금 수정한 부분이 원인 (범위가 좁음!)
```

나중에 몰아서 테스트하면, 어느 수정이 문제인지 찾기 어렵습니다.

#### 습관 3: 경계값 생각하기

대부분의 버그는 "보통 상황"이 아니라 **"특이한 상황"**에서 발생합니다.

```
항상 이것들을 생각하세요:
- nil (값이 없으면?)
- 빈 배열/문자열 (데이터가 0개이면?)
- 0 또는 음수 (수량이 0이면? 마이너스이면?)
- 아주 긴 문자열 (이름이 200자이면?)
- 특수 문자 (이름에 <script>가 들어있으면?)
```

#### 습관 4: Claude에게 리뷰 요청하기

코드를 완성한 후, 커밋하기 전에 Claude에게 리뷰를 요청합니다.

```
이 코드에서 버그가 날 만한 부분을 찾아줘.
특히 nil 처리, 경계값, 에러 처리를 중점적으로 봐줘.

[코드 또는 파일 경로]
```

사람은 자기 코드의 버그를 잘 못 찾습니다. 다른 눈(Claude)이 봐주면 놓치는 부분을 잡을 수 있습니다.

#### 습관 5: 에러 로그 기록하기

만난 에러와 해결 방법을 기록합니다. 같은 에러를 두 번 만나면 이전 기록을 보고 즉시 해결할 수 있습니다.

```
## 내 에러 로그

### NoMethodError: undefined method 'title' for nil
- 날짜: 2026-04-02
- 원인: Post.find 대신 Post.find_by를 써야 했음
- 해결: find_by + nil 체크 추가
- 교훈: find는 못 찾으면 에러, find_by는 nil 반환
```

이 기록이 쌓이면 **디버깅 속도가 점점 빨라집니다**. 복리 학습의 디버깅 버전입니다.

## QUIZ

다음 질문에 답해봅니다:

1. 프로그램이 정상적으로 실행되지만 결과가 틀린 경우, 어떤 유형의 에러인가요? 이 유형이 가장 어려운 이유는?

2. 스택 트레이스를 읽을 때 가장 먼저 확인해야 할 두 가지는?

3. "어제까지 동작하던 기능이 오늘 갑자기 안 됩니다." 이 상황에서 가장 효율적인 디버깅 전략은?

4. 디버깅 5단계(재현 → 읽기 → 격리 → 수정 → 검증) 중 가장 많이 건너뛰는 단계는 어디이고, 왜 건너뛰면 안 되나요?

5. "에러를 숨기는 수정"과 "원인을 해결하는 수정"의 차이는 무엇인가요? 예시를 들어 설명해보세요.

정답:
- 1번: 논리 에러. 에러 메시지가 없어서 **발견 자체가 어려움**. 코드의 의도(프로그래머가 생각한 동작)와 실제 동작의 차이를 이해해야 하므로, 에러 메시지를 읽는 것보다 훨씬 어려움.
- 2번: (1) **첫 줄의 에러 타입과 메시지** — 어떤 종류의 문제인지 파악. (2) **app/으로 시작하는 내 코드 라인** — 프레임워크 코드보다 내 코드가 원인인 경우가 99%.
- 3번: **git bisect**. 어제의 커밋(good)과 오늘의 커밋(bad) 사이에서 이진 탐색으로 문제를 일으킨 정확한 커밋을 찾을 수 있음. 수십 개의 커밋이 있어도 7번 정도의 테스트로 범인을 특정할 수 있음.
- 4번: **재현(Reproduce)** 단계. "에러가 났다"라고 바로 코드를 고치려 하는데, 재현 조건을 먼저 확인하지 않으면 (1) 잘못된 부분을 고치거나, (2) 고쳤는데 다른 조건에서 또 발생하거나, (3) 수정 후 "진짜 고쳐졌는지" 검증이 불가능해짐.
- 5번: "에러를 숨기는 수정"은 `rescue nil`처럼 에러 메시지만 안 나오게 하는 것. 원인은 그대로인데 증상만 숨김. 다른 곳에서 더 심각한 문제로 나타남. "원인을 해결하는 수정"은 `find_by + nil 체크`처럼 왜 nil이 되는지 원인을 파악하고, nil일 때 적절히 처리하는 것. 예: `user.title rescue "없음"` (숨기기) vs `if user.nil? then redirect_to(...) end` (해결).
