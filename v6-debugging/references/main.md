## EXPLAIN

### 에러의 3가지 유형

모든 에러는 크게 3가지로 나뉩니다. 유형을 알면 접근 전략이 달라집니다.

**1. 구문 에러 (Syntax Error)**
- 코드의 "문법"이 틀린 경우
- 예: 괄호 안 닫음, 오타, 들여쓰기 잘못
- 특징: 프로그램이 아예 실행되지 않음
- 난이도: 가장 쉬움 (에디터가 바로 표시해줌)

```
# 구문 에러 예시
def hello(name
  puts "Hello, #{name}"  # 괄호가 안 닫힘
end
```

**2. 런타임 에러 (Runtime Error)**
- 실행 중에 발생하는 에러
- 예: nil 참조, 0으로 나누기, 파일 없음
- 특징: 특정 조건에서만 발생할 수 있음
- 난이도: 중간 (스택 트레이스를 읽어야 함)

```
# 런타임 에러 예시
user = User.find(999)  # 존재하지 않는 ID
user.name              # NoMethodError: undefined method 'name' for nil
```

**3. 논리 에러 (Logic Error)**
- 프로그램은 돌아가지만 결과가 틀린 경우
- 예: 계산식 잘못, 조건문 역전, 변수 혼동
- 특징: 에러 메시지가 없어서 발견이 어려움
- 난이도: 가장 어려움 (직접 찾아야 함)

```
# 논리 에러 예시
def discount_price(price, discount_percent)
  price * discount_percent / 100  # price - (price * discount_percent / 100) 이어야 함
end
```

### 체계적 디버깅 5단계

에러를 만나면 당황하지 말고 이 순서를 따릅니다:

**1단계: 재현 (Reproduce)**
- 에러가 항상 발생하는가, 가끔 발생하는가?
- 어떤 조건에서 발생하는가?
- 최소 재현 단계를 찾는다

**2단계: 읽기 (Read)**
- 에러 메시지를 **끝까지** 읽는다
- 스택 트레이스에서 **내 코드** 부분을 찾는다
- 에러 타입과 메시지를 정확히 파악한다

**3단계: 가설 (Hypothesize)**
- "왜 이 에러가 날까?" 가설을 2~3개 세운다
- 가장 가능성 높은 것부터 검증한다

**4단계: 검증 (Verify)**
- 가설을 하나씩 테스트한다
- 변수 값 출력, 로그 확인, 조건 변경 등
- 한 번에 한 가지만 변경한다

**5단계: 수정 & 방지 (Fix & Prevent)**
- 원인을 확인하고 수정한다
- 같은 에러가 재발하지 않도록 테스트를 추가한다

### 스택 트레이스 읽는 법

스택 트레이스는 에러가 발생한 경로를 보여주는 "지도"입니다.

```
NoMethodError: undefined method 'title' for nil:NilClass
  from app/views/posts/show.html.erb:5
  from app/controllers/posts_controller.rb:12:in 'show'
  from actionpack (7.1.0) lib/action_controller/metal/basic_implicit_render.rb:6
```

읽는 순서:
1. **첫 줄**: 에러 타입과 메시지 → `NoMethodError: nil에서 title 호출`
2. **내 코드 찾기**: `app/` 으로 시작하는 줄 → `show.html.erb:5`, `posts_controller.rb:12`
3. **아래에서 위로**: 실행 흐름을 거꾸로 추적
4. **핵심 질문**: "왜 여기서 nil인가?"

Claude에게 스택 트레이스를 공유할 때:
```
이런 에러가 발생했어:
[스택 트레이스 전체 붙여넣기]

어떤 부분을 확인해야 할까?
```

### git bisect: 범인 찾기 도구

"어제까지 됐는데 오늘 안 돼" 상황에서 가장 강력한 도구입니다.

git bisect는 이진 탐색으로 문제를 일으킨 커밋을 찾습니다:

```
git bisect start
git bisect bad          # 현재 상태: 에러 있음
git bisect good abc123  # 이 커밋에서는 정상이었음
# git이 중간 커밋으로 이동 → 테스트 → good/bad 반복
# 최종적으로 문제 커밋을 찾아줌
git bisect reset        # 원래 상태로 복귀
```

100개의 커밋이 있어도 7번만 테스트하면 범인을 찾습니다.

### 예방 전략: 버그를 만들지 않는 습관

디버깅의 최고 전략은 **버그를 안 만드는 것**입니다.

1. **작게 커밋하기**: 변경이 작으면 문제 범위도 작음
2. **테스트 먼저 작성하기**: 기대 결과를 먼저 정의
3. **경계값 생각하기**: nil, 빈 배열, 0, 음수 등
4. **Claude에게 리뷰 요청하기**: "이 코드에서 버그가 날 만한 부분을 찾아줘"

## EXECUTE

### Step 1: 에러 유형 분류 연습 (5분)

Claude에게 요청합니다:

```

내가 어떤 유형인지 맞춰볼게.

내 기술 스택: [Ruby on Rails / JavaScript / Python 등]
```

Claude가 제시하는 에러를 보고 유형을 맞춰봅니다.

### Step 2: 스택 트레이스 읽기 연습 (10분)

실제 프로젝트에서 에러를 찾거나, Claude에게 연습 문제를 요청합니다:

```
Rails 프로젝트에서 발생할 법한 스택 트레이스 예시를 하나 만들어줘.
내가 읽고 원인을 추측해볼게.

포함할 것:
- 에러 메시지
- 스택 트레이스 (내 코드 + 프레임워크 코드 혼합)
- 관련 코드 스니펫
```

5단계(재현→읽기→가설→검증→수정)를 적용해서 분석합니다.

### Step 3: git bisect 체험 (10분)

프로젝트에서 git bisect를 사용해봅니다:

```
git bisect를 연습하고 싶어.

내 프로젝트에서 안전하게 git bisect를 체험할 수 있는 방법을 안내해줘.
실제 버그를 찾을 필요는 없고, 사용법을 익히고 싶어.

안내할 것:
1. git bisect start/good/bad 명령어
2. 중간 커밋에서 확인하는 방법
3. git bisect reset으로 복귀하는 방법
```

경험을 정리합니다:

포함할 것:

- 스택 트레이스 읽기 순서
- git bisect 사용 시점과 명령어

내 기술 스택: [기술 스택]

## QUIZ

다음 질문에 답해봅니다:

1. 프로그램이 정상적으로 실행되지만 결과가 틀린 경우, 어떤 유형의 에러인가요? 이 유형이 가장 어려운 이유는?

2. 스택 트레이스를 읽을 때 가장 먼저 확인해야 할 두 가지는?

3. "어제까지 동작하던 기능이 오늘 갑자기 안 됩니다." 이 상황에서 가장 효율적인 디버깅 전략은?

4. 디버깅 5단계(재현→읽기→가설→검증→수정) 중 가장 많이 건너뛰는 단계는 어디이고, 왜 건너뛰면 안 되나요?

정답:
- 1번: 논리 에러. 에러 메시지가 없어서 발견 자체가 어렵고, 코드의 의도와 실제 동작의 차이를 이해해야 하기 때문.
- 2번: (1) 첫 줄의 에러 타입과 메시지, (2) app/으로 시작하는 내 코드 라인. 프레임워크 코드보다 내 코드가 원인인 경우가 대부분.
- 3번: git bisect. 어제의 커밋(good)과 오늘의 커밋(bad) 사이에서 이진 탐색으로 문제를 일으킨 정확한 커밋을 찾을 수 있음. 수십 개의 커밋이 있어도 몇 번의 테스트로 범인을 특정할 수 있음.
- 4번: 재현(Reproduce) 단계. "에러가 났다"라고 바로 코드를 고치려 하는데, 재현 조건을 먼저 확인하지 않으면 (1) 잘못된 부분을 고치거나, (2) 고쳤는데 다른 조건에서 또 발생하거나, (3) 수정 검증이 불가능해짐.
