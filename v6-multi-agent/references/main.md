## EXPLAIN

### 왜 여러 에이전트가 필요한가?

혼자 일하는 것과 팀으로 일하는 것의 차이를 생각해봅니다.

**혼자 일할 때**: 코드 작성 → 테스트 → 문서 작성 → 리뷰 (순서대로, 하나씩)

**팀으로 일할 때**: 한 명은 코드 작성, 한 명은 테스트 작성, 한 명은 문서 작성 (동시에)

Claude Code에서도 같은 원리가 적용됩니다. 하나의 에이전트로 모든 걸 순서대로 하는 대신, 여러 에이전트가 **독립적인 작업을 동시에** 처리할 수 있습니다.

### Multi-Agent의 3가지 패턴

**패턴 1: 병렬 실행 (Parallel)**
독립적인 작업을 동시에 처리합니다.

```
에이전트 A: 프론트엔드 컴포넌트 수정
에이전트 B: 백엔드 API 수정
에이전트 C: 테스트 작성
→ 모두 끝나면 통합
```

적합한 경우: 서로 다른 파일을 수정하는 독립 작업

**패턴 2: 파이프라인 (Pipeline)**
한 에이전트의 결과를 다음 에이전트가 사용합니다.

```
에이전트 A: 요구사항 분석 → 결과 전달
에이전트 B: 분석 기반 코드 작성 → 결과 전달  
에이전트 C: 코드 리뷰 및 개선
```

적합한 경우: 검토-구현-검증 같은 단계별 작업

**패턴 3: 전문가 위원회 (Committee)**
같은 문제를 여러 관점에서 분석합니다.

```
에이전트 A: 보안 관점에서 검토
에이전트 B: 성능 관점에서 검토
에이전트 C: 사용자 경험 관점에서 검토
→ 의견을 종합해서 결정
```

적합한 경우: 중요한 아키텍처 결정, 코드 리뷰

### 병렬 작업의 판단 기준

모든 작업에 Multi-Agent가 필요한 건 아닙니다.

**Multi-Agent가 적합한 경우:**
- 작업이 서로 독립적 (같은 파일을 수정하지 않음)
- 작업량이 충분히 큼 (10분 이상 걸리는 작업)
- 결과를 나중에 합칠 수 있음

**단일 에이전트가 나은 경우:**
- 작업이 순서 의존적 (A의 결과가 있어야 B를 시작)
- 같은 파일을 동시에 수정해야 함
- 작업이 작고 빠르게 끝남 (5분 이내)

### 세션 관리 기초

여러 에이전트를 쓸 때 중요한 것은 **컨텍스트 관리**입니다.

- 각 에이전트는 별도의 세션에서 동작합니다
- 에이전트 간에 정보를 공유하려면 **파일**을 사용합니다
- 작업 시작 전에 **분배 계획**을 세워야 합니다

실전 팁:
- 에이전트에게 명확한 범위를 지정하세요 (파일, 디렉토리, 역할)
- 동시 최대 2~3개 에이전트가 적당합니다
- 결과를 합치는 단계를 미리 계획하세요

## EXECUTE

### Step 1: 병렬화 가능한 작업 찾기 (5분)

자신의 프로젝트에서 동시에 진행할 수 있는 작업을 찾습니다.

Claude에게 요청합니다:

```
내 프로젝트에서 병렬로 진행할 수 있는 독립적인 작업 조합을 찾아줘.

조건:
- 서로 다른 파일을 수정하는 작업
- 각각 10분 이상 걸릴 만한 작업
- 결과를 나중에 합칠 수 있는 작업

프로젝트 상태: [간단한 설명 또는 git status 공유]
```

### Step 2: 작업 분배 계획 세우기 (5분)

찾은 작업을 에이전트별로 분배합니다:

```
아래 작업을 Multi-Agent로 분배해줘.

작업 목록:
- [작업 1]
- [작업 2]  
- [작업 3]

각 에이전트에 대해 정리해줘:
1. 담당 범위 (수정할 파일/디렉토리)
2. 구체적인 지시 내용
3. 완료 기준
4. 다른 에이전트와의 의존 관계
```

### Step 3: 실전 병렬 작업 (15분)

계획대로 실행합니다. 코드 에디터에서:

1. 메인 Claude Code 세션에서 에이전트 A의 작업을 시작합니다
2. 새 터미널/세션을 열어 에이전트 B의 작업을 시작합니다
3. 각 에이전트의 진행 상황을 확인합니다
4. 모든 에이전트가 끝나면 결과를 통합합니다

막히면 Claude에게 물어봅니다:

```
에이전트 A에서 [이런 상황]이 생겼는데, 에이전트 B 작업에 영향이 있을까?
```

### Step 4: 워크플로우 정리 & 저장 (5분)

경험을 정리합니다:

```
오늘 Multi-Agent로 작업한 경험을 my-agent-workflow.md로 저장해줘.

포함할 내용:
- 사용한 패턴 (병렬/파이프라인/위원회)
- 작업 분배 방법
- 잘 된 점과 개선할 점
- 앞으로 Multi-Agent를 쓸 상황 목록
```

## QUIZ

다음 질문에 답해봅니다:

1. Multi-Agent의 3가지 패턴(병렬, 파이프라인, 위원회) 중 "코드 작성 → 리뷰 → 수정" 흐름에 가장 적합한 패턴은?

2. 다음 상황에서 Multi-Agent를 써야 할까요, 단일 에이전트가 나을까요?
   - (a) 같은 models/user.rb 파일의 서로 다른 메서드 수정
   - (b) 프론트엔드 페이지 3개를 각각 독립적으로 수정
   - (c) 한 줄짜리 버그 수정

3. 여러 에이전트가 동시에 작업할 때 가장 중요한 관리 원칙은?

정답:
- 1번: 파이프라인 패턴. 한 단계의 결과가 다음 단계의 입력이 되는 순서 의존적 작업이므로.
- 2번: (a) 단일 에이전트 — 같은 파일을 수정하면 충돌 위험. (b) Multi-Agent — 서로 다른 파일, 독립적, 양이 충분. (c) 단일 에이전트 — 작업이 너무 작아 분배 오버헤드가 더 큼.
- 3번: 각 에이전트의 작업 범위를 명확히 분리하는 것. 같은 파일이나 리소스에 동시 접근하지 않도록 사전에 범위를 지정해야 충돌을 방지할 수 있음.
