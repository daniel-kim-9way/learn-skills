## EXPLAIN

### "혼자 vs 팀" — 언제 여러 에이전트가 필요한가

식당을 운영한다고 상상해보세요.

**혼자 운영할 때**: 주문받기 → 요리하기 → 서빙하기 → 설거지하기 (순서대로, 하나씩)
손님 5명이면 마지막 손님은 한참을 기다려야 합니다.

**팀으로 운영할 때**: 한 명은 주문, 한 명은 요리, 한 명은 서빙 (동시에 진행)
손님 5명이어도 대기 시간이 짧습니다.

Claude Code에서도 같은 원리입니다.

**단일 에이전트**: 프론트엔드 수정 → 완료 → 백엔드 수정 → 완료 → 테스트 작성 → 완료
총 소요시간: 30분

**멀티 에이전트**: 에이전트 A는 프론트엔드, 에이전트 B는 백엔드, 에이전트 C는 테스트 (동시 진행)
총 소요시간: 10분 (가장 오래 걸리는 작업 기준)

단, 모든 작업에 팀이 필요한 건 아닙니다. **한 줄 버그 수정에 3명을 투입하면 오히려 비효율**입니다. 언제 팀으로 일할지 판단하는 것이 핵심입니다.

### Multi-Agent의 3가지 패턴

#### 패턴 1: 병렬 실행 (Parallel) — "각자 맡은 영역"

독립적인 작업을 동시에 처리합니다. 가장 많이 쓰는 패턴입니다.

```
에이전트 A: 프론트엔드 컴포넌트 수정  ─────────→ 완료
에이전트 B: 백엔드 API 수정          ─────────→ 완료
에이전트 C: 테스트 작성              ─────────→ 완료
                                                  ↓
                                              결과 통합
```

**적합한 경우:**
- 서로 다른 파일을 수정하는 독립 작업
- 프론트엔드 / 백엔드 / 테스트를 동시에 진행
- 여러 페이지를 각각 독립적으로 수정

**실전 예시:**
```
작업 분배:
- 에이전트 A: app/views/users/ 폴더의 UI 수정
- 에이전트 B: app/controllers/api/ 폴더의 API 수정
- 에이전트 C: test/ 폴더에 새 테스트 추가

규칙: 각 에이전트는 자기 폴더만 수정. 다른 폴더 절대 건드리지 않기.
```

#### 패턴 2: 파이프라인 (Pipeline) — "릴레이 경주"

한 에이전트의 결과를 다음 에이전트가 이어받습니다. 순서가 중요한 작업에 사용합니다.

```
에이전트 A: 요구사항 분석
         ↓ (결과 전달)
에이전트 B: 분석 기반 코드 작성
         ↓ (결과 전달)
에이전트 C: 코드 리뷰 및 개선
```

**적합한 경우:**
- 검토 → 구현 → 검증 같은 단계별 작업
- 한 단계의 결과가 다음 단계의 입력이 되는 작업
- 품질 체크 파이프라인 (코드 작성 → 리뷰 → 수정)

**실전 예시:**
```
1단계 에이전트: "app/models/user.rb를 분석해서 개선점을 report.md에 정리해줘"
2단계 에이전트: "report.md의 개선점을 기반으로 user.rb를 리팩토링해줘"
3단계 에이전트: "리팩토링된 user.rb의 테스트를 실행하고 결과를 review.md에 정리해줘"
```

#### 패턴 3: 전문가 위원회 (Committee) — "다각도 검토"

같은 문제를 여러 관점에서 동시에 분석합니다. 중요한 결정을 내릴 때 사용합니다.

```
에이전트 A: 보안 관점에서 검토    ──→ 의견 A
에이전트 B: 성능 관점에서 검토    ──→ 의견 B  ──→ 종합 판단
에이전트 C: 사용자 경험 관점 검토  ──→ 의견 C
```

**적합한 경우:**
- 중요한 아키텍처 결정 (DB 구조, 기술 스택 선택)
- 배포 전 최종 검토
- 여러 해결 방법 중 최선의 선택이 필요할 때

**실전 예시:**
```
모든 에이전트에게 같은 코드를 공유하고:
- 에이전트 A: "이 코드에서 보안 취약점을 찾아서 security-review.md에 정리해줘"
- 에이전트 B: "이 코드의 성능 병목을 찾아서 performance-review.md에 정리해줘"  
- 에이전트 C: "이 코드의 UX 문제를 찾아서 ux-review.md에 정리해줘"
→ 세 파일을 종합해서 최종 결정
```

### 병렬 작업의 판단 기준 — "이 작업, 나눠야 할까?"

모든 작업에 Multi-Agent가 필요한 건 아닙니다. 잘못 나누면 오히려 시간이 더 걸립니다.

**Multi-Agent가 적합한 5가지 신호:**

| 신호 | 예시 | 이유 |
|------|------|------|
| 서로 다른 파일 수정 | 프론트엔드 vs 백엔드 | 충돌 위험 없음 |
| 각 작업이 10분 이상 | 페이지 3개 각각 수정 | 병렬화 효과가 큼 |
| 결과를 나중에 합칠 수 있음 | 각자 다른 기능 개발 | 통합이 단순 |
| 독립적 테스트 가능 | 각 기능이 별개로 동작 | 개별 검증 가능 |
| 전문 영역이 다름 | 보안/성능/UX 검토 | 관점 분리 |

**단일 에이전트가 나은 5가지 신호:**

| 신호 | 예시 | 이유 |
|------|------|------|
| 같은 파일 수정 | user.rb의 두 메서드 수정 | 동시 수정 시 충돌 |
| 순서 의존적 | A 결과 없으면 B 불가 | 기다려야 해서 병렬화 무의미 |
| 작업이 작음 (5분 이내) | 한 줄 버그 수정 | 분배 오버헤드가 더 큼 |
| 맥락 공유 필요 | 전체 흐름을 이해해야 하는 작업 | 에이전트 간 소통 비용 |
| DB 마이그레이션 포함 | 스키마 변경 + 코드 수정 | 순서가 중요 |

**빠른 판단 플로차트:**

```
이 작업을 2개 이상으로 나눌 수 있는가?
  ├─ 아니오 → 단일 에이전트
  └─ 예 → 나눈 작업들이 같은 파일을 수정하는가?
           ├─ 예 → 단일 에이전트
           └─ 아니오 → 각 작업이 10분 이상 걸리는가?
                       ├─ 아니오 → 단일 에이전트 (분배 비용 > 절약 시간)
                       └─ 예 → Multi-Agent 사용!
```

### 세션 관리 기초 — 에이전트 간 소통

여러 에이전트를 쓸 때 가장 중요한 것은 **컨텍스트 관리**입니다.

#### 핵심 원칙 3가지

**1. 범위를 명확히 분리한다**

각 에이전트에게 "너의 영역은 여기까지"를 정확히 알려줍니다.

```
나쁜 예: "프론트엔드 고쳐줘" (범위 모호)
좋은 예: "app/views/users/ 폴더의 파일만 수정해. 
         다른 폴더는 절대 건드리지 마." (범위 명확)
```

**2. 파일로 소통한다**

에이전트 간에 정보를 전달할 때는 **파일**을 사용합니다. 각 에이전트는 별도의 세션에서 동작하기 때문에, 채팅 내용을 공유할 수 없습니다.

```
에이전트 A: 분석 결과를 analysis-result.md에 저장
에이전트 B: analysis-result.md를 읽고 작업 시작
```

**3. 동시 최대 2~3개**

처음에는 2개부터 시작합니다. 익숙해지면 3개까지 늘립니다. 4개 이상은 관리 비용이 급격히 증가합니다.

```
추천 조합:
- 2개: 메인 작업 + 테스트 (가장 안전)
- 3개: 프론트 + 백엔드 + 테스트 (효율적)
- 4개 이상: 관리 비용 ↑, 충돌 위험 ↑ (비추천)
```

#### 결과 통합하기

모든 에이전트가 작업을 끝내면, 결과를 합치는 단계가 필요합니다.

```
통합 단계:
1. 각 에이전트의 변경사항 확인 (git diff)
2. 충돌이 있는지 확인
3. 전체 테스트 실행
4. 문제 있으면 수정
```

통합은 항상 **하나의 에이전트**가 담당합니다. 통합 작업을 또 나누면 혼란이 생깁니다.

### 실전 시나리오: 이렇게 사용합니다

**시나리오: 서비스에 "프로필 페이지"를 추가**

단일 에이전트로 하면:
```
1. 프로필 페이지 UI 작성 (10분)
2. 프로필 수정 API 작성 (10분)
3. 테스트 작성 (10분)
총 30분
```

Multi-Agent로 하면:
```
에이전트 A: "app/views/profiles/에 프로필 페이지 UI를 만들어줘. 
           Tailwind CSS 사용. 모바일 반응형."

에이전트 B: "app/controllers/profiles_controller.rb와 
           routes.rb에 프로필 CRUD API를 추가해줘."

(A, B가 동시 진행 — 10분)

에이전트 C (A, B 완료 후): "profiles 관련 테스트를 작성하고 전체 테스트를 실행해줘."

총 15분 (10분 병렬 + 5분 테스트)
```

시간이 절반으로 줄었습니다.


## EXECUTE

### Step 1: 병렬화 가능한 작업 찾기 (5분)

자신의 프로젝트에서 동시에 진행할 수 있는 작업을 찾습니다. Claude에게 요청합니다:

```
내 프로젝트에서 병렬로 진행할 수 있는 독립적인 작업 조합을 찾아줘.

조건:
- 서로 다른 파일을 수정하는 작업
- 각각 10분 이상 걸릴 만한 작업
- 결과를 나중에 합칠 수 있는 작업

프로젝트 상태: [간단한 설명 또는 git status 결과 공유]

추천 조합을 2~3가지 제안하고, 각 조합의 장단점을 알려줘.
```

### Step 2: 작업 분배 계획 세우기 (5분)

찾은 작업을 에이전트별로 구체적으로 분배합니다:

```
아래 작업을 Multi-Agent로 분배해줘.

작업 목록:
- [작업 1]
- [작업 2]

각 에이전트에 대해 정리해줘:
1. 담당 범위 (수정할 파일/디렉토리 — 구체적으로)
2. 구체적인 지시 내용 (복사해서 바로 쓸 수 있게)
3. 완료 기준 (어떻게 되면 "끝"인지)
4. 다른 에이전트와의 의존 관계 (있으면)
5. 절대 건드리면 안 되는 파일/폴더

통합 단계도 정리해줘:
- 모든 에이전트 완료 후 확인할 것
- 전체 테스트 실행 방법
```

### Step 3: 실전 병렬 작업 (15분)

계획대로 실행합니다:

**방법 1: 터미널 창 나누기**
1. 코드 에디터에서 터미널을 2개 엽니다
2. 터미널 1에서 에이전트 A의 작업을 시작합니다
3. 터미널 2에서 에이전트 B의 작업을 시작합니다
4. 각 에이전트의 진행 상황을 번갈아 확인합니다

**방법 2: Claude Code의 /agents 기능 활용 (가능한 경우)**
```
다음 작업을 병렬로 실행해줘:

에이전트 A:
[Step 2에서 정리한 에이전트 A 지시 내용]

에이전트 B:
[Step 2에서 정리한 에이전트 B 지시 내용]

모두 끝나면 결과를 통합하고 전체 테스트를 실행해줘.
```

작업 중 문제가 생기면:
```
에이전트 A에서 [이런 상황]이 생겼어.
에이전트 B 작업에 영향이 있을까?
계속 진행해도 될까, 아니면 B를 멈춰야 할까?
```

### Step 4: 경험 정리 및 저장 (5분)

오늘의 경험을 정리합니다:

```
오늘 Multi-Agent로 작업한 경험을 my-agent-workflow.md로 저장해줘.

포함할 내용:
- 사용한 패턴 (병렬/파이프라인/위원회)
- 작업 분배 방법 (어떤 기준으로 나눴는지)
- 실제 절약된 시간 (단일 vs 멀티 비교)
- 잘 된 점과 개선할 점
- 앞으로 Multi-Agent를 쓸 상황 목록
- 주의할 점 (충돌, 통합 등)
```


## QUIZ

다음 질문에 답해봅니다:

1. Multi-Agent의 3가지 패턴(병렬, 파이프라인, 위원회) 중 "코드 작성 → 리뷰 → 수정" 흐름에 가장 적합한 패턴은?

2. 다음 상황에서 Multi-Agent를 써야 할까요, 단일 에이전트가 나을까요?
   - (a) 같은 models/user.rb 파일의 서로 다른 메서드 수정
   - (b) 프론트엔드 페이지 3개를 각각 독립적으로 수정
   - (c) 한 줄짜리 버그 수정
   - (d) 보안/성능/UX 3가지 관점에서 코드 리뷰
   - (e) DB 마이그레이션 후 관련 코드 수정

3. 여러 에이전트가 동시에 작업할 때 가장 중요한 관리 원칙은?

4. 에이전트 간에 정보를 전달하려면 어떤 방법을 사용해야 하고, 왜 그런가요?

5. Multi-Agent 작업 후 "통합 단계"에서 반드시 해야 할 일은?

정답:
- 1번: 파이프라인 패턴. "코드 작성 → 리뷰 → 수정"은 한 단계의 결과가 다음 단계의 입력이 되는 순서 의존적 작업이므로 파이프라인이 적합. 병렬로 하면 리뷰할 코드가 아직 없는데 리뷰를 시작하는 꼴.
- 2번: (a) 단일 에이전트 — 같은 파일을 수정하면 충돌 위험. (b) Multi-Agent — 서로 다른 파일, 독립적, 양이 충분. (c) 단일 에이전트 — 작업이 너무 작아 분배 오버헤드가 더 큼. (d) Multi-Agent(위원회 패턴) — 같은 코드를 다른 관점으로 동시 검토. (e) 단일 에이전트 — 마이그레이션이 먼저 완료되어야 코드 수정 가능(순서 의존).
- 3번: 각 에이전트의 작업 범위를 명확히 분리하는 것. 어떤 파일/폴더를 수정할 수 있고 어떤 것은 절대 건드리면 안 되는지 사전에 지정해야 충돌을 방지할 수 있음.
- 4번: 파일을 통해 전달. 각 에이전트는 별도 세션에서 동작하므로 채팅 내용을 공유할 수 없음. analysis-result.md 같은 파일에 결과를 저장하면 다른 에이전트가 읽고 이어서 작업 가능.
- 5번: (1) 각 에이전트의 변경사항을 git diff로 확인, (2) 파일 충돌 여부 확인, (3) 전체 테스트 실행. 통합은 반드시 하나의 에이전트가 담당해야 하며, 통합 작업을 또 나누면 혼란이 생김.
