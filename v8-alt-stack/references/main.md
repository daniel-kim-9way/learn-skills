## EXPLAIN

### 왜 다른 기술을 알아야 하는가 — "Rails만이 답이 아닌 경우"

Rails를 배웠다고 모든 프로젝트에 Rails를 써야 하는 것은 아닙니다. 목수가 망치만 가지고 있으면 모든 문제가 못으로 보이듯, **도구의 폭**이 넓어야 최적의 선택을 할 수 있습니다.

하지만 중요한 것은 — 첫 번째 도구를 **깊이** 배운 사람이 두 번째 도구를 훨씬 빠르게 배운다는 점입니다. 왜? 프로그래밍의 핵심 개념(모델, 라우팅, 인증, 배포)은 **모든 스택에서 공통**이기 때문입니다.

Rails에서 배운 것:
- **MVC 패턴** → Next.js에서는 pages/api + components
- **Active Record** → Supabase에서는 테이블 + RLS
- **라우팅** → Next.js에서는 파일 기반 라우팅
- **배포** → Vercel에서는 git push만으로 자동 배포

> "하나를 깊이 알면 열을 빠르게 배운다."

### 언제 Rails가 최적이 아닌가

Rails가 강력한 경우:
- 데이터 중심 CRUD 서비스 (게시판, 예약, 관리 시스템)
- 빠른 풀스택 개발
- 복잡한 비즈니스 로직

Rails가 아닌 선택이 더 나은 경우:
- **정적 콘텐츠 사이트** (블로그, 포트폴리오) → Next.js + Vercel
- **실시간 기능** (채팅, 알림) → Supabase Realtime
- **AI/ML 직접 구현** → Python + FastAPI
- **빠른 아이디어 검증** (비개발자) → 노코드 (Bubble, Webflow)
- **모바일 앱 백엔드** → Supabase + Flutter/React Native

### Next.js + Vercel — 프론트엔드의 왕

#### Next.js란?

React 기반의 **풀스택 프레임워크**입니다. React를 직접 쓰는 것보다 구조적이고, Rails처럼 "규칙을 따르면 빠르게 만들 수 있는" 프레임워크입니다.

핵심 특징:
- **파일 기반 라우팅**: `pages/about.js` → `/about` (Rails의 `routes.rb`가 필요 없음)
- **서버 사이드 렌더링(SSR)**: SEO에 강함
- **정적 사이트 생성(SSG)**: 블로그, 문서 사이트에 최적
- **API Routes**: 간단한 백엔드 로직을 파일 하나로

#### Vercel이란?

Next.js를 만든 회사의 배포 플랫폼입니다. **git push 한 번**으로 배포가 끝납니다.

```
# Rails 배포 과정
Railway/Render 설정 → 환경 변수 → DB 마이그레이션 → 배포 → 도메인 설정

# Vercel 배포 과정
GitHub 연결 → git push → 끝. (자동 HTTPS, 자동 CDN)
```

#### 한 페이지를 Next.js로 만들어보기

Next.js의 느낌을 체험하기 위해, 자기소개 페이지를 만들어봅니다:

```javascript
// pages/index.js (이 파일 하나가 홈페이지)
export default function Home() {
  return (
    <div style={{ padding: '2rem', maxWidth: '600px', margin: '0 auto' }}>
      <h1>안녕하세요, [이름]입니다</h1>
      <p>저는 AI와 함께 서비스를 만드는 빌더입니다.</p>
      <h2>만든 서비스</h2>
      <ul>
        <li><a href="[URL]">첫 번째 서비스</a> — [한 줄 설명]</li>
      </ul>
    </div>
  );
}
```

Rails에서는 라우팅 + 컨트롤러 + 뷰가 따로였지만, Next.js에서는 **파일 하나**에 다 있습니다.

### Supabase — "Firebase인데 PostgreSQL"

#### Supabase란?

Google의 Firebase 대안으로, **PostgreSQL 기반**의 백엔드 서비스입니다. Rails로 치면 DB + 인증 + 스토리지가 한 번에 제공됩니다.

핵심 특징:
- **대시보드에서 DB 관리**: 마이그레이션 없이 클릭으로 테이블 생성
- **인증 내장**: 이메일/비밀번호, Google, GitHub 로그인
- **실시간 구독**: 데이터 변경 시 자동 업데이트
- **스토리지**: 파일 업로드/다운로드
- **Row Level Security**: 사용자별 데이터 접근 제어

#### Rails의 마이그레이션 vs Supabase 대시보드

```ruby
# Rails 방식
rails generate model Post title:string body:text user:references
rails db:migrate
```

```
# Supabase 방식
대시보드 → New Table → 컬럼 추가 → Save
(또는 SQL Editor에서 직접 SQL 실행)
```

### Rails vs Next.js 장단점 비교표

| 관점 | Rails | Next.js + Vercel |
|------|-------|-----------------|
| **학습 곡선** | 중간 (Convention over Configuration) | 중간 (React + Next.js 개념) |
| **풀스택** | O (백엔드 + 뷰) | O (프론트 + API Routes) |
| **DB 관리** | Active Record (강력) | 외부 DB 연결 필요 (Prisma 등) |
| **배포 용이성** | 중간 (Railway/Render) | 매우 쉬움 (Vercel git push) |
| **정적 사이트** | 약함 | 매우 강함 (SSG) |
| **실시간 기능** | ActionCable (설정 필요) | 외부 서비스 연결 |
| **SEO** | 기본 지원 | SSR/SSG로 매우 강함 |
| **커스터마이징** | 매우 높음 | 높음 |
| **AI 협업** | 좋음 (Claude Code) | 좋음 (v0, Claude Code) |
| **한국어 자료** | 적음 | 많음 |
| **커뮤니티 크기** | 중간 | 매우 큼 |
| **적합한 프로젝트** | CRUD, 관리 시스템, SaaS | 콘텐츠 사이트, SPA, 마케팅 |

### "언제 어떤 스택" 가이드

프로젝트 성격에 따라 최적의 스택이 다릅니다:

| 만들고 싶은 것 | 추천 스택 | 이유 |
|---------------|----------|------|
| 예약/관리 시스템 | **Rails** | 복잡한 CRUD + 비즈니스 로직 |
| 블로그/포트폴리오 | **Next.js + Vercel** | 정적 생성 + 빠른 배포 |
| 커뮤니티/게시판 | **Rails** | 데이터 관계 + 인증 |
| 랜딩 페이지 | **Next.js + Vercel** | 빠른 배포 + 예쁜 UI |
| 채팅/실시간 앱 | **Supabase + Next.js** | 실시간 구독 |
| AI 서비스 | **Python FastAPI** or **Rails** | ML 라이브러리 / API 연동 |
| 아이디어 검증 | **Bubble/Webflow** | 코드 없이 빠른 MVP |
| 모바일 앱 백엔드 | **Supabase** | SDK + 인증 + 실시간 |

> **정답은 없습니다.** 중요한 것은 각 도구의 특성을 알고, 상황에 맞는 선택을 하는 것입니다.

### Python + Django/FastAPI 간략 소개

데이터 분석, AI/ML과 자연스럽게 연결되는 스택입니다.

- **Django**: Python의 Rails. "배터리 포함" 철학. 관리자 페이지 자동 생성.
- **FastAPI**: 가볍고 빠른 API 서버. AI 모델 서빙에 최적.

Rails 경험이 있으면 Django는 **하루 만에** 감을 잡을 수 있습니다. 구조가 매우 비슷합니다.

### 노코드/로우코드 (Bubble, Webflow)

코드 없이 드래그 앤 드롭으로 제작하는 플랫폼입니다.

- **Bubble**: 풀스택 앱 제작 (DB + 로직 + UI)
- **Webflow**: 웹사이트 제작 (디자인 중심)
- **장점**: 가장 빠른 MVP 출시, 비개발자도 가능
- **단점**: 커스터마이징 한계, 플랫폼 종속, 비용 증가

> 코딩을 배운 사람이 노코드를 쓰면 **훨씬 효과적**입니다. 노코드의 한계를 알고, 그 안에서 최적의 결과를 낼 수 있기 때문입니다.

## QUIZ

### Q1: 다양한 스택을 아는 이유
> "여러 기술 스택을 알아두는 가장 큰 이유는?"

선택지:
1. 이력서에 기술을 많이 쓸 수 있어서
2. 상황에 맞는 최적의 도구를 선택할 수 있어서
3. 모든 스택을 마스터해야 해서
4. Rails가 부족해서

정답: 2번
해설: 모든 스택을 깊이 배울 필요는 없습니다. 하지만 각 도구의 특성을 알면 프로젝트 성격에 맞는 최적의 선택을 할 수 있습니다. 목수의 도구함에 망치만 있어서는 안 됩니다.

### Q2: 첫 번째 기술을 깊이 배우는 효과
> "하나의 기술을 깊이 배운 사람이 새로운 기술을 접할 때 유리한 이유는?"

선택지:
1. 코드 문법이 비슷해서
2. 프로그래밍의 핵심 개념(모델, 라우팅, 배포 등)이 스택을 넘어 공통이기 때문에
3. 문서를 읽는 속도가 빨라져서
4. AI가 더 잘 도와줘서

정답: 2번
해설: Rails에서 배운 MVC 패턴, 데이터베이스 설계, 인증, 배포 개념은 어떤 스택에서든 동일합니다. Next.js에서 "페이지"라고 부르든 Django에서 "뷰"라고 부르든, 핵심은 같습니다. 도구만 다를 뿐, 개념은 공유됩니다.
