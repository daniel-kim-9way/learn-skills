## EXPLAIN

### 왜 다른 스택을 알아야 하는가

Rails를 배웠다고 모든 프로젝트에 Rails를 써야 하는 것은 아닙니다. 목수가 망치만 가지고 있으면 모든 문제가 못으로 보이듯, **도구의 폭**이 넓어야 최적의 선택을 할 수 있습니다.

하지만 중요한 것은 — 첫 번째 도구를 **깊이** 배운 사람이 두 번째 도구를 훨씬 빠르게 배운다는 점입니다.

### 주요 대안 스택 소개

#### 1. Next.js + Vercel
- **특징**: React 기반 풀스택 프레임워크
- **장점**: 프론트엔드가 강력, 정적 사이트 생성(SSG) 가능, Vercel 배포가 극도로 간편
- **적합한 경우**: 콘텐츠 중심 사이트, SPA, 마케팅 페이지

#### 2. Supabase
- **특징**: Firebase의 오픈소스 대안, PostgreSQL 기반
- **장점**: 인증/DB/스토리지를 한 번에 제공, 실시간 기능 내장
- **적합한 경우**: 빠른 백엔드 셋업이 필요할 때, 모바일 앱 백엔드

#### 3. Python + Django/FastAPI
- **특징**: 데이터 분석, AI/ML과 자연스럽게 연결
- **장점**: AI 기능 직접 구현, 데이터 파이프라인 구축 용이
- **적합한 경우**: AI/데이터 중심 서비스

#### 4. 노코드/로우코드 (Bubble, Webflow)
- **특징**: 코드 없이 드래그 앤 드롭으로 제작
- **장점**: 가장 빠른 MVP 출시
- **적합한 경우**: 아이디어 검증, 비기술 창업자

### Rails와 비교하면

| 관점 | Rails | Next.js | Supabase | 노코드 |
|------|-------|---------|----------|--------|
| 학습 곡선 | 중간 | 중간 | 낮음 | 매우 낮음 |
| 풀스택 | O | O (API Routes) | 백엔드만 | O |
| 배포 용이성 | 중간 | 매우 쉬움 | 매우 쉬움 | 내장 |
| 커스터마이징 | 매우 높음 | 높음 | 중간 | 낮음 |
| AI 협업 | 좋음 | 좋음 | 보통 | 제한적 |

학생에게 질문한다:

> "Rails로 서비스를 만들어보셨는데, '다른 기술로 만들면 어떨까?'라고 생각한 적 있나요? 혹시 들어본 기술이나 관심 있는 스택이 있으면 말해주세요."

EXPLAIN의 4가지 대안 스택을 간략히 소개한다.

### Turn 2: 하나 골라서 체험하기

> "위 스택 중 하나를 골라서 간단히 체험해봅시다. 어떤 게 가장 궁금하세요?"

학생이 선택한 스택에 따라 체험을 안내한다:

**Next.js 선택 시:**
> "간단한 페이지를 하나 만들어보겠습니다. `npx create-next-app`으로 프로젝트를 생성하고, 자기소개 페이지를 만들어봅시다."

**Supabase 선택 시:**
> "Supabase 대시보드에서 테이블을 하나 만들고, 데이터를 넣어보겠습니다. Rails의 migration과 비교해보세요."

**노코드 선택 시:**
> "Webflow이나 Bubble에서 랜딩 페이지를 하나 만들어봅시다. 드래그 앤 드롭이 코드 작성과 어떻게 다른지 느껴보세요."

**Django/FastAPI 선택 시:**
> "FastAPI로 간단한 API를 하나 만들어보겠습니다. Rails의 컨트롤러와 비교해보세요."

핵심: 깊이 배우는 것이 아니라 **느낌을 체험**하는 것이 목표.

### Turn 3: Rails 경험과 비교

> "방금 체험한 [선택한 스택]과 Rails를 비교하면 어떤 느낌이었나요? 세 가지 관점에서 말해주세요:"
> 1. **시작 속도**: 프로젝트를 만들고 첫 화면을 보기까지
> 2. **이해도**: 무슨 일이 일어나는지 이해가 되는 정도
> 3. **자유도**: 원하는 대로 바꿀 수 있는 정도

## QUIZ

### Q1: 다양한 스택을 아는 이유
> "여러 기술 스택을 알아두는 가장 큰 이유는?"

선택지:
1. 이력서에 기술을 많이 쓸 수 있어서
2. 상황에 맞는 최적의 도구를 선택할 수 있어서
3. 모든 스택을 마스터해야 해서
4. Rails가 부족해서

정답: 2번
해설: 모든 스택을 깊이 배울 필요는 없습니다. 하지만 각 도구의 특성을 알면 프로젝트 성격에 맞는 최적의 선택을 할 수 있습니다.

### Q2: 첫 번째 기술을 깊이 배우는 효과
> "하나의 기술을 깊이 배운 사람이 새로운 기술을 접할 때 유리한 이유는?"

선택지:
1. 코드 문법이 비슷해서
2. 프로그래밍의 핵심 개념(모델, 라우팅, 배포 등)이 스택을 넘어 공통이기 때문에
3. 문서를 읽는 속도가 빨라져서
4. AI가 더 잘 도와줘서

정답: 2번
해설: Rails에서 배운 MVC 패턴, 데이터베이스 설계, 인증, 배포 개념은 어떤 스택에서든 동일합니다. 도구만 다를 뿐, 핵심 개념은 공유됩니다.
