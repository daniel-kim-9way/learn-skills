## EXPLAIN

### 두 번째는 다르다 — "처음엔 4주, 이번엔 1주"

첫 번째 서비스를 만들 때는 모든 것이 처음이었습니다:
- "문제를 어떻게 정의하지?"
- "사용자가 누구지?"
- "MVP 범위는 어디까지?"
- "기술은 뭘 써야 하지?"
- "배포는 어떻게 하지?"

매 단계에서 **개념 학습 + 실행**을 동시에 해야 했습니다. 하지만 두 번째부터는 **프레임워크**가 생깁니다. 질문의 순서를 알고, 각 질문에 답하는 방법을 압니다.

이것이 **학습의 복리 효과**입니다.

### 복리 효과의 실체

첫 서비스에서는 이런 일이 벌어졌습니다:

| 단계 | 첫 서비스 (C1+C2) | 두 번째 서비스 |
|------|-------------------|---------------|
| 문제 정의 | "문제가 뭔지부터 모르겠다" (1-2시간) | "이 패턴이지" (5분) |
| 사용자 분석 | "페르소나가 뭐지?" (1시간) | "구체적 1명" (3분) |
| MVP 범위 | "기능 10개 넣고 싶은데..." (2시간) | "핵심 2개만" (5분) |
| 기술 선택 | "Rails? Django? React?" (고민) | "검증된 것 재활용" (2분) |
| DB 설계 | "모델이 뭐지?" (학습+실행) | "이 패턴 재활용" (10분) |

C1+C2에 **수 시간~수일**이 걸렸다면, 이번에는 **30분이면 충분**합니다.

### 빠른 설계의 비밀: 패턴 재활용

첫 서비스에서 체화된 4가지 패턴:

**패턴 1: 문제 정의 (Problem Statement)**
> "누가 + 어떤 상황에서 + 무엇이 불편한가"

이 공식을 외울 필요가 없습니다. 이미 한 번 해봤기 때문에 자연스럽게 나옵니다.

**패턴 2: 사용자 정의 (Persona)**
> 추상적인 "20대 직장인"이 아니라, 구체적 1명으로 시작

"우리 회사 김대리"처럼 실제 사람을 떠올리면 기능이 명확해집니다.

**패턴 3: MVP 범위 (Scope)**
> 핵심 기능 2-3개만 선택 → Must-have와 Nice-to-have 분류

첫 서비스에서 "기능 10개 넣고 싶었는데 3개로 줄이니까 오히려 좋았다"는 경험이 있습니다.

**패턴 4: 기술 설계 (Tech Stack)**
> 검증된 스택 재활용 or 목적에 맞는 새 스택 선택

"이미 Rails로 해봤으니 같은 스택이면 빠르고, 새 스택이면 배움이 있다" — 선택의 기준이 명확합니다.

### 첫 서비스에서 재사용할 수 있는 것 체크리스트

두 번째 서비스를 만들 때, 첫 서비스에서 **그대로 가져올 수 있는 것**들이 있습니다:

**코드/기술 레벨:**
- [ ] 프로젝트 초기 설정 (Rails new + 기본 gem)
- [ ] 인증 시스템 (devise 또는 직접 구현한 패턴)
- [ ] 기본 레이아웃 + Tailwind 설정
- [ ] 배포 파이프라인 (Railway/Render 설정)
- [ ] .env 관리 방식

**프로세스 레벨:**
- [ ] 문제 정의 → MVP 범위 설정 프로세스
- [ ] AI에게 효과적으로 지시하는 패턴
- [ ] 에러 디버깅 접근법 (에러 메시지 → 검색 → AI 질문)
- [ ] Git 커밋 + 백업 루틴
- [ ] 매일 빌딩 루틴 (시간, 장소, 방식)

**마인드셋 레벨:**
- [ ] "완벽보다 출시" 마인드
- [ ] "막히면 다른 방식으로 물어보기"
- [ ] "작게 쪼개서 하나씩" 접근법

### 속도 비교가 주는 자신감

첫 서비스 기획에 며칠이 걸렸다면, 두 번째는 30분이면 됩니다. 세 번째는 15분이면 됩니다.

프로 빌더는 아이디어가 떠오르면 카페에서 커피 한 잔 마시는 동안 기획을 끝냅니다. 그게 가능한 이유는 패턴이 체화되어 있기 때문입니다.

> "프로와 아마추어의 차이는 재능이 아니라 **반복 횟수**다."

당신은 이미 한 번 해봤습니다. 두 번째는 **확실히** 다릅니다.

## EXECUTE

### Turn 1: 새 아이디어 선택 (3분)

학생에게 질문한다:

> "두 번째 서비스를 만들어볼 거예요. 첫 서비스를 만들면서 떠올랐던 **'이것도 있으면 좋겠다'**는 아이디어가 있나요? 없다면 아래에서 골라도 좋습니다:"
> - 첫 서비스를 만들면서 느낀 보조 도구 (예: 사용자 피드백 수집 도구)
> - 주변 사람이 겪는 다른 문제
> - 첫 서비스와 완전히 다른 분야의 아이디어
> - 회고에서 발견한 "이런 게 있으면 좋겠다"

핵심: 오래 고민하지 않는다. 3분 안에 결정한다. **아이디어의 질보다 실행의 속도**가 목표다.

학생이 아이디어를 말하면 바로 다음으로 넘어간다.

---

### Turn 2: 빠른 문제 정의 (5분) — C1 패턴 압축 적용

C1에서 했던 것을 **한 턴으로 압축**한다:

> "좋아요! 빠르게 갑시다. 한 문장으로 답해주세요 — 생각하지 말고 바로 쓰세요:"
> 1. **누가** 이 서비스를 쓰나요? (구체적 1명의 이름과 상황)
> 2. **어떤 상황**에서 불편을 느끼나요? (구체적 장면 하나)
> 3. 이 서비스가 있으면 **어떻게 달라지나요?** (Before → After)

학생이 답하면 즉시 비교:
> "C1에서는 이 과정에 여러 턴이 걸렸는데, 지금은 한 턴에 끝났죠? 이게 패턴의 힘입니다."

너무 모호하면 한 번만 구체화를 요청한다. 두 번 이상 다듬지 않는다 — 속도가 핵심이다.

---

### Turn 3: MVP 범위 + 재사용 체크리스트 (10분)

> "핵심 기능을 3개만 고르세요. 첫 서비스에서 배운 교훈 — '작게 시작하기'를 기억하면서요."

학생이 기능을 나열하면 즉시 분류한다:
- **Must-have** (출시에 반드시 필요) — 최대 2개
- **Nice-to-have** (v2에서 추가) — 나머지 전부

그 다음, 재사용 체크리스트를 확인한다:

> "첫 서비스에서 재사용할 수 있는 것을 체크해봅시다:"
> - 인증 시스템 같은 거? (Y/N)
> - 레이아웃/디자인 패턴? (Y/N)
> - 배포 설정? (Y/N)
> - CRUD 패턴? (Y/N)

재사용할 수 있는 것이 많을수록 빌드 시간이 줄어든다는 것을 강조한다.

---

### Turn 4: 기술 설계 빠른 결정 (5분)

> "기술 스택은 어떻게 하시겠어요?"

두 가지 선택지를 명확히 제시한다:

**옵션 A: 같은 스택 (속도 우선)**
- 장점: 설정 시간 0, 패턴 재활용, 2주 안에 확실히 완성
- 단점: 새로운 학습 없음

**옵션 B: 새로운 스택 (학습 우선)**
- 장점: 기술 폭 확장, 비교 경험
- 단점: 학습 시간 필요, 완성 불확실

> "어떤 선택이든 정답입니다. 이번 스킬에서는 **설계까지만** 합니다. 빌드는 v8-sprint에서 하게 됩니다."

---

### Turn 5: 속도 비교 + 설계서 저장

> "지금까지 약 20-25분이 지났습니다. 첫 서비스 기획은 얼마나 걸렸는지 기억나시나요? 비교해보면 어떤 느낌인가요?"

학생의 답변을 받은 후 `my-second-service-plan.md` 파일을 생성한다:

```markdown
# 두 번째 서비스 설계서

## 서비스 개요
- 서비스명: [가칭]
- 한 줄 설명: [누가 + 무엇을 + 왜]

## 문제 정의
- 대상 사용자: [구체적 페르소나]
- 현재 불편: [상황 설명]
- 기대 변화: [Before → After]

## MVP 기능 (우선순위순)
1. [Must-have] 기능명 — 설명
2. [Must-have] 기능명 — 설명
3. [Nice-to-have → v2] 기능명 — 설명

## 재사용 체크리스트
- [x/o] 인증 시스템
- [x/o] 레이아웃/디자인
- [x/o] 배포 파이프라인
- [x/o] CRUD 패턴

## 기술 설계
- 스택: [선택한 기술 스택]
- 이유: [선택 이유]
- 선택 타입: [속도 우선 / 학습 우선]

## 속도 비교
- 첫 서비스 기획: [C1+C2에 걸린 대략적 시간]
- 두 번째 서비스 기획: 약 30분
- 속도 향상: [X]배
- 체감 변화: [학생의 답변]

## 다음 단계
→ v8-sprint에서 이 설계서를 바탕으로 2주 스프린트 빌드

---
작성일: [오늘 날짜]
```

학생에게 안내:
> "30분 만에 서비스 설계를 끝냈습니다! 첫 번째 때와의 속도 차이가 바로 **당신의 성장**입니다. 이 설계서는 v8-sprint에서 바로 사용됩니다. '다음'을 입력하면 퀴즈로 넘어갑니다."

## QUIZ

### Q1: 패턴 재활용의 핵심
> "두 번째 서비스 기획이 빠른 이유는?"

선택지:
1. 두 번째라 대충 해도 되니까
2. 첫 서비스에서 익힌 사고 패턴(문제 정의, 사용자 분석, MVP 범위 설정)을 재활용하니까
3. 더 쉬운 아이디어를 골랐으니까
4. AI가 더 잘 도와주니까

정답: 2번
해설: 속도가 빨라진 이유는 문제 정의, 사용자 분석, MVP 범위 설정의 **사고 패턴**이 체화되었기 때문입니다. 이것이 학습의 복리 효과입니다. 3번째, 4번째로 갈수록 더 빨라집니다.

### Q2: 재사용 체크리스트의 가치
> "첫 서비스에서 재사용할 수 있는 것을 체크하는 이유는?"

선택지:
1. 코드를 복사-붙여넣기 하려고
2. 이미 검증된 패턴과 설정을 활용해서 빌드 시간을 단축하려고
3. 첫 서비스와 똑같이 만들려고
4. 기술 목록을 늘리려고

정답: 2번
해설: 재사용의 핵심은 복사가 아니라 **검증된 패턴 활용**입니다. 인증 시스템, 배포 파이프라인, CRUD 패턴 등 한 번 해본 것은 두 번째에 훨씬 빠르게 적용할 수 있습니다. "바퀴를 다시 발명하지 마라"는 말과 같습니다.
