## EXPLAIN

### 스프린트란 무엇인가

스프린트(Sprint)는 정해진 기간 안에 목표를 달성하는 **집중 작업 방식**입니다. 소프트웨어 업계에서 수십 년간 검증된 방법론이지만, 핵심 원리는 놀라울 정도로 간단합니다:

1. **시간을 정한다** — 2주
2. **할 일을 정한다** — 이 기간에 완성할 기능 목록
3. **매일 진행한다** — 하루 목표를 세우고 실행
4. **끝나면 출시한다** — 완벽하지 않아도 배포

> "완벽한 계획은 없다. 하지만 계획이 없는 것보다 나쁜 계획이 낫다." — 윈스턴 처칠

### 왜 2주인가

- **1주**: 너무 짧아서 의미 있는 기능을 만들기 어렵다
- **2주**: 긴장감은 유지하면서 충분한 결과물을 낼 수 있는 최적의 기간
- **한 달**: 너무 길어서 중간에 방향을 잃기 쉽다

2주는 "마감 효과"가 작동하는 기간입니다. 데드라인이 있으면 인간은 놀라울 정도로 집중합니다.

### 첫 서비스 vs 두 번째 서비스 — 패턴 압축

첫 서비스(C3+C4)에서는 개념 학습과 빌드를 동시에 했습니다. 두 번째 서비스에서는 이미 아는 것을 **실행만** 하면 됩니다.

| 항목 | 첫 서비스 (C3+C4) | 두 번째 서비스 (v8-sprint) |
|------|-------------------|--------------------------|
| Rails 프로젝트 생성 | 개념부터 학습 (30분+) | `rails new` 바로 실행 (5분) |
| DB 설계 | "모델이 뭐지?" 학습 | 패턴 재활용 (15분) |
| 인증 시스템 | devise 설정 학습 | 이전 설정 참고 (10분) |
| 핵심 기능 | CRUD가 뭔지부터 | "이 패턴이지" 바로 구현 |
| Tailwind 디자인 | 클래스 검색하면서 | 자주 쓰는 패턴 재활용 |
| 배포 | 환경 설정 삽질 (2시간+) | 검증된 방법 반복 (30분) |
| 예상 기간 | 수 주 | **2주 이내** |

### 2주 스프린트 일별 목표 — 완전 가이드

큰 프로젝트를 "2주 안에 완성"이라고만 생각하면 막막합니다. 하지만 **하루 단위**로 쪼개면 실행 가능해집니다.

#### Phase 1: 기반 (Day 1-3) — 프로젝트 + 모델 + 인증

**Day 1: 프로젝트 셋업**
- Rails new + GitHub 저장소 생성
- 기본 gem 설치 (Tailwind, devise 등)
- 첫 커밋 + 푸시
- 목표: `rails server`로 Welcome 페이지 확인

**Day 2: DB 설계 + 모델 생성**
- v8-second-plan의 설계서를 보며 모델 정의
- `rails generate model` 실행
- 관계(association) 설정
- 시드 데이터 작성
- 목표: `rails console`에서 데이터 생성/조회 확인

**Day 3: 인증 + 기본 레이아웃**
- 인증 시스템 설정 (devise 또는 직접 구현)
- 기본 레이아웃 + 네비게이션 바
- 홈페이지 라우팅
- 목표: 회원가입 → 로그인 → 홈페이지 플로우 완성

#### Phase 2: 핵심 기능 (Day 4-7) — 기능 2개 집중 빌드

**Day 4-5: 핵심 기능 #1**
- MVP의 Must-have 기능 1번
- 컨트롤러 + 뷰 + 라우팅
- 기본 CRUD 또는 커스텀 로직
- 목표: 기능 1이 동작하는 것을 확인

**Day 6-7: 핵심 기능 #2**
- MVP의 Must-have 기능 2번
- 기능 1과의 연결 (데이터 흐름)
- 사용자 플로우 전체 테스트
- 목표: "이 서비스를 쓸 수 있다"는 최소 수준 달성

> **Day 7 체크포인트**: 여기서 서비스를 누군가에게 보여줄 수 있는가? YES면 계속. NO면 범위를 줄인다.

#### Phase 3: 마감 (Day 8-10) — 디자인 + 배포

**Day 8: 디자인 개선**
- Tailwind로 전체 UI 정리
- 모바일 반응형 확인
- 핵심 페이지 3개에 집중 (홈, 기능1, 기능2)
- 목표: "보기 좋다"는 수준

**Day 9: 랜딩 페이지**
- 서비스 소개 페이지 제작
- 5섹션 구조 (C4 패턴 재활용)
- CTA(Call to Action) 배치
- 목표: 처음 방문한 사람이 "이게 뭔지" 이해

**Day 10: 배포**
- 배포 플랫폼 설정 (Railway/Render)
- 환경 변수 설정
- 데이터베이스 마이그레이션
- 도메인 연결 (선택)
- 목표: URL로 접속 가능

#### Phase 4: 런치 (Day 11-14) — 런치 + 피드백

**Day 11: 버그 수정 + QA**
- 배포된 서비스를 실제로 사용해본다
- 버그 리스트 작성 + 수정
- 엣지 케이스 확인
- 목표: "부끄럽지 않은" 수준

**Day 12: 런치 준비**
- 런치 체크리스트 확인 (C4 패턴)
- 소개 글 작성 (SNS, 커뮤니티)
- 스크린샷 / 데모 영상 준비
- 목표: 공유할 준비 완료

**Day 13: 런치!**
- 서비스 공개
- 주변 5명에게 공유
- 첫 피드백 수집
- 목표: 실제 사용자 최소 3명

**Day 14: 피드백 반영 + 회고**
- 수집한 피드백 정리
- 빠르게 고칠 수 있는 것 바로 수정
- v2 목록에 추가할 것 정리
- 스프린트 회고 (KPT)
- 목표: "다음에 더 잘 만들겠다"는 구체적 교훈

### 하루 빌딩 루틴

매일 작업할 때 이 루틴을 따르면 효율이 올라갑니다:

```
[5분] 어제 한 것 확인 — git log, 메모 확인
[5분] 오늘 할 것 결정 — 위 일별 목표 참고
[40-50분] 집중 빌딩 — 타이머 설정, 딴짓 금지
[5분] 진행 기록 — 한 일 / 막힌 것 / 내일 할 것
```

> 매일 1시간이면 2주 = 14시간. 14시간 안에 서비스를 만드는 것이 목표입니다.

### 막힐 때의 15분 규칙

빌드 중 막히면:
1. **5분**: 에러 메시지를 읽고 스스로 해결 시도
2. **5분**: AI에게 에러 메시지 + 코드를 보여주며 질문
3. **5분**: 다른 방식으로 접근 (우회 방법, 기능 축소)
4. **15분이 지나도 안 풀리면**: 그 기능을 v2로 미루고 다음으로 넘어간다

> "15분 규칙: 15분 안에 안 풀리면 방향을 바꿔라."

## EXECUTE

### Turn 1: 스프린트 계획 수립

> "v8-second-plan에서 만든 설계서를 바탕으로 2주 스프린트를 계획해봅시다. 먼저, **하루에 빌딩에 투자할 수 있는 시간**이 얼마나 되나요? 솔직하게 말해주세요."

학생의 가용 시간에 맞춰 스프린트 일정을 조정한다:
- 하루 1시간 → 14일 스프린트 (표준)
- 하루 30분 → 일부 Day를 합치거나 기간 연장
- 하루 2시간+ → 빠른 스프린트 가능

> "좋아요! [X]일 스프린트로 가겠습니다. MVP 기능 목록을 다시 확인해볼까요?"

v8-second-plan의 MVP 기능을 불러와서 일별 목표에 맞춰 배치한다. EXPLAIN의 4 Phase 구조를 학생의 상황에 맞게 조정한다.

### Turn 2: Day 1-3 가이드 — 프로젝트 + 모델 + 인증

> "첫 서비스 때 프로젝트를 셋업하면서 가장 시간이 많이 걸렸던 부분이 뭐였나요?"

학생의 답변을 듣고, 이번에는 그 부분을 빠르게 넘기는 전략을 제시한다.

빠른 셋업 실행:
1. `rails new [프로젝트명]` — 또는 선택한 스택으로 생성
2. GitHub 저장소 생성 + 첫 커밋

> "이 5단계를 지금 바로 실행해봅시다. 첫 서비스 때처럼 각 개념을 설명하진 않을 거예요. 이미 아시니까요. 막히는 부분만 말해주세요."

**함께 실행**하되, 학생이 스스로 하도록 유도한다. AI는 코드를 대신 쓰는 것이 아니라 "이 다음에 뭐 해야 하지?"에 답하는 역할.

### Turn 3: Day 4-7 가이드 — 핵심 기능 빌드

> "핵심 기능 구현 단계입니다. 첫 서비스에서 사용했던 패턴 중 **이번에도 재활용할 수 있는 것**은 뭐가 있을까요?"

학생과 함께 재활용 가능한 패턴을 식별한다:
- 인증 시스템 (같은 방식 반복)
- CRUD 패턴 (모델만 다르고 구조는 동일)
- UI 레이아웃 (Tailwind 컴포넌트 재활용)
- 폼 처리 패턴 (form_with, 검증)
- 플래시 메시지 패턴

Claude Code 활용 팁:
> "각 기능을 구현할 때 이렇게 지시하면 빠릅니다:"
> - "[첫 서비스]의 [기능] 패턴을 참고해서 [새 기능]을 만들어줘"
> - "이 모델의 CRUD를 만들어줘. 디자인은 [첫 서비스]의 [페이지] 스타일로"
> - 기존 코드를 **템플릿**으로 활용하는 것이 가장 빠른 방법

Day 7 체크포인트를 반드시 확인:
> "지금 만든 서비스를 누군가에게 보여줄 수 있나요? 핵심 기능이 동작하나요?"

### Turn 4: Day 8-10 가이드 — 디자인 + 배포

> "디자인과 배포 단계입니다. 여기가 첫 서비스에서 가장 시간을 많이 잡아먹었던 구간일 수 있어요. 배포할 때 어려웠던 점을 기억하시나요?"

디자인 빠른 접근법:
1. 핵심 페이지 3개만 집중 (홈, 기능1, 기능2)
2. Tailwind 컴포넌트 라이브러리 활용
3. "80% 완성도"를 목표로 — 완벽한 디자인은 나중에

배포 체크리스트 (C4 패턴 재활용):
1. 환경 변수 설정 — `.env`의 모든 키가 production에 있는지
2. 데이터베이스 마이그레이션 — `rails db:migrate`
3. 배포 플랫폼 설정 — 이전과 같은 플랫폼이면 설정 복사
4. SSL/도메인 연결 (선택)

> "이번에는 이 과정이 훨씬 자연스럽게 느껴질 거예요. '아, 이거 해봤지' 하는 감각이 있을 겁니다."

## QUIZ

### Q1: 스프린트의 핵심 가치
> "스프린트 방식의 가장 큰 장점은?"

선택지:
1. 완벽한 서비스를 만들 수 있다
2. 정해진 기간 안에 집중하여 출시 가능한 결과물을 만든다
3. 팀이 필요 없다
4. 계획 없이 자유롭게 만들 수 있다

정답: 2번
해설: 스프린트의 핵심은 **시간 제한** 안에서 **출시 가능한 결과물**을 만드는 것입니다. 완벽할 필요 없이, 사용자에게 가치를 전달할 수 있는 수준이면 됩니다. 데드라인이 집중력을 만듭니다.

### Q2: 15분 규칙
> "빌드 중 막혔을 때 '15분 규칙'의 핵심은?"

선택지:
1. 15분 쉬고 다시 하기
2. 15분 안에 해결 안 되면 방향을 바꾸거나 v2로 미루기
3. 15분 동안 검색만 하기
4. 15분마다 커밋하기

정답: 2번
해설: 15분 규칙의 핵심은 **한 문제에 매몰되지 않는 것**입니다. 5분 자체 시도 → 5분 AI 질문 → 5분 우회 시도. 그래도 안 되면 v2로 미루고 다음 기능으로 넘어갑니다. 스프린트에서는 **전체 완성**이 개별 기능 완벽보다 중요합니다.
